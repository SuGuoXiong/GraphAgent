# 10. 人机协作功能设计

## 1. 概述

### 1.1 背景

当前 GraphAgent 的编排执行是全自动的：用户发起任务后，GuardAgent → PlanAgent → SubAgent 链式执行直到完成或超时。虽然已有用户主动中断机制（见 [8.会话中断与恢复机制设计](8.会话中断与恢复机制设计.md)），但缺少 Agent 主动向用户请求信息或授权的"反向交互"能力。

典型场景：

- **信息澄清**：PlanAgent 分解任务时发现用户意图模糊（如"帮我处理一下那个文件"——哪个文件？），需要用户补充信息后再继续
- **高风险授权**：SubAgent 即将执行 `rm -rf` 或 `DROP TABLE` 等危险操作，需要用户明确批准
- **选项确认**：Agent 分析出多种可行方案，让用户选择偏好方案后继续

### 1.2 目标

- **Agent 主动提问**：Agent 在执行过程中可以调用 `ask_user` 工具向用户发起问题
- **问题类型支持**：支持自由文本输入和预定义选项选择两种交互模式
- **会话中断与恢复**：Agent 提问后自动中断会话并保存检查点，用户回答后恢复执行
- **协议原生支持**：通过 ACP 协议新增 `ASK_USER` 推送事件和 `REPLY_USER` 请求事件
- **Hook 驱动**：通过 `before_tool_call` Type 2 Hook 拦截 `ask_user` 调用，统一处理中断逻辑

### 1.3 非目标

- 不支持多轮嵌套问答（用户回答触发新的 `ask_user` 后需再次回答——这由 Agent 自行决定是否再次调用 `ask_user`，机制上已支持）
- 不支持超时自动回退（如果用户长时间不回答，Agent 不会自动做决策）
- 不支持文件上传、语音等富媒体交互（仅文本问答）

## 2. 核心概念

### 2.1 `ask_user` 工具

一个空的占位工具，实际逻辑完全由 Hook 接管：

```python
# src/graph_agent/tools/ask_user_tools.py

from graph_agent.tools.base import tool

@tool(
    name="ask_user",
    description=(
        "当需要用户提供更多信息、在多个选项中做出选择、"
        "或者需要用户批准某个操作时使用此工具。"
        "参数 question 为需要向用户提问的问题文本；"
        "参数 options 为可选的选项列表，用户可以从中选择一个；"
        "参数 require_approval 为 True 时表示需要用户批准（展示确认/取消按钮）。"
    ),
)
def ask_user(
    question: str,
    options: list[str] | None = None,
    require_approval: bool = False,
) -> str:
    """向用户提问——实际逻辑由 before_tool_call Hook 接管。"""
    return ""
```

**参数说明**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `question` | `str` | 是 | 向用户展示的问题文本 |
| `options` | `list[str] \| None` | 否 | 可选选项列表，提供后用户可选择一个；不提供则用户自由输入 |
| `require_approval` | `bool` | 否 | 是否请求用户批准（展示批准/拒绝按钮），默认 False |

### 2.2 协作流程状态机

```
Agent 正常执行
    │
    │  Agent 调用 ask_user(question="...", options=[...], require_approval=True)
    ▼
┌─────────────────┐
│  before_tool_call Hook 拦截 ask_user  │
│  → 保存检查点                        │
│  → 抛出 AskUserException             │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  ACPServer 捕获 AskUserException     │
│  → 会话状态 → AWAITING_USER          │
│  → 推送 ASK_USER 事件到客户端        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Web UI 展示问题 / 选项 / 批准按钮    │
│  用户进行回答或选择                   │
└────────┬────────┘
         │
         │  客户端发送 REPLY_USER 请求
         ▼
┌─────────────────┐
│  ACPServer 处理 REPLY_USER           │
│  → 将用户回答作为 ToolResult 注入会话 │
│  → 从检查点恢复执行                   │
└────────┬────────┘
         │
         ▼
Agent 继续执行（收到 ask_user 的 ToolResult，内容为用户回答）
```

### 2.5 AskUserException

专用的异常类型，携带用户问题信息：

```python
class AskUserException(Exception):
    """Agent 向用户提问时抛出的异常。

    由 before_tool_call Hook 中拦截 ask_user 工具调用时抛出，
    被 ACPServer.execute_turn() 捕获并转换为 ASK_USER 事件。
    """
    def __init__(self, question: str, options: list[str] | None = None,
                 require_approval: bool = False, state: dict | None = None):
        self.question = question
        self.options = options
        self.require_approval = require_approval
        self.state = state  # 当前编排状态（用于保存检查点）
        super().__init__(question)
```

#### 3.2.2 `ask_user_hook` — 拦截 Hook

```python
# src/graph_agent/hook/builtin/ask_user_hook.py

from graph_agent.hook.base import hook, HookContext, HookDecision, HookAction, HookType
from graph_agent.acp.checkpoint import InterruptException, AskUserException

@hook(
    checkpoint="before_tool_call",
    priority=10,  # 高优先级，在 tracer 等观测 Hook 之前执行
    hook_type=HookType.CONTROL,
    description="拦截 ask_user 工具调用，中断会话并向用户提问"
)
def ask_user_interceptor(ctx: HookContext) -> HookDecision:
    if ctx.tool_name != "ask_user":
        return HookDecision(action=HookAction.CONTINUE)

    question = ctx.tool_args.get("question", "")
    options = ctx.tool_args.get("options")
    require_approval = ctx.tool_args.get("require_approval", False)

    raise AskUserException(
        question=question,
        options=options,
        require_approval=require_approval,
        state=ctx.agent_state,
    )
```

**优先级 10 的原因**：

- 该 Hook 需要在其他所有 Hook 之前执行（数据校验 Hook 默认 priority=100，Tracer Hook priority=500）
- 确保在日志记录、参数校验等 Hook 之前就拦截 `ask_user`，避免无效操作

#### 3.2.3 ACPServer 改动

**捕获 AskUserException**（在 `execute_turn` 中）：

```python
# 在现有的 InterruptException 捕获块之前新增：

except AskUserException as e:
    # Agent 向用户提问：保存检查点，推送 ASK_USER 事件
    state = e.state or {}
    checkpoint = serialize_checkpoint(state, session_id, "ask_user")
    # 将 ask_user 的调用信息存入检查点
    checkpoint["ask_user_context"] = {
        "question": e.question,
        "options": e.options,
        "require_approval": e.require_approval,
    }
    ctx.checkpoint = checkpoint
    ctx.status = SessionStatus.AWAITING_USER.value
    self._session_manager.save_session(session_id)

    events.extend(collector.drain())
    events.append(ACPMessage.event(PushEvent.ASK_USER, {
        "session_id": session_id,
        "question": e.question,
        "options": e.options,
        "require_approval": e.require_approval,
        "checkpoint_phase": checkpoint.get("phase", ""),
    }))
    return events
```

**处理 REPLY_USER**：

```python
def _handle_reply_user(self, request: ACPMessage) -> ACPMessage:
    """处理用户对 ask_user 的回答。"""
    session_id = request.payload.get("session_id", "")
    reply = request.payload.get("reply", "")
    selected_option = request.payload.get("selected_option")

    if not session_id:
        return ACPMessage.error(ErrorCode.INVALID_REQUEST, "缺少 session_id",
                                request_id=request.id)

    ctx = self._session_manager.get_context(session_id)
    if ctx is None:
        return ACPMessage.error(ErrorCode.SESSION_NOT_FOUND,
                                f"会话 '{session_id}' 不存在", request_id=request.id)

    if ctx.status != SessionStatus.AWAITING_USER.value:
        return ACPMessage.error(ErrorCode.SESSION_NOT_PAUSED,
                                f"会话状态为 {ctx.status}，未在等待用户回复",
                                request_id=request.id)

    if not ctx.checkpoint or "ask_user_context" not in ctx.checkpoint:
        return ACPMessage.error(ErrorCode.CHECKPOINT_INVALID,
                                "检查点中缺少 ask_user 上下文",
                                request_id=request.id)

    # 构建用户回答字符串（注入为 ToolResult）
    ask_ctx = ctx.checkpoint["ask_user_context"]
    response_text = _build_user_reply(ask_ctx, reply, selected_option)

    # 存储用户回答以供 resume 时注入
    ctx.checkpoint["user_reply"] = response_text

    return ACPMessage.response(ResponseEvent.ACK, {
        "reply_accepted": True,
        "session_id": session_id,
    }, request_id=request.id)


def _build_user_reply(ask_ctx: dict, reply: str, selected_option: int | None) -> str:
    """根据问题上下文和用户回答构建回复文本。"""
    if selected_option is not None and ask_ctx.get("options"):
        options = ask_ctx["options"]
        if 0 <= selected_option < len(options):
            return f"用户选择了: {options[selected_option]}"
    if ask_ctx.get("require_approval"):
        return f"用户{'批准' if reply.lower() in ('yes', '是', 'ok', 'approve', '批准', '确认') else '拒绝'}了该操作"
    return reply or "(用户未提供回答)"
```

**恢复执行时注入用户回答**（修改 `execute_turn` 的恢复路径）：

```python
# 在正常模式的恢复路径中，如果检查点包含 user_reply，
# 则在反序列化后将用户回答作为 ToolResult 注入：

    user_reply = ctx.checkpoint.pop("user_reply")
```

### 3.3 `_inject_user_reply` 辅助函数

恢复执行时，需要将用户回答作为 `ask_user` 工具的 ToolResult 注入到消息历史中，这样 Agent 的 LLM 就能看到用户回答：

```python
def _inject_user_reply(state: dict, user_reply: str) -> dict:
    """将用户回答作为 ask_user 的 ToolResult 注入到状态中。

    在 ga_messages 中查找最后一个 ask_user 的 ToolUseBlock，
    然后追加对应的 ToolResultBlock。
    """
    from graph_agent.message import (
        MessageBlock, ContentBlock, ToolResultBlock,
        create_assistant_message,
    )
    from graph_agent.message.message_type import MessageType
    from graph_agent.message.convert import agent_messages_to_langchain
    from langchain_core.messages import ToolMessage

    ga_msgs = state.get("ga_messages", [])
    ask_user_tool_id = None

    # 找到最后一个 ask_user 的 ToolUse
    for msg in reversed(ga_msgs):
        if hasattr(msg, 'content') and isinstance(msg.content, list):
            for block in msg.content:
                if hasattr(block, 'tool_use') and block.tool_use:
                    if block.tool_use.tool_name == "ask_user":
                        ask_user_tool_id = block.tool_use.tool_id
                        break
        if ask_user_tool_id:
            break

    if ask_user_tool_id:
        # 追加 ToolResult 到 ga_messages
        result_block = ContentBlock(
            block_type="tool_result",
            tool_result=ToolResultBlock(
                tool_id=ask_user_tool_id,
                tool_name="ask_user",
                output=user_reply,
                status="success",
            ),
        )
        ga_msgs.append(MessageBlock(
            role="tool",
            content=[result_block],
            message_type=MessageType.TOOL_RESULT,
        ))

        # 同步到 LangChain messages
        lc_messages = state.get("messages", [])
        lc_messages.append(ToolMessage(
            content=user_reply,
            tool_call_id=ask_user_tool_id,
            name="ask_user",
        ))
        state["messages"] = lc_messages

    state["ga_messages"] = ga_msgs
    # 清理 ask_user 上下文（避免重复注入）
    return state
```

## 7. 与现有系统的关系

### 7.1 与现有中断机制的关系

| 维度 | 用户主动中断（现有） | ask_user 中断（新增） |
|------|-------------------|---------------------|
| 触发方 | 用户点击暂停按钮 | Agent 调用 ask_user 工具 |
| 触发方式 | `_handle_interrupt` → `interrupt_event.set()` | Hook 拦截 → `AskUserException` |
| 会话状态 | `PAUSED` | `AWAITING_USER` |
| 检查点 | 保存完整编排状态 | 保存完整编排状态 + `ask_user_context` |
| 恢复方式 | `RESUME_SESSION`（空输入） | `REPLY_USER` + `RESUME_SESSION`（注入 ToolResult） |
| 用户体验 | 被动等待 | 主动交互问答 |

两者共用同一套检查点序列化/反序列化机制，`AWAITING_USER` 是 `PAUSED` 的子类型（语义更精确）。

### 7.2 与 Hook 机制的关系

`ask_user_hook` 是一个优先级为 10 的 Type 2 CONTROL Hook，注册在 `before_tool_call` 检查点。它通过抛出 `AskUserException`（非 `HookDecision(SKIP/ABORT)`）来实现中断，因为：

- `SKIP` 会返回一个假的 ToolResult，Agent 会误认为工具已成功执行
- `ABORT` 会抛出 `HookAbortError`，不携带用户问题信息
- 抛出专用异常 `AskUserException` 可以让 `ACPServer` 精确识别并执行专用恢复流程

### 7.3 与 ACP 协议的关系

| 组件 | 交互方式 |
|------|---------|
| `ASK_USER` 推送事件 | 服务端在捕获 `AskUserException` 后，通过 SSE 推送给客户端 |
| `REPLY_USER` 请求事件 | 客户端通过 HTTP POST `/acp/message` 发送，携带 `reply` 和 `selected_option` |
| `RESUME_SESSION` 请求 | 客户端在收到 REPLY_USER 的 ACK 后发送，触发实际恢复执行 |

---

## 8. 错误处理

### 8.1 异常场景

| 场景 | 处理方式 |
|------|---------|
| 用户未回答就发送 RESUME_SESSION | 返回 `SESSION_NOT_PAUSED` 错误（状态为 AWAITING_USER，非 PAUSED） |
| 检查点中缺少 ask_user_context | 返回 `CHECKPOINT_INVALID` 错误 |
| 会话已过期 | 返回 `SESSION_EXPIRED`，检查点失效 |
| ask_user 工具参数格式错误 | LLM 调用时由函数签名约束，格式错误的调用不会发生 |
| Hook 未被注册 | ask_user 工具的 `func` 返回空字符串，Agent 收到空结果后可能自行处理 |

### 8.2 超时处理

- 会话在 `AWAITING_USER` 状态下的超时与普通 `IDLE` 一致（`session_timeout`）
- 超时后会话进入 `EXPIRED`，检查点被清理
- 用户重新发起任务需创建新会话
