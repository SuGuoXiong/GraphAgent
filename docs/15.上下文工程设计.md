# GraphAgent 上下文工程设计文档

## 1. 概述

GraphAgent 的上下文管理采用**四层分层架构**，每层在上层基础上做减法，向下游 Agent 提供逐步精简的上下文视图。核心原则是：**每个 Agent 只看到完成其职责所需的最小上下文**，避免无关信息稀释 LLM 注意力。

```
┌─────────────────────────────────────────────────────────────────┐
│                      Layer 1: 持久化存储                          │
│              全量消息（含审核/派发/工具调用），磁盘持久化               │
│              职责：完整可追溯，问题排查，审计日志                      │
├─────────────────────────────────────────────────────────────────┤
│                      Layer 2: GuardAgent 战略视图                  │
│              过滤内部消息 + 压缩 + 保留关键语义                        │
│              职责：意图识别、方案审核、结果验收                         │
│              [扩展点] 用户长期偏好                                   │
├─────────────────────────────────────────────────────────────────┤
│                      Layer 3: PlanAgent 战术视图                    │
│              Layer 2 + GuardAgent 意图分析结果                       │
│              职责：任务分解、方案制定、结果汇总                         │
│              [扩展点] Agent 记忆检索                                 │
├─────────────────────────────────────────────────────────────────┤
│                      Layer 4: SubAgent 执行视图                     │
│              仅包含完成任务所需的最小上下文                             │
│              职责：执行具体子任务（工具调用、数据处理等）                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.1 设计动机

当前系统中，所有编排节点（GuardAgent / PlanAgent / SubAgent）共享同一个 `state["messages"]`。这带来三个问题：

1. **上下文膨胀**：编排流程每轮产生 8+ 条内部消息，全部注入 `state["messages"]`，后续节点被迫消费冗余信息
2. **注意力稀释**：SubAgent 执行时收到完整的对话历史（含审核消息、PlanAgent 方案等），但其任务仅需知道"做什么"和"依赖谁的结果"
3. **职责模糊**：各层 Agent 自行调用 `_format_context()` 做硬编码截断（最后 10 条，每条约 500 字符），缺乏统一的分层策略

四层架构将上下文工程从"各 Agent 自行裁剪"升级为"系统级分层注入"，每层 Agent 接收经过针对性精简的上下文。

### 1.2 与现有系统的关系

本文档是对 `docs/6.多轮对话系统设计.md` 的升级和重构。现有系统中的以下模块直接映射到新架构：

| 现有模块 | 映射到新架构 |
|---------|------------|
| `ConversationHistory.messages` | Layer 1（持久化存储） |
| `get_context_messages()` + 压缩器 | Layer 2（GuardAgent 战略视图） |
| `state["messages"]` + `_format_context()` | 重构为 Layer 2/3/4 分层注入 |
| SubAgent `task_description` | 重构为 Layer 4（SubAgent 执行视图） |

## 2. 四层架构详解

### 2.1 Layer 1 —— 持久化存储

**定位**：最底层，保存每次对话中所有 Agent 产生的全部消息，不做任何裁剪。

**数据来源**：
- 用户输入（`USER_INPUT`）
- GuardAgent 输出（`GUARD_INTENT_ANALYSIS`、`GUARD_PLAN_REVIEW`、`GUARD_RESULT_REVIEW`）
- PlanAgent 输出（`PLAN_PROPOSAL`、`PLAN_TASK_DISPATCH`、`PLAN_RESULT_SYNTHESIS`）
- SubAgent 输出（`SUBAGENT_TASK_RESULT`）
- 工具调用记录（`TOOL_CALL`、`TOOL_RESULT`）
- 最终回复（`FINAL_ANSWER`）
- 系统消息（`SYSTEM_PROMPT`、`SYSTEM_NOTIFICATION`）

**存储形式**：`ConversationHistory.messages: list[MessageBlock]`

**职责**：
- 完整可追溯 —— 磁盘上的 JSON 文件包含每轮对话的全部编排流程记录
- 问题排查 —— 出现异常时可回溯完整的 Agent 交互链
- 审计日志 —— 审核消息（`GUARD_PLAN_REVIEW` / `GUARD_RESULT_REVIEW`）的 JSON 内容被完整保留

**实现**：沿用现有 `ConversationHistory` + `ConversationPersistence`，无需修改。

```python
# Layer 1 接口（现有实现，保持不变）
class ConversationHistory:
    session_id: str
    messages: list[MessageBlock]   # 全量，永不裁剪
    turn_count: int

    def add_user_message(self, content: str) -> MessageBlock: ...
    def add_agent_messages(self, messages: list[MessageBlock]) -> None: ...
    def add_final_answer(self, content: str) -> None: ...
```

**数据流出口**：作为 Layer 2 的输入源。

### 2.2 Layer 2 —— GuardAgent 战略视图

**定位**：在 Layer 1 基础上过滤掉内部编排产物，在达到压缩阈值时执行压缩。为 GuardAgent（意图识别、方案审核、结果验收）提供战略级上下文。

**数据来源**：Layer 1 的全量消息。

**处理步骤**：

#### 2.2.1 上下文过滤

沿用现有 `ContextFilter` 规则（`docs/6.多轮对话系统设计.md` 第 6 节），排除以下消息类型：

| 排除的消息类型 | 理由 |
|--------------|------|
| `GUARD_PLAN_REVIEW` | 纯审核 JSON，无语义价值 |
| `GUARD_RESULT_REVIEW` | 纯审核 JSON，无语义价值 |
| `PLAN_TASK_DISPATCH` | 任务派发指令，语义已被 SubAgent 结果覆盖 |
| `TOOL_CALL` | 工具调用细节，语义已被 SubAgent 结果覆盖 |
| `TOOL_RESULT` | 工具调用细节，语义已被 SubAgent 结果覆盖 |

此外，新增过滤规则：**被驳回的中间产物**——未通过 GuardAgent 审核的 `PLAN_PROPOSAL` 和 `PLAN_RESULT_SYNTHESIS` 应当排除。这类消息是已被重新生成的过期版本，保留在上下文中只会造成混淆。

**实现方式**：在 `MessageBlock.metadata` 中增加 `approved` 标记，过滤时检查。

**标记设置时机**：GuardAgent 的 `_review_plan` / `_review_result` 返回审核结果后，由编排节点将审核结论（`approved` / `feedback`）回写到对应的 `PLAN_PROPOSAL` / `PLAN_RESULT_SYNTHESIS` 消息的 `metadata` 中：

```python
# 方案审核通过/驳回后，在 plan_node 或编排 glue 中回写标记
def _mark_plan_review_outcome(
    plan_proposal_msg: MessageBlock,
    review_result: dict,
) -> None:
    """将 GuardAgent 方案审核结论回写到 PLAN_PROPOSAL 消息。"""
    plan_proposal_msg.metadata["approved"] = review_result.get("approved", True)
    plan_proposal_msg.metadata["review_feedback"] = review_result.get("feedback", "")

# 同理，结果验收后回写 PLAN_RESULT_SYNTHESIS 的审核结论
def _mark_result_review_outcome(
    synthesis_msg: MessageBlock,
    review_result: dict,
) -> None:
    """将 GuardAgent 结果验收结论回写到 PLAN_RESULT_SYNTHESIS 消息。"""
    synthesis_msg.metadata["approved"] = review_result.get("approved", True)
    synthesis_msg.metadata["review_feedback"] = review_result.get("feedback", "")
```

**调用时机**：标记动作在 `guard_node` 中执行——当 GuardAgent 完成方案审核或结果验收后，在返回前对对应的提案/汇总消息回写审核结论：

```python
# guard.py guard_node 中的调用模式
def _find_latest_message(state: OrchestrationState, msg_type: MessageType) -> MessageBlock | None:
    """从 ga_messages 中按类型查找最新一条消息。"""
    for m in reversed(state.get("ga_messages", [])):
        if MessageType(m.message_type) == msg_type:
            return m
    return None

def guard_node(state: OrchestrationState) -> dict:
    if phase == OrchestrationPhase.PLAN_REVIEW:
        result = _review_plan(state)
        # 回写：标记被审核的 PLAN_PROPOSAL 消息
        plan_msg = _find_latest_message(state, MessageType.PLAN_PROPOSAL)
        if plan_msg:
            _mark_plan_review_outcome(plan_msg, result)
    elif phase == OrchestrationPhase.RESULT_REVIEW:
        result = _review_result(state)
        # 回写：标记被审核的 PLAN_RESULT_SYNTHESIS 消息
        synth_msg = _find_latest_message(state, MessageType.PLAN_RESULT_SYNTHESIS)
        if synth_msg:
            _mark_result_review_outcome(synth_msg, result)
    ...
```

**过滤逻辑**：对已明确被驳回的提案消息（`approved` 为 `False`）进行排除；未设置标记的历史消息视为通过（向后兼容）。

```python
# 新增过滤逻辑：排除被驳回的中间产物
def _is_rejected_proposal(msg: MessageBlock) -> bool:
    """检查是否为被驳回的方案/汇总消息。"""
    msg_type = MessageType(msg.message_type)
    if msg_type not in (MessageType.PLAN_PROPOSAL, MessageType.PLAN_RESULT_SYNTHESIS):
        return False
    # 仅当明确标记为 False 时才排除；未标记（None）视为通过（兼容历史数据）
    return msg.metadata.get("approved") is False
```

#### 2.2.2 自适应压缩

沿用现有两级压缩机制（`docs/6.多轮对话系统设计.md` 第 7 节）：

```
总 Token 占比        触发动作
────────────────────────────────────────────
< 60%                不压缩
60% ~ 80%            普通压缩：按优先级 P4→P1 删除低价值消息
> 80%                高度压缩：LLM 摘要替换历史消息
```

压缩保护窗口：最近 `keep_recent_turns` 轮对话不参与压缩。

**`extra_messages` 参数的必要性**：GuardAgent 在方案审核（`_review_plan`）和结果验收（`_review_result`）阶段执行时，PlanAgent 刚产生的 `PLAN_PROPOSAL` / `PLAN_RESULT_SYNTHESIS` 消息还在 `state["ga_messages"]` 中，尚未持久化到 `ConversationHistory`。通过 `extra_messages` 参数将这些消息临时纳入上下文构建，确保 GuardAgent 能审核到最新内容，同时不修改 `history`（这些消息将在本轮结束后统一追加）。

**数据流出口**：作为 Layer 3 的输入源。

### 2.3 Layer 3 —— PlanAgent 战术视图

**定位**：在 Layer 2 基础上增加 GuardAgent 意图识别结果。为 PlanAgent（任务分解、方案制定、结果汇总）提供战术级上下文。

**数据来源**：Layer 2 过滤压缩后的消息 + GuardAgent 意图分析结果。

**核心区别**：Layer 2 为 GuardAgent 服务，Layer 3 为 PlanAgent 服务。两者的关键差异在于 Layer 3 额外包含：

1. **GuardAgent 意图分析结果**（`GUARD_INTENT_ANALYSIS`）：提炼后的用户意图、问题类型、复杂度评估。PlanAgent 依赖此结果做任务分解
2. **可用的 SubAgent 清单**：以结构化形式注入，供 PlanAgent 匹配技能

**处理步骤**：

#### 2.3.1 上下文构建

```
Layer 2（GuardAgent 战略视图）
    +
GuardAgent 意图分析结果（当前轮次的 GUARD_INTENT_ANALYSIS）
    +
SubAgent 能力清单（结构化描述）
    =
Layer 3（PlanAgent 战术视图）
```

注意：PlanAgent 不需要看到 `GUARD_PLAN_REVIEW` 和 `GUARD_RESULT_REVIEW`，这些已在 Layer 2 中被过滤。

### 2.4 Layer 4 —— SubAgent 执行视图

**定位**：PlanAgent 完成任务分解后，为每个 SubAgent 注入完成任务所需的**最小上下文**。这是四层架构中裁剪最激进的一层。

**设计原则**：SubAgent 是执行者，不是决策者。它不需要了解完整的对话历史、不需要看到 GuardAgent 的审核过程、不需要知道其他并行 SubAgent 在做什么。它只需要知道：**做什么、用什么做、依赖谁的结果**。

**每个 SubAgent 接收的上下文包含三部分**：

#### 2.4.1 任务详细信息（Task Detail）

- **任务描述** — PlanAgent 分解的具体任务文本（`SubTask.description`）
- **输入参数** — 解析占位符后的 `SubTask.input_data`
- **期望产出** — 任务计划中声明的 `expected_output_format`

#### 2.4.2 依赖的前置输入（Dependency Inputs）

- **前置任务结果** — 当前任务 `dependencies` 列表中所有已完成 SubAgent 的执行结果
- **数据流转标注** — 标明每个前置结果的来源（哪个 SubAgent 产出）和用途（当前任务的哪个参数引用它）

```python
def _format_dependency_inputs(
    task: SubTask,
    all_tasks: list[SubTask],
) -> str:
    """格式化依赖的前置输入。

    为每个依赖项标注来源 Agent 和结果内容，让 SubAgent 理解数据血缘。
    """
    if not task.dependencies:
        return "(无依赖输入)"

    parts = []
    for dep_id in task.dependencies:
        dep_task = next((t for t in all_tasks if t.task_id == dep_id), None)
        if dep_task is None:
            parts.append(f"- [{dep_id}]: (任务不存在)")
            continue
        if dep_task.status != "completed":
            parts.append(f"- [{dep_id}]: (任务状态: {dep_task.status})")
            continue

        agent_name = dep_task.assigned_agent or "unknown"
        result_preview = dep_task.result[:2000] if dep_task.result else "(无结果)"
        parts.append(
            f"### 前置任务 [{dep_id}] (执行者: {agent_name})\n"
            f"**任务描述**: {dep_task.description[:300]}\n"
            f"**执行结果**:\n{result_preview}"
        )
    return "\n\n".join(parts)
```

#### 2.4.3 任务背景的关键描述（Task Background）

- **用户原始意图摘要** — 从 Layer 3 的 `GUARD_INTENT_ANALYSIS` 中提取的一句话意图描述
- **整体目标** — 任务计划的 `overall_goal`，让 SubAgent 理解其工作在全局中的位置
- **上下文片段** — 从 Layer 2 中提取与当前子任务直接相关的历史消息（如用户提供的文件路径、参数值等）

```python
def _extract_relevant_context(
    guard_context: list[MessageBlock],
    task: SubTask,
    max_tokens: int = 1000,
) -> str:
    """从战略视图中提取与当前任务相关的上下文片段。

    策略：从任务描述中提取关键词（支持中文 bigram + 英文空格分词），
    扫描 guard_context 中的 USER_INPUT 和 FINAL_ANSWER 消息，
    提取与关键词有交集的片段。
    """
    keywords = _extract_keywords(task.description)

    relevant_parts = []
    token_budget = max_tokens
    for msg in guard_context:
        msg_type = MessageType(msg.message_type)
        if msg_type not in (MessageType.USER_INPUT, MessageType.FINAL_ANSWER):
            continue
        content = msg.content if isinstance(msg.content, str) else ""
        if not content:
            continue
        overlap = sum(1 for kw in keywords if kw in content.lower())
        if overlap >= 2:
            snippet = content[:500]
            line = f"[{msg_type.value}]: {snippet}"
            relevant_parts.append(line)
            token_budget -= estimate_tokens(line)  # 使用项目已有的 Token 估算
            if token_budget <= 0:
                break

    return "\n---\n".join(relevant_parts) if relevant_parts else "(无特别相关上下文)"


def _extract_keywords(text: str, min_len: int = 2) -> set[str]:
    """从文本中提取关键词。

    混合策略：
    - 英文/数字：按空格分词，过滤短词
    - 中文：使用字符级 bigram 滑动窗口（"分析代码" → {"分析", "析代", "代码"}）
      作为轻量级替代方案，避免引入分词库依赖
    """
    keywords = set()
    # 英文/数字分词
    import re
    tokens = re.findall(r'[a-zA-Z0-9_]{2,}', text)
    keywords.update(t.lower() for t in tokens)

    # 中文字符 bigram
    chinese_chars = re.findall(r'[一-鿿]', text)
    for i in range(len(chinese_chars) - 1):
        keywords.add(chinese_chars[i] + chinese_chars[i + 1])
    # 单字也保留（用于匹配文件路径、工具名等）
    for ch in chinese_chars:
        keywords.add(ch)

    return {k for k in keywords if len(k) >= min_len}
```

> **局限性说明**：当前关键词提取采用简单的规则匹配（英文空格分词 + 中文 bigram），不涉及语义理解。未来可升级为 Embedding 相似度匹配，或调用 LLM 做上下文相关性判断。当前的轻量级方案适合快速实现和低延迟场景。

#### 2.4.4 上下文组装

```python
def build_subagent_context(
    task: SubTask,
    all_tasks: list[SubTask],
    overall_goal: str,
    intent_summary: str,
    guard_context: list[MessageBlock],
    expected_output_format: str = "",
) -> str:
    """构建单个 SubAgent 的执行视图上下文。

    组装四部分内容：
    1. 任务背景
    2. 任务详细信息
    3. 依赖的前置输入
    4. 相关上下文片段
    """
    parts = []

    # 1. 任务背景
    parts.append("## 任务背景")
    parts.append(f"**用户意图**: {intent_summary}")
    parts.append(f"**整体目标**: {overall_goal}")
    if expected_output_format:
        parts.append(f"**期望产出格式**: {expected_output_format}")

    # 2. 任务详细信息
    parts.append("\n## 当前任务")
    parts.append(f"**任务ID**: {task.task_id}")
    parts.append(f"**所需技能**: {task.required_skill}")
    parts.append(f"**任务描述**:\n{task.description}")

    # 3. 依赖的前置输入
    parts.append("\n## 前置任务输出")
    parts.append(_format_dependency_inputs(task, all_tasks))

    # 4. 相关上下文
    parts.append("\n## 相关上下文")
    parts.append(_extract_relevant_context(guard_context, task))

    return "\n".join(parts)
```

#### 2.4.5 接口定义

```python
class SubAgentContextBuilder:
    """Layer 4 上下文构建器 —— SubAgent 执行视图。"""

    def __init__(self, max_context_tokens: int = 8000):
        self._max_context_tokens = max_context_tokens

    def build_for_task(
        self,
        task: SubTask,
        all_tasks: list[SubTask],
        overall_goal: str,
        intent_analysis: MessageBlock,
        guard_context: list[MessageBlock],
        expected_output_format: str = "",
    ) -> str:
        """
        为单个 SubAgent 构建最小执行上下文。

        Returns:
            格式化的上下文文本，作为 SubAgent 的 HumanMessage 注入。
        """
        ...

    def build_for_layer(
        self,
        layer_tasks: list[SubTask],
        all_tasks: list[SubTask],
        overall_goal: str,
        intent_analysis: MessageBlock,
        guard_context: list[MessageBlock],
        expected_output_format: str = "",
    ) -> dict[str, str]:
        """
        为一层 DAG 任务构建执行上下文。

        与 build_all 一次性构建所有任务不同，此方法在每层执行前调用，
        确保当前层的依赖任务结果（来自前序层）已就绪。

        Returns:
            {task_id: context_text} 映射
        """
        contexts = {}
        for task in layer_tasks:
            contexts[task.task_id] = self.build_for_task(
                task=task,
                all_tasks=all_tasks,
                overall_goal=overall_goal,
                intent_analysis=intent_analysis,
                guard_context=guard_context,
                expected_output_format=expected_output_format,
            )
        return contexts
```

> **为什么采用逐层构建（`build_for_layer`）而非一次性构建所有任务（`build_all`）？**
>
> 在多层 DAG 中，第 N 层任务依赖第 N-1 层的结果。如果一次性构建所有任务上下文，第 N 层任务的 `_format_dependency_inputs` 中引用的前置任务结果还是空的（尚未执行）。逐层构建确保每层任务上下文中的"前置任务输出"部分包含最新的已完成结果。

## 3. 编排流程集成

### 3.1 整体数据流

```
                          ┌──────────────────────┐
                          │  ConversationHistory  │
                          │  (Layer 1: 全量存储)   │
                          └──────────┬───────────┘
                                     │
                          ┌──────────▼───────────┐
                          │  GuardContextBuilder  │
                          │  (Layer 2: 过滤+压缩)  │
                          └──────────┬───────────┘
                                     │
                          ┌──────────▼───────────┐
                          │  PlanContextBuilder   │
                          │  +intent +subagents   │
                          │  (Layer 3: 战术视图)   │
                          └──────────┬───────────┘
                                     │
                          ┌──────────▼───────────┐
                          │ SubAgentContextBuilder│
                          │   per-task minimal    │
                          │  (Layer 4: 执行视图)   │
                          └──────────────────────┘
```

### 3.2 编排流程中的调用时序

与方案 1（专用 Builder 节点）不同，本设计采用**Agent 自行构建上下文**的方案：每个 Agent 节点在执行时内部调用对应的 `ContextBuilder`，从当前 state 和 history 中实时构建上下文。这避免了编排循环中上下文过时的问题。

```
1. 用户输入到达
2. 入口层（debug.py / ACP server）将用户输入追加到 history，构建初始 state
3. GuardAgent 节点执行（自行调用 GuardContextBuilder 构建 Layer 2）
   → 产出 intent_analysis
4. PlanAgent 节点执行（自行调用 PlanContextBuilder 构建 Layer 3）
   → 产出 TaskPlan
5. SubAgent 节点执行（自行调用 SubAgentContextBuilder 构建 Layer 4）
   → 并行执行子任务
6. PlanAgent 再次执行（结果汇总，自行构建 Layer 3）
   → 产出 final_answer
7. GuardAgent 再次执行（结果验收，自行构建 Layer 2）
   → 通过则结束，驳回则回到步骤 4 或 6
8. 持久化 Layer 1
```

关键点：步骤 6 和步骤 7 是编排循环的再次进入，GuardAgent 和 PlanAgent 会**重新调用** Builder 构建上下文，因此能看到步骤 3-5 中新产生的消息（plan proposal、subagent results 等）。

### 3.3 与编排节点的集成

核心策略：**每个 Agent 节点在每次执行时自行调用对应的 ContextBuilder 构建上下文**，而非依赖外部节点预先构建。这确保无论编排循环多少次（guard ⇄ plan ⇄ sub_exec），上下文始终是最新的。

#### 3.3.1 GuardAgent 节点改造

**当前**：GuardAgent 调用 `_format_context(state["messages"])` 做硬编码截断（最后 10 条，每条约 500 字符）。

**改造后**：GuardAgent 在执行任何阶段前，内部调用 `GuardContextBuilder.build()` 构建 Layer 2 上下文。

**为什么 Layer 4 在 `subagent_exec_node` 中逐层构建而非一次性构建？**

因为多层 DAG 中，第 2 层任务依赖第 1 层的执行结果。`_format_dependency_inputs` 需要从 `all_tasks` 中读取前置任务的 `result` 字段，而这个字段在任务执行完成后才会被赋值。逐层构建确保每层任务的上下文中"前置任务输出"部分包含最新已完成的结果。

```python
# orchestration/graph.py —— 节点拓扑保持不变
def build_orchestration_graph():
    builder = StateGraph(OrchestrationState)

    builder.add_node("guard", guard_node)
    builder.add_node("plan", plan_node)
    builder.add_node("sub_exec", subagent_exec_node)
```

```python
        """将用户偏好格式化为上下文消息（Layer 2 注入时调用）。"""
        prefs = self.get_preferences(user_id)
        if not prefs:
            return None
        lines = [f"- {k}: {v}" for k, v in prefs.items()]
        return create_system_message(
            content="## 用户长期偏好\n" + "\n".join(lines),
            message_type=MessageType.SYSTEM_NOTIFICATION,
        )
```

**未来实现方向**：
- 存储后端：SQLite / JSON 文件 / 向量数据库
- 偏好发现：从 GuardAgent 的审核反馈中自动提取（"用户已连续 3 次拒绝 PPTX 方案"）
- 偏好衰减：长时间未确认的偏好自动过期

### 6.3 扩展点集成方式

两个扩展点的集成方式一致：在对应 Layer 的 Builder 中持有抽象基类引用，构建上下文时尝试获取扩展信息。当前空壳实现直接返回空结果，不影响主流程。

```python
# GuardContextBuilder.__init__
def __init__(
    self,
    config: SessionConfig,
    preference_store: UserPreferenceStore | None = None,
):
    self._preference_store = preference_store or UserPreferenceStore()  # 默认空壳

# PlanContextBuilder.__init__
def __init__(
    self,
    config: SessionConfig,
    memory_store: AgentMemoryStore | None = None,
):
    self._memory_store = memory_store or AgentMemoryStore()  # 默认空壳
```

这种设计确保：
- **零侵入**：当前不实现扩展功能时，空壳对象直接返回空结果，不影响主流程
- **可替换**：未来只需实现抽象基类的具体子类并注入 Builder 即可激活功能
- **可测试**：单元测试可通过 Mock 注入模拟扩展行为

- Phase 3 依赖 Phase 2（Agent 节点需要 Builder）
- Phase 4 依赖 Phase 3（回写机制需要 Agent 节点已能产出结构化审核结果）

### 与方案 1（专用 Builder 节点）的对比

| 维度 | 方案 1: 图内 Builder 节点 | 方案 2: Agent 自行构建（本方案） |
|------|-------------------------|-------------------------------|
| 编排图复杂度 | 新增 3 个节点 + 修改路由边 | 不改变现有图结构 |
| 上下文新鲜度 | 仅首次进入时构建，循环时过时 | 每次执行重新构建，始终最新 |
| 代码侵入性 | 图结构 + Agent 节点 | 仅 Agent 节点内部 |
| 重复构建开销 | 无（构建一次） | 有（循环时重复构建），可通过 state 缓存避免 |
| 适用性 | 仅适合单向无环流程 | 适合有循环的编排流程 |

当前 GraphAgent 编排图是循环的（guard ⇄ plan ⇄ sub_exec），因此选择方案 2。

## 9. 设计优势

1. **注意力聚焦** —— 每层 Agent 只看到完成职责所需的最小上下文，避免注意力被无关信息稀释
2. **渐进裁剪** —— 四层逐级做减法：全量 → 过滤+压缩 → 追加意图 → 最小执行视图
3. **职责清晰** —— GuardAgent 看战略视图（"用户想做什么"），PlanAgent 看战术视图（"怎么分解任务"），SubAgent 看执行视图（"这个任务怎么做"）
4. **可扩展** —— Layer 2/3 预留了用户偏好和记忆检索扩展点，通过抽象基类 + 空壳实现达到零侵入
5. **复用现有资产** —— Layer 1/2 直接复用现有的 `ConversationHistory`、`ContextFilter`、压缩器等模块，不影响已有功能
6. **可降级** —— 扩展点默认空壳实现，不注入偏好/记忆时系统行为不变，未来激活扩展只需替换实现
