# GraphAgent 三层编排架构设计文档

## 1. 概述

### 1.1 设计灵感

借鉴公司经营管理的三层架构：

| 角色 | 企业类比 | 对应 Agent | 核心职责 |
|------|---------|-----------|---------|
| 高层管理 | CEO/VP | **GuardAgent** | 识别战略意图、审核方案质量、验收最终结果 |
| 中层管理 | 部门经理 | **PlanAgent** | 制定战术方案、调集资源、分配任务、汇总成果 |
| 基层员工 | 技能员工 | **SubAgent** | 执行具体任务、调用工具、反馈结果 |

### 1.2 核心流程

```
用户输入
  → GuardAgent [意图识别]
    → PlanAgent [任务分解 + 方案制定]
      → GuardAgent [方案审核]
        ├─ 驳回 → PlanAgent [修订方案] → GuardAgent [再审]
        └─ 通过 → PlanAgent [派发子任务]
                    ├→ SubAgent_A [执行任务A] → 结果A
                    ├→ SubAgent_B [执行任务B] → 结果B
                    └→ SubAgent_C [执行任务C] → 结果C
                  → PlanAgent [汇总结果 + 生成终稿]
    → GuardAgent [结果审核]
      ├─ 驳回 → PlanAgent [修订] → GuardAgent [再审]
      └─ 通过 → 返回用户
```

### 1.3 设计原则

1. **关注点分离**：每层 Agent 只关注自身职责，不越界
2. **审核闭环**：关键决策点（方案、结果）必须通过 GuardAgent 审核
3. **并行执行**：无依赖的子任务由各自 SubAgent 并行处理
4. **统一消息**：所有 Agent 间通信均使用 `MessageBlock` 格式
5. **渐进增强**：基于现有 `AgentState` 扩展，不破坏已有架构
6. **权限最小化**：每层 Agent 只能看到自身职责所需的信息——GuardAgent 不可见任何 Tool，PlanAgent 不可见 Tool 但可见 SubAgent 技能列表（从配置文件获取），SubAgent 仅可见其系统提示词中声明的 Tool

---

### 1.4 工具可见性与权限模型

三层 Agent 对工具（Tool）和 SubAgent 信息的可见性遵循**最小权限原则**：

| 角色 | 可见 Tool | 可见 SubAgent | 信息来源 |
|------|----------|--------------|----------|
| **GuardAgent** | 不可见、不可调用 | 不可见 | 仅 LLM 自身推理能力 |
| **PlanAgent** | 不可见、不可调用 | 可见（名称、描述、技能标签） | `config/subagents.yaml` 配置文件 |
| **SubAgent** | 仅可见自身系统提示词中声明的 Tool | 不可见其他 SubAgent | 自身系统提示词 + ToolCenter 按需注入 |

**设计意图**：

- **GuardAgent** 是纯粹的质量把关者，依靠 LLM 的语义理解能力审核方案和结果，不需要也不应该接触底层工具实现细节
- **PlanAgent** 是资源调度者，需要知道有哪些 SubAgent 可用（及其技能），以便将任务分解并派发给合适的执行者；但 PlanAgent 不需要知道 SubAgent 内部使用什么工具——这属于执行细节
- **SubAgent** 是执行者，它的能力边界由其系统提示词精确定义：提示词中声明了哪些工具，它就拥有哪些工具。不在提示词中的工具对当前 SubAgent 完全不可见

```
GuardAgent                  PlanAgent                   SubAgent
┌──────────┐           ┌──────────────────┐       ┌──────────────┐
│ 只看不摸  │           │ 看 SubAgent 清单   │       │ 只用自己的工具 │
│ 无 Tool  │  ──────▶  │ 无 Tool 访问      │ ────▶ │ 工具在提示词中 │
│ 纯推理   │           │ 从配置文件读取     │       │ 动态创建实例   │
└──────────┘           └──────────────────┘       └──────────────┘
```

## 3. 状态设计

### 3.1 OrchestrationState

编排流程需要比基础 `AgentState` 更丰富的状态信息。在 `AgentState` 基础上扩展：

```python
# src/graph_agent/orchestration/state.py

from typing import Annotated, TypedDict
from dataclasses import dataclass, field
from enum import Enum
import operator

from graph_agent.state import AgentState
from graph_agent.message import MessageBlock


class OrchestrationPhase(Enum):
    """编排流程所处阶段"""
    INTENT_ANALYSIS = "intent_analysis"       # GuardAgent 分析用户意图
    PLAN_GENERATION = "plan_generation"       # PlanAgent 制定方案
    PLAN_REVIEW = "plan_review"               # GuardAgent 审核方案
    TASK_EXECUTION = "task_execution"         # SubAgent 并行执行子任务
    RESULT_SYNTHESIS = "result_synthesis"     # PlanAgent 汇总结果
    RESULT_REVIEW = "result_review"           # GuardAgent 审核最终结果
    COMPLETED = "completed"                   # 任务完成


@dataclass
class TaskPlan:
    """PlanAgent 生成的任务计划"""
    plan_id: str                                    # 计划唯一ID
    overall_goal: str                               # 总体目标描述
    sub_tasks: list[SubTask]                         # 子任务列表
    execution_strategy: str                          # 执行策略说明（串行/并行）
    expected_output_format: str                      # 期望输出格式
    created_at: str = ""                             # 创建时间戳


@dataclass
class SubTask:
    """单个子任务定义"""
    task_id: str                                     # 子任务ID
    description: str                                 # 任务描述
    required_skill: str                              # 所需技能名称
    assigned_agent: str = ""                         # 被分配的 SubAgent 名称
    input_data: dict = field(default_factory=dict)   # 输入数据
    dependencies: list[str] = field(default_factory=list)  # 依赖的其他子任务ID
    status: str = "pending"                          # pending / running / completed / failed
    result: str = ""                                 # 执行结果
    error: str = ""                                  # 错误信息


class OrchestrationState(AgentState):
    """编排状态，继承 AgentState 的双通道消息机制"""
    # 编排流程控制
    phase: OrchestrationPhase                                 # 当前阶段
    intent: str                                               # GuardAgent 识别的用户意图
    guard_feedback: str                                       # GuardAgent 的审核反馈

    # 任务计划
    task_plan: TaskPlan | None                                # 当前任务计划

    # SubAgent 执行结果收集
    sub_results: Annotated[dict[str, str], operator.or_]      # {task_id: result}（并行写入用 or_ reducer）

    # 审核状态
    plan_approved: bool                                       # 方案是否通过审核
    result_approved: bool                                     # 结果是否通过审核
    review_retries: int                                       # 审核重试次数
    max_review_retries: int                                   # 最大审核重试次数
```

## 4. 提示词系统设计

### 4.1 设计理念

提示词与代码分离，存放在项目根目录的 `prompts/` 目录下，以纯文本文件（`.txt`）形式管理。程序运行时通过 `PromptLoader` 从文件系统读取，实现**热加载**能力——修改提示词无需重启服务。

### 4.2 目录结构

```
prompts/                              # 提示词根目录（可配置路径）
├── guard/                            # GuardAgent 专用提示词
│   ├── intent_analysis.txt           # [意图识别] 分析用户需求的提示词
│   ├── plan_review.txt               # [方案审核] 审核 PlanAgent 方案的提示词
│   └── result_review.txt             # [结果审核] 验收最终结果的提示词

│   ├── plan_generation.txt           # [方案制定] 将意图分解为子任务的提示词
│   ├── task_dispatch.txt             # [任务派发] 匹配 SubAgent 并派发任务的提示词
│   └── result_synthesis.txt          # [结果汇总] 汇总各 SubAgent 结果的提示词
```

### 4.3 PromptLoader 设计

`PromptLoader` 负责从 `prompts/` 目录加载提示词文本，提供缓存、变量插值、默认值回退等能力。

```python
# src/graph_agent/orchestration/prompt_loader.py

import os
from pathlib import Path
from typing import Optional


class PromptLoader:
    """从文件系统加载 Agent 提示词。

    使用方式:
        loader = PromptLoader()
        intent_prompt = loader.load("guard", "intent_analysis")
        plan_prompt  = loader.load("plan", "plan_generation")
        sub_prompt   = loader.load("subagent", "file_agent")
    """

    # 默认提示词（当文件不存在时回退使用）
    _FALLBACKS: dict[str, dict[str, str]] = {
        "guard": {
            "intent_analysis": "你是一个 GuardAgent，负责准确理解用户需求并提炼核心意图。",
            "plan_review": "你是一个 GuardAgent，负责审核执行方案的合理性和完整性。",
            "result_review": "你是一个 GuardAgent，负责验收最终结果是否符合用户期望。",
        },
        "plan": {
            "plan_generation": "你是一个 PlanAgent，负责将任务分解为可执行的子任务。",
            "task_dispatch": "你是一个 PlanAgent，负责为每个子任务匹配对应的 SubAgent 并派发。",
            "result_synthesis": "你是一个 PlanAgent，负责汇总各 SubAgent 的执行结果。",
        },
    }

    def __init__(self, prompts_root: str | None = None):
        """
        Args:
            prompts_root: 提示词根目录路径。默认为项目根目录下的 prompts/。
                         可通过环境变量 PROMPTS_ROOT 覆盖。
        """
        if prompts_root is None:
            prompts_root = os.getenv("PROMPTS_ROOT")
        if prompts_root is None:
            # 默认：项目根目录下的 prompts/
            project_root = Path(__file__).parent.parent.parent.parent
            prompts_root = str(project_root / "prompts")
        self._root = Path(prompts_root)
        self._cache: dict[str, str] = {}

    def load(self, agent_type: str, prompt_name: str) -> str:
        """加载指定 Agent 的提示词。

        Args:
            agent_type: Agent 类型（"guard" / "plan" / "subagent"）
            prompt_name: 提示词名称（不含 .txt 后缀）

        Returns:
            提示词文本内容

        加载优先级：
            1. 从 prompts/{agent_type}/{prompt_name}.txt 文件读取
            2. 文件不存在时使用内置 _FALLBACKS 默认值
            3. 无默认值时抛出 FileNotFoundError
        """
        cache_key = f"{agent_type}/{prompt_name}"
        if cache_key in self._cache:
            return self._cache[cache_key]

        file_path = self._root / agent_type / f"{prompt_name}.txt"
        if file_path.exists():
            content = file_path.read_text(encoding="utf-8")
            self._cache[cache_key] = content
            return content

        # 回退到内置默认值
        fallback = self._get_fallback(agent_type, prompt_name)
        self._cache[cache_key] = fallback
        return fallback

    def load_with_context(self, agent_type: str, prompt_name: str,
                          **variables: str) -> str:
        """加载提示词并用变量插值填充。

        提示词中使用 {variable_name} 占位符，调用时传入对应变量值。

        Example:
            loader.load_with_context(
                "subagent", "file_agent",
                task_description="读取文件并提取数字",
                available_tools="read_file, calculator",
            )
        """
        template = self.load(agent_type, prompt_name)
        return template.format(**variables)

    def _get_fallback(self, agent_type: str, prompt_name: str) -> str:
        prompts = self._FALLBACKS.get(agent_type, {})
        if prompt_name in prompts:
            return prompts[prompt_name]
        raise FileNotFoundError(
            f"提示词文件不存在且无默认值: prompts/{agent_type}/{prompt_name}.txt"
        )

    def clear_cache(self) -> None:
        """清除缓存，强制下次加载时重新读取文件（热加载）。"""
        self._cache.clear()

    def list_all(self) -> dict[str, list[str]]:
        """列出 prompts/ 目录下所有可用的提示词。"""
        result: dict[str, list[str]] = {}
        if not self._root.exists():
            return result
        for agent_dir in self._root.iterdir():
            if agent_dir.is_dir():
                prompts = [
                    p.stem for p in agent_dir.glob("*.txt")
                ]
                result[agent_dir.name] = prompts
        return result
```

### 4.4 提示词文件格式

每个 `.txt` 文件包含完整的系统提示词，使用 `{variable}` 占位符标记需要动态注入的变量。以 `prompts/guard/intent_analysis.txt` 为例：

```
你是一个 GuardAgent（质量把关者），负责准确理解用户需求并提炼核心意图。

## 职责
1. 分析用户输入，识别用户真正想要完成的目标
2. 判断任务复杂度：简单任务可直接给出答案，复杂任务需要制定执行方案
3. 提炼意图为一句话描述，传递给 PlanAgent 进行任务分解

## 输出格式
请以 JSON 格式输出你的分析结果:
{{
  "intent": "用户意图的一句话描述",
  "complexity": "simple | medium | complex",
  "key_requirements": ["需求点1", "需求点2"],
  "notes": "补充说明（如有）"
}}

## 当前对话上下文
{conversation_context}
```

### 4.5 提示词变量插值机制

提示词中可以包含 `{variable_name}` 占位符，运行时由 `load_with_context()` 填充：

| 变量名 | 来源 | 适用 Agent | 说明 |
|--------|------|-----------|------|
| `{conversation_context}` | `state["messages"]` | Guard / Plan | 当前对话历史 |
| `{task_description}` | `SubTask.description` | SubAgent | 子任务描述 |
| `{available_tools}` | SubAgent 系统提示词中声明 | SubAgent | 当前 SubAgent 可用的工具列表（不在提示词中声明的工具不可见） |
| `{available_subagents}` | `config/subagents.yaml` | PlanAgent | 所有 SubAgent 的名称、描述和技能标签（不含工具信息） |
| `{agent_name}` | `SubAgentConfig.name` | SubAgent | SubAgent 名称 |
| `{guard_feedback}` | `state["guard_feedback"]` | Guard / Plan | GuardAgent 审核反馈 |
| `{sub_results}` | `state["sub_results"]` | PlanAgent | 已完成的子任务结果 |

### 4.6 SubAgent 提示词示例（含工具声明）

SubAgent 的系统提示词中需要声明其可用的工具列表，这是该 SubAgent 的**能力边界**——未在提示词中声明的工具对该 SubAgent 不可见。工具列表通过 `{available_tools}` 变量注入。

以 `prompts/subagent/file_agent.txt` 为例：

```
你是一个 FileAgent（文件系统操作专家）。

## 可用工具
你可以使用以下工具来完成任务。你不可以使用未在以下列表中的工具：
{available_tools}

## 职责
1. 根据任务描述，选择合适的工具完成文件系统操作
2. 读取文件后，准确理解和提取文件内容中的信息
3. 写入文件后，确认写入操作是否成功
4. 如果操作失败，清晰说明失败原因

## 任务上下文
任务描述: {task_description}
```

### 4.7 与 Agent 初始化的集成

每个 Agent 在节点首次执行时通过 `PromptLoader` 加载对应的系统提示词，作为 LLM 调用的 `SystemMessage` 注入。

```python

# GuardAgent 加载意图分析提示词（不注入任何工具信息）
```

## 5. Agent 设计

### 5.1 GuardAgent

**职责**：意图识别 → 方案审核 → 结果验收，全程把控质量

**工具权限**：GuardAgent **不可见、不可调用任何 Tool**。它仅依赖 LLM 的语义理解和推理能力来完成意图分析、方案审核和结果验收，不接触任何底层工具实现。

**提示词加载**：

```python
# src/graph_agent/orchestration/guard.py

from graph_agent.orchestration.prompt_loader import PromptLoader

_prompt_loader = PromptLoader()

def _get_guard_system_prompt(state: OrchestrationState) -> str:
    phase = state["phase"]
    prompt_name = {
        OrchestrationPhase.INTENT_ANALYSIS: "intent_analysis",
        OrchestrationPhase.PLAN_REVIEW: "plan_review",
        OrchestrationPhase.RESULT_REVIEW: "result_review",
    }[phase]
    return _prompt_loader.load_with_context(
        "guard", prompt_name,
        conversation_context=_format_context(state["messages"]),
    )
```

**节点实现**：

```python
# src/graph_agent/orchestration/guard.py

def guard_node(state: OrchestrationState) -> dict:
    """GuardAgent 节点：根据当前 phase 执行对应职责"""
    phase = state["phase"]

    if phase == OrchestrationPhase.INTENT_ANALYSIS:
        return _analyze_intent(state)
    elif phase == OrchestrationPhase.PLAN_REVIEW:
        return _review_plan(state)
    elif phase == OrchestrationPhase.RESULT_REVIEW:
        return _review_result(state)
    else:
        return {}
```

**路由逻辑**：

```python
def guard_router(state: OrchestrationState) -> str:
    """GuardAgent 的路由决策"""
    phase = state["phase"]

    if phase == OrchestrationPhase.INTENT_ANALYSIS:
        return "plan"              # 意图分析完成 → PlanAgent 制定方案

    if phase == OrchestrationPhase.PLAN_REVIEW:
        if state["plan_approved"] or state["review_retries"] >= state["max_review_retries"]:
            return "plan"          # 方案通过或超过重试 → PlanAgent 执行
        return "plan"              # 方案驳回 → PlanAgent 修订

    if phase == OrchestrationPhase.RESULT_REVIEW:
        if state["result_approved"] or state["review_retries"] >= state["max_review_retries"]:
            return END             # 结果通过 → 返回用户
        return "plan"              # 结果驳回 → PlanAgent 修订

    return END
```

### 5.2 PlanAgent

**职责**：任务分解 → 方案制定 → 派发执行 → 结果汇总

**工具权限**：PlanAgent **不可见、不可调用任何 Tool**。它只能看到从配置文件 `config/subagents.yaml` 中加载的 SubAgent 清单（名称、描述、技能标签），不能看到 SubAgent 内部使用了哪些工具。PlanAgent 根据技能标签将子任务匹配给对应的 SubAgent，不关心执行细节。

```python
def _dispatch_tasks(state: OrchestrationState) -> dict:
    """根据执行策略派发子任务"""
    task_plan = state["task_plan"]
    sub_tasks = task_plan.sub_tasks

    # 检查是否可以并行执行
    is_parallel = task_plan.execution_strategy == "parallel"

    dispatch_messages = []
    for task in sub_tasks:
        if task.status == "pending":
            # 检查依赖是否满足
            deps_met = all(
                state["sub_results"].get(dep_id) is not None
                for dep_id in task.dependencies
            )
            if deps_met:
                task.status = "running"
                dispatch_messages.append(
                    _create_task_dispatch_message(task)
                )

    return {
        "task_plan": task_plan,            # 更新任务状态
        "ga_messages": dispatch_messages,  # 派发消息
    }
```

### 5.3 SubAgent

**职责**：执行被分配的子任务，调用工具或自身技能完成工作

**工具权限**：SubAgent **仅能看见和使用其系统提示词中声明的 Tool**。不在提示词中的工具对当前 SubAgent 完全不可见。工具列表从配置文件 `config/subagents.yaml` 中读取，运行时注入到提示词中。

**设计要点**：

- SubAgent 的元数据（名称、描述、技能标签、工具列表、提示词文件名）全部来自配置文件

- SubAgent 实例在程序运行过程中根据配置**动态创建**
- PlanAgent 通过 `required_skill` 匹配合适的 SubAgent
- 每个 SubAgent 在自己的 ReAct 循环中独立完成子任务

**SubAgent 注册中心**（从配置文件加载）：

```python
# src/graph_agent/orchestration/subagent.py

import os
import yaml
from pathlib import Path
from dataclasses import dataclass, field
from graph_agent.orchestration.prompt_loader import PromptLoader


@dataclass
class SubAgentConfig:
    """SubAgent 配置——从 config/subagents.yaml 加载，不在代码中硬编码"""
    name: str                             # SubAgent 唯一名称
    description: str                      # 能力描述（供 PlanAgent 理解，写入 PlanAgent 提示词）
    skills: list[str]                     # 技能标签列表（供 PlanAgent 匹配）
    tools: list[str]                      # 可使用的工具名称列表（运行时注入 SubAgent 提示词）
    prompt_file: str                      # 提示词文件名（相对于 prompts/subagent/）
    llm_config: dict | None = None        # 可选独立 LLM 配置
    max_iterations: int = 5               # ReAct 最大迭代次数

    def load_system_prompt(self, loader: PromptLoader, **variables) -> str:
        """从 prompts/subagent/{prompt_file} 加载系统提示词并填充变量。

        工具列表通过 {available_tools} 变量注入提示词，
        不在提示词中声明的工具对当前 SubAgent 不可见。
        """
        variables.setdefault("available_tools", ", ".join(self.tools))
        return loader.load_with_context("subagent", self.prompt_file, **variables)


class SubAgentRegistry:
    """SubAgent 注册中心——从 config/subagents.yaml 配置文件加载。

    支持热加载：调用 reload() 可重新读取配置文件，无需重启服务。
    """

    def __init__(self, config_path: str | None = None, prompt_loader: PromptLoader | None = None):
        self._agents: dict[str, SubAgentConfig] = {}
        self._prompt_loader = prompt_loader or PromptLoader()
        self._config_path = config_path or self._default_config_path()
        self._load_from_config()

    @staticmethod
    def _default_config_path() -> str:
        project_root = Path(__file__).parent.parent.parent.parent
        return str(project_root / "config" / "subagents.yaml")

    def _load_from_config(self) -> None:
        """从 YAML 配置文件加载 SubAgent 定义。"""
        with open(self._config_path, "r", encoding="utf-8") as f:
            data = yaml.safe_load(f)
        self._agents.clear()
        for item in data.get("subagents", []):
            config = SubAgentConfig(
                name=item["name"],
                description=item["description"],
                skills=item["skills"],
                tools=item["tools"],
                prompt_file=item["prompt_file"],
                max_iterations=item.get("max_iterations", 5),
                llm_config=item.get("llm_config"),
            )
            self._agents[config.name] = config

    def reload(self) -> None:
        """热加载：重新读取配置文件。"""
        self._prompt_loader.clear_cache()
        self._load_from_config()

    def get(self, name: str) -> SubAgentConfig:
        return self._agents[name]

    def find_by_skill(self, skill: str) -> list[SubAgentConfig]:
        """根据技能标签查找匹配的 SubAgent。"""
        return [agent for agent in self._agents.values() if skill in agent.skills]

    def list_all(self) -> list[SubAgentConfig]:
        return list(self._agents.values())

    def describe_all_for_llm(self) -> str:
        """生成供 PlanAgent 使用的 SubAgent 清单文本（仅含名称、描述、技能，不含工具）。

        PlanAgent 通过此文本了解可用的 SubAgent 及其能力，
        但它看不到也无需知道 SubAgent 内部使用什么工具。
        """
        lines = []
        for agent in self._agents.values():
            skills_str = ", ".join(agent.skills)
            lines.append(f"- **{agent.name}**: {agent.description}（技能: {skills_str}）")
        return "\n".join(lines)
```

**SubAgent 执行流程**（ReAct 循环）：

```
1. 接收任务消息 (MessageBlock, message_type="task_assignment")
2. 解析任务描述和输入数据
3. 通过 PromptLoader 加载对应提示词文件 (prompts/subagent/{name}.txt)
4. 构造 SystemMessage + 任务上下文 → 开启 ReAct 循环:
   a. LLM 分析当前状态，决定下一步行动
   b. 如需调用工具 → 执行工具 → 获得结果
   c. 判断任务是否完成
   d. 未完成且未超迭代 → 回到 a
5. 返回结果 MessageBlock (message_type="task_result")
```

**SubAgent 路由节点**：

```python
def subagent_router(state: OrchestrationState) -> list[dict]:
    """
    根据 task_plan 中的 sub_tasks，为每个 running 状态的任务
    创建对应的 SubAgent 执行子图调用
    """
    task_plan = state["task_plan"]
    commands = []

    for task in task_plan.sub_tasks:
        if task.status == "running" and not task.result:
            # 查找匹配的 SubAgent
            candidates = registry.find_by_skill(task.required_skill)
            if candidates:
                task.assigned_agent = candidates[0].name
                commands.append({
                    "command": f"go_{candidates[0].name}",
                    "update": {
                        "sub_tasks": task_plan.sub_tasks,
                    }
                })

    return commands
```

## 6. 编排图设计

### 6.1 图结构

```
graph LR
    START --> guard_intent[GuardAgent<br/>意图识别]
    guard_intent --> plan_generate[PlanAgent<br/>制定方案]
    plan_generate --> guard_review[GuardAgent<br/>方案审核]
    guard_review -->|驳回| plan_generate
    guard_review -->|通过| plan_dispatch[PlanAgent<br/>派发任务]
    plan_dispatch --> subagent_exec[SubAgent<br/>并行执行]
    subagent_exec --> plan_synthesize[PlanAgent<br/>汇总结果]
    plan_synthesize --> guard_result[GuardAgent<br/>结果审核]
    guard_result -->|驳回| plan_synthesize
    guard_result -->|通过| END
```

### 6.2 节点定义

| 节点名 | 函数 | 触发条件 | 职责 |
|--------|------|---------|------|
| `guard` | `guard_node` | 始终（多阶段复用） | 根据 `phase` 执行意图分析/方案审核/结果审核 |
| `plan` | `plan_node` | Guard 或 SubAgent 完成后 | 根据 `phase` 执行方案制定/任务派发/结果汇总 |
| `sub_exec` | `subagent_exec_node` | PlanAgent 派发任务后 | 并行调用各 SubAgent 执行子任务 |

### 6.3 构建代码

```python
# src/graph_agent/orchestration/graph.py

from langgraph.graph import StateGraph, START, END
from graph_agent.orchestration.state import OrchestrationState, OrchestrationPhase
from graph_agent.orchestration.guard import guard_node, guard_router
from graph_agent.orchestration.plan import plan_node, plan_router
from graph_agent.orchestration.subagent import subagent_exec_node, SubAgentRegistry


# 全局单例：从 config/subagents.yaml 加载 SubAgent 注册信息
_registry = SubAgentRegistry()


def build_orchestration_graph() -> StateGraph:
    builder = StateGraph(OrchestrationState)

    builder.add_node("guard", guard_node)
    builder.add_node("plan", plan_node)
    builder.add_node("sub_exec", subagent_exec_node)

    builder.add_edge(START, "guard")

    # Guard → Plan / END
    builder.add_conditional_edges("guard", guard_router, {
        "plan": "plan",
        END: END,
    })

    # Plan → Guard / SubAgents
    builder.add_conditional_edges("plan", plan_router, {
        "guard": "guard",
        "sub_exec": "sub_exec",
        END: END,
    })

    # SubAgents → Plan (结果汇总)
    builder.add_edge("sub_exec", "plan")

    return builder.compile()
```

## 7. Skill 系统设计

### 7.1 设计思路

Skill 是对 SubAgent 能力的抽象描述。一个 SubAgent 可以拥有多个 Skill，PlanAgent 在分配任务时根据 `required_skill` 进行匹配。

Skill 与 Tool 的关系：
- **Tool**：原子级能力，如 `read_file`、`calculator`
- **Skill**：组合级能力，如 `data_analysis`（可能包含 read_file + calculator + 等等）
- 一个 Skill 可以映射到多个 Tool，也可以包含 SubAgent 的内部推理能力

### 7.2 Skill 定义


### 9.1 Agent 间消息格式

所有 Agent 间通信均使用 `MessageBlock`，通过 `message_type` 区分消息语义：

|----------------|-------------|---------|

| PlanAgent → SubAgent | `task_assignment` | 子任务描述 + 输入数据 |
| SubAgent → PlanAgent | `task_result` | 子任务执行结果 |
| PlanAgent → GuardAgent | `final_result` | 汇总后的最终结果 |
| GuardAgent → 用户 | `agent_response` | 最终回复内容 |

## 12. 关键设计决策

### 12.1 为什么 GuardAgent 审核用 LLM 而非代码规则？

LLM 审核能理解语义上的质量标准（答案是否完整、逻辑是否合理），代码规则只能做形式检查。初期全部用 LLM 审核，后续可在 GuardAgent 中混合代码规则（如格式校验）以降低 token 消耗。

### 12.2 为什么 SubAgent 可以有自己的 LLM 配置？

不同 SubAgent 的任务复杂度差异大。简单的 `time_agent` 可用轻量模型（如 gpt-3.5-turbo），复杂的 `code_agent` 可用强模型（如 claude-opus），以优化成本。

### 12.3 审核超时的兜底策略？

`max_review_retries`（默认 3）防止 GuardAgent 和 PlanAgent 陷入无限审核循环。超过重试次数后，GuardAgent 接受当前方案/结果并继续。

### 12.4 子任务中的依赖处理？

`SubTask.dependencies` 定义了任务间的依赖关系。PlanAgent 在派发时检查依赖状态，仅当依赖任务完成（`sub_results` 中有结果）后才派发后续任务。无依赖的任务可并行执行。

### 12.5 为什么 SubAgent 配置从 YAML 文件加载而非代码硬编码？

随着工具模块不断扩展，SubAgent 的数量和种类也会持续增长。配置文件驱动的注册方式使得新增 SubAgent 无需修改任何 Python 代码——只需在 `config/subagents.yaml` 中添加一行记录，并写好对应的提示词文件即可。这极大降低了扩展门槛，便于团队协作和快速迭代。

### 12.6 为什么 SubAgent 的工具可见性由系统提示词控制？

将工具列表写入系统提示词具有双重作用：
1. **为 LLM 提供上下文**：LLM 需要知道自己有哪些工具可用，提示词是天然的信息通道
2. **作为权限边界**：不在提示词中声明的工具，LLM 不会尝试调用，天然实现了最小权限
这种方式无需额外的权限检查基础设施，LLM 自身的行为约束就是第一道防线。配合 ToolCenter 级别的过滤（仅注入声明的工具），构成双重保障。

### 12.7 为什么 PlanAgent 不能看到工具？

PlanAgent 的职责是"调兵遣将"（任务分解与派发），而不是"亲自上阵"（执行具体操作）。让 PlanAgent 看不见工具，可以强制其专注于编排职能，避免"PlanAgent 自己动手把活干了"的情况。这也使得 SubAgent 的内部实现可以自由变更（更换工具、调整策略），而 PlanAgent 不受影响——只要技能标签不变即可。
