# GraphAgent 会话进程隔离设计文档

## 1. 概述

### 1.1 背景与动机

当前 GraphAgent ACP Server 采用单进程架构，所有会话在同一个进程中以协程方式并发处理。这种设计存在以下问题：

| 问题 | 说明 |
|------|------|
| **资源竞争** | 多个会话共享同一进程的 CPU、内存和 GIL（Python 全局解释器锁），高负载时互相影响 |
| **故障隔离缺失** | 一个会话的异常（如内存泄漏、死循环、LLM 调用阻塞）可能影响所有其他会话 |
| **资源回收不精细** | 无法按会话粒度释放资源，只能整体重启服务 |
| **扩展受限** | 单进程无法利用多核优势，也无法将不同会话分布到不同机器 |

**核心目标**：实现以进程为粒度的会话隔离，每个会话在独立的 Worker 进程中运行，主进程负责路由和管理，实现故障隔离和精细化资源控制。

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **进程级隔离** | 每个会话运行在独立的 Worker 进程中，互不干扰 |
| **故障隔离** | 单个会话崩溃不影响其他会话和主服务 |
| **自动回收** | 检测到会话长时间无响应后，保存会话状态并释放进程资源 |
| **透明迁移** | 对上层调用方（Transport 层、Web UI）透明，不改变 ACP 协议 |
| **优雅降级** | Worker 进程异常退出时自动保存检查点，支持后续恢复 |
| **跨平台** | 支持 Windows、Linux、macOS |

### 2.2 进程角色

#### 2.2.1 Master Process（主进程）

**职责：**

1. **请求路由**：接收 Transport 层的请求，根据 `session_id` 路由到对应的 Worker 进程
2. **进程池管理**：创建、监控、销毁 Worker 进程
3. **IPC 消息分发**：维护与每个 Worker 的 IPC 通道，发送指令、接收事件
4. **健康监控**：周期性地检查 Worker 进程的心跳，检测僵死/无响应进程
5. **会话持久化协调**：Worker 退出时触发会话状态保存
6. **SSE 事件转发**：将 Worker 进程产生的事件转发到对应的 SSE 客户端队列

**不负责：**

- 实际的 LLM 调用和 Graph 执行（由 Worker 完成）
- 会话历史管理（由 Worker 内部持有，持久化文件由双方共享访问）

#### 2.2.2 Worker Process（工作进程）

**职责：**

1. **Graph 执行**：运行 `OrchestrationGraph.ainvoke()` 处理用户消息
2. **会话状态管理**：持有 `ConversationHistory` 和 `ConversationContext`
3. **事件上报**：通过 IPC 通道向 Master 实时上报流式事件（阶段变化、LLM 流、工具调用等）
4. **心跳响应**：定期响应 Master 的心跳检测
5. **检查点管理**：处理 `InterruptException` / `AskUserException` 时保存检查点
6. **优雅退出**：收到 `SHUTDOWN` 指令后保存状态再退出

**不负责：**

- 直接与客户端通信（不感知 HTTP/SSE）
- 会话的磁盘持久化调度（由 Master 统一调度，Worker 执行实际写入）

## 3. IPC 通信设计

### 3.1 通道设计

基于 Python `multiprocessing` 模块，每个 Worker 进程与 Master 之间建立**双队列通道**：

```
Master                              Worker
  │                                    │
  │  ┌─ command_queue ──────────────►  │  (Master → Worker)
  │  │   指令: EXECUTE, SHUTDOWN,       │
  │  │         HEARTBEAT, CONFIGURE     │
  │  │                                  │
  │  ◄─ event_queue ────────────────┐  │  (Worker → Master)
  │      事件: STREAM_CHUNK,         │  │
  │            TOOL_CALL,            │  │
  │            PHASE_CHANGE,         │  │
  │            FINAL_ANSWER,         │  │
  │            HEARTBEAT_ACK,        │  │
  │            ERROR                 │  │
  │                                    │
```

**选择 `multiprocessing.Queue` 的理由：**

- Python 标准库，无需额外依赖
- 支持跨平台（Windows / Linux / macOS）
- 内置序列化和流控
- 与 `asyncio` 可通过 `asyncio.to_thread` 桥接

### 3.2 IPC 消息协议

```python
# src/graph_agent/acp/ipc/protocol.py

from dataclasses import dataclass, field
from enum import Enum
from typing import Any


class IPCCommand(Enum):
    """Master → Worker 指令类型"""
    EXECUTE = "execute"           # 执行一轮对话
    SHUTDOWN = "shutdown"         # 优雅关闭（保存状态后退出）
    FORCE_KILL = "force_kill"     # 强制终止
    HEARTBEAT = "heartbeat"       # 心跳探测
    CONFIGURE = "configure"       # 更新配置


class IPCEvent(Enum):
    """Worker → Master 事件类型"""
    STREAM_CHUNK = "stream_chunk"       # LLM 流式 token
    STREAM_START = "stream_start"       # LLM 流开始
    STREAM_END = "stream_end"           # LLM 流结束
    TOOL_CALL = "tool_call"             # 工具调用
    TOOL_RESULT = "tool_result"         # 工具结果
    PHASE_CHANGE = "phase_change"       # 编排阶段切换
    AGENT_MESSAGE = "agent_message"     # Agent 消息
    FINAL_ANSWER = "final_answer"       # 最终回复
    EXECUTION_COMPLETE = "execution_complete"   # 执行完成
    EXECUTION_PAUSED = "execution_paused"       # 执行暂停（中断）
    EXECUTION_RESUMED = "execution_resumed"     # 执行恢复
    ASK_USER = "ask_user"               # 向用户提问
    HEARTBEAT_ACK = "heartbeat_ack"     # 心跳应答
    ERROR = "error"                     # 错误
    WORKER_READY = "worker_ready"       # Worker 初始化完成


@dataclass
class IPCMessage:
    """IPC 消息的标准化结构"""
    id: str                              # 消息唯一 ID
    type: str                            # "command" 或 "event"
    name: str                            # IPCCommand 或 IPCEvent 的值
    session_id: str                      # 关联的会话 ID
    timestamp: str                       # ISO 8601 UTC 时间戳
    payload: dict[str, Any] = field(default_factory=dict)
    error: str | None = None             # 错误信息（仅在 type="event" 且 name="error" 时）

    @classmethod
    def command(cls, cmd: IPCCommand, session_id: str, payload: dict | None = None) -> "IPCMessage":
        ...

    @classmethod
    def event(cls, evt: IPCEvent, session_id: str, payload: dict | None = None) -> "IPCMessage":
        ...
```

### 4.2 Worker 事件循环

```python
async def worker_event_loop(
    channel: IPCClientChannel,
    graph,
    session_id: str,
    history: ConversationHistory,
    ...
) -> None:
    """Worker 主事件循环。"""

    while True:
        # 回收非阻塞事件（等待指令同时周期性处理内部任务）
        cmd = await channel.receive_command(timeout=1.0)

        if cmd is None:
            # 无指令，处理心跳
            continue

        if cmd.name == IPCCommand.EXECUTE.value:
            user_input = cmd.payload.get("content", "")
            live_push_cb = _make_live_push(channel, session_id)

            try:
                # 执行编排图
                events = await execute_turn_in_worker(
                    graph=graph,
                    session_id=session_id,
                    user_input=user_input,
                    history=history,
                    live_push=live_push_cb,
                    config=...,
                )
                # 终端事件已在 execute_turn_in_worker 内通过 live_push 上报

            except InterruptException:
                # 中断 → 保存检查点 → 上报 EXECUTION_PAUSED
                ...
            except AskUserException:
                # 向用户提问 → 保存检查点 → 上报 ASK_USER
                ...
            except Exception as e:
                # 其他异常 → 上报 ERROR 事件
                await channel.send_event(IPCEvent.ERROR, {
                    "error": str(e),
                    "error_type": type(e).__name__,
                })

        elif cmd.name == IPCCommand.HEARTBEAT.value:
            await channel.send_event(IPCEvent.HEARTBEAT_ACK, {
                "timestamp": time.time(),
                "status": "alive",
            })

        elif cmd.name == IPCCommand.SHUTDOWN.value:
            # 保存会话状态
            save_session_state(session_id, history, ...)
            await channel.send_event(IPCEvent.EXECUTION_COMPLETE, {
                "reason": "worker_shutdown",
            })
            break

        elif cmd.name == IPCCommand.FORCE_KILL.value:
            # 立即退出，不保存状态
            break
```

### 4.3 Worker 内执行流程

```python
async def execute_turn_in_worker(
    graph,
    session_id: str,
    user_input: str,
    history: ConversationHistory,
    live_push: Callable,
    config: dict,
) -> None:
    """在 Worker 进程内执行一轮对话。

    此函数与当前 ACPServer.execute_turn() 逻辑基本一致，
    但事件上报方式从直接返回列表变为通过 live_push 实时推送。
    """

    # 1. 压缩检查
    comp_result = compress_if_needed(history, ...)
    if comp_result.did_compress:
        live_push(_make_acp_event(PushEvent.PHASE_CHANGED, ...))

    # 2. 准备上下文
    if checkpoint:
        initial_state = deserialize_checkpoint(checkpoint)
        live_push(_make_acp_event(PushEvent.EXECUTION_RESUMED, ...))
    else:
        history.add_user_message(user_input)
        context_msgs = history.get_context_messages()
        initial_state = {"messages": context_lc, "_session_id": session_id}

    # 3. 安装 Tracer Hook（路由到 live_push）
    hook = _WorkerTracerHook(live_push, session_id)
    hook.install()

    try:
        # 4. 执行编排图
        result = await asyncio.wait_for(
            graph.ainvoke(initial_state),
            timeout=execution_timeout,
        )

        # 5. 处理结果
        drain_events()
        history.add_agent_messages(result.get("ga_messages", []))
        final_answer = extract_final_answer(result)
        live_push(_make_acp_event(PushEvent.FINAL_ANSWER, {"content": final_answer}))
        live_push(_make_acp_event(PushEvent.EXECUTION_COMPLETE, ...))

    except InterruptException as e:
        checkpoint = serialize_checkpoint(e.state, session_id, "interrupt")
        live_push(_make_acp_event(PushEvent.EXECUTION_PAUSED, ...))

    except AskUserException as e:
        checkpoint = serialize_checkpoint(state, session_id, "ask_user")
        live_push(_make_acp_event(PushEvent.ASK_USER, ...))

    except asyncio.TimeoutError:
        live_push(_make_acp_event(PushEvent.EXECUTION_PAUSED, {"reason": "timeout"}))

    finally:
        hook.uninstall()
        # 持久化会话
        save_session_to_disk(session_id, history, checkpoint, ...)
```

## 5. SessionProcessManager 设计

### 5.1 核心职责

```python
# src/graph_agent/acp/process_manager.py

class SessionProcessManager:
    """会话进程管理器 —— Master 侧的核心组件。

    职责：
    1. Worker 进程的创建、监控、销毁
    2. IPC 通道的创建与管理
    3. 心跳检测与无响应进程回收
    4. 进程池大小控制（防止过度 fork）
    5. 优雅关闭（全部 Worker 保存状态后退出）

    生命周期：
    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │  IDLE    │───►│ SPAWNING │───►│  ACTIVE  │───►│DRAINING  │───► TERMINATED
    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                         │               │               │
                         ▼               ▼               ▼
                     创建子进程      正常运行        保存状态
                     初始化IPC      处理请求        释放资源
                     加载会话       心跳监控        进程退出
    """

    def __init__(self, config: ProcessManagerConfig):
        # 配置
        self._config = config

        # 活跃的 Worker 管理
        self._workers: dict[str, WorkerHandle] = {}
        # key: session_id, value: WorkerHandle

        # 进程池上限控制
        self._max_workers = config.max_workers  # 默认 50

        # 健康监控
        self._health_check_task: asyncio.Task | None = None
        self._health_check_interval = config.health_check_interval  # 默认 30s

        # 关闭信号
        self._shutdown_event = asyncio.Event()

    # ── 生命周期 ──────────────────────────────────────────

    async def start(self):
        """启动进程管理器，开始健康监控。"""
        ...

    async def stop(self, graceful: bool = True):
        """停止所有 Worker 进程。

        Args:
            graceful: True 时先发送 SHUTDOWN 等待保存，再终止；
                      False 时直接 FORCE_KILL。
        """
        ...

@dataclass
class WorkerHandle:
    """Worker 进程句柄 —— 封装进程引用和 IPC 通道。"""

    session_id: str
    process: mp.Process
    channel: IPCServerChannel
    created_at: float          # 创建时间戳
    last_active_at: float      # 最后活跃时间戳
    last_heartbeat: float      # 最后心跳时间戳
    status: str                # "spawning" | "active" | "idle" | "draining" | "dead"

    def is_alive(self) -> bool:
        """Worker 进程是否存活。"""
        return self.process.is_alive() if self.process else False

    def is_stale(self, timeout: float) -> bool:
        """心跳是否超时（判定为无响应）。"""
        return time.time() - self.last_heartbeat > timeout

    def touch(self):
        """标记为活跃。"""
        self.last_active_at = time.time()
```

### 6.4 回收策略

```python
class WorkerRecycleStrategy:
    """Worker 回收策略 —— 决定在达到上限时回收哪个 Worker。"""

    @staticmethod
    def select_victim(workers: dict[str, WorkerHandle],
                       max_workers: int) -> list[str]:
        """选择需要回收的 Worker。

        优先级（从高到低）：
        1. 已死亡但未清理的 Worker（CRASHED/DRAINING）
        2. 心跳超时（无响应）的 Worker
        3. 空闲时间最长的 Worker（LRU）
        """
        candidates = []

        for sid, handle in workers.items():
            # 优先级分数：越低越先被回收
            if handle.status == "dead":
                priority = 0  # 立即清理
            elif handle.is_stale(heartbeat_timeout):
                priority = 1  # 无响应
            else:
                # 按空闲时间排序
                idle_seconds = time.time() - handle.last_active_at
                priority = 2 + (1.0 / (idle_seconds + 1))  # 空闲越久分数越低

            candidates.append((sid, priority))

        # 按优先级升序排列（最该被回收的排在前面）
        candidates.sort(key=lambda x: x[1])
        to_recycle_count = len(workers) - max_workers
        return [sid for sid, _ in candidates[:max(0, to_recycle_count)]]
```

## 10. 异常处理

### 10.1 异常场景矩阵

| 场景 | Master 行为 | Worker 行为 | 客户端感知 |
|------|-------------|-------------|------------|
| **Worker 进程崩溃** | 检测 `is_alive() == False`，从磁盘恢复会话，必要时创建新 Worker | N/A（已崩溃） | SSE 收到 `error` 事件（可恢复），会话可重新加载 |
| **Worker 心跳超时** | 重试 N 次后标记为 STALE，强制终止并回收 | 被 `terminate()` 强制结束 | SSE 收到 `error` 事件，会话可恢复 |
| **Worker 执行超时** | 等待 `execution_timeout` 后发送 SHUTDOWN | 捕获 `asyncio.TimeoutError`，保存检查点后退出 | SSE 收到 `execution_paused` 事件，可恢复 |
| **IPC 队列满** | 日志警告，等待消费 | 日志警告，等待消费 | 事件延迟到达（可接受） |
| **Worker 启动失败** | 重试 3 次，超过后返回错误 | N/A | 收到 `SESSION_NOT_FOUND` 或 `INTERNAL_ERROR` |
| **优雅关闭时 Worker 不响应** | 等待 `graceful_timeout` 后降级为 `terminate()` | 被强制终止 | 下次加载时从检查点恢复 |
| **内存超限** | 标记 Worker 需要回收，选择新 Worker 处理请求 | 收到 SHUTDOWN 后保存退出 | SSE 事件流中断后重连 |

### 10.2 崩溃恢复流程

```
Worker 崩溃
    │
    ▼
Master 检测到 is_alive() == False
    │
    ▼
检查磁盘上是否有保存的会话文件
    │
    ├── 有会话文件 ──► 标记为可恢复
    │     │
    │     ▼
    │   通知 SSE 客户端: error (recoverable=true)
    │     │
    │     ▼
    │   客户端发起 LOAD_SESSION
    │     │
    │     ▼
    │   创建新 Worker + 从磁盘恢复状态
    │     │
    │     ▼
    │   正常执行
    │
    └── 无会话文件 ──► 标记为不可恢复
          │
          ▼
        通知 SSE 客户端: error (recoverable=false)
          │
          ▼
        会话永久丢失，需 CREATE_SESSION 重新开始
```

## 11. 数据持久化

### 11.1 持久化策略

```
持久化时机：
  1. 每轮对话完成后：Worker 自动保存会话历史
  2. 检查点创建时：Interrupt / AskUser 触发时保存
  3. Worker 关闭时：SHUTDOWN 指令触发最终保存
  4. 定期自动保存：每 N 轮对话或有检查点变更时

文件格式（继承现有设计）：
  data/conversations/{session_id}.json

文件内容结构：
  {
    "session_id": "sess_abc123",
    "messages": [...],
    "turn_count": 5,
    "status": "idle",
    "checkpoint": { ... } | null,
    "created_at": "2026-06-02T10:00:00Z",
    "updated_at": "2026-06-02T11:30:00Z",
    "worker_meta": {                         // [新增]
      "last_worker_pid": 12345,
      "last_worker_started_at": "...",
      "worker_shutdown_reason": "idle_timeout"
    }
  }
```

### 11.2 并发写入保护

由于 Master 不持有会话数据，Worker 是唯一写入者，无需文件锁。但需要考虑以下边缘情况：

- **Worker 在写入过程中崩溃**：采用"写临时文件 + 原子重命名"策略
- **旧 Worker 残留与新 Worker 启动**：新 Worker 启动时清理旧的 `.tmp` 文件

```python
def safe_save(session_id: str, data: dict, storage_dir: str) -> None:
    """原子保存：写入临时文件后重命名，防止写入中途崩溃损坏数据。"""
    import tempfile, os

    file_path = Path(storage_dir) / f"{session_id}.json"
    tmp_path = Path(storage_dir) / f"{session_id}.tmp"

    with open(tmp_path, "w", encoding="utf-8") as f:
        json.dump(data, f, ensure_ascii=False, indent=2)

    # 原子替换（同一文件系统上 rename 是原子操作）
    os.replace(tmp_path, file_path)
```

## 13. 风险与限制

### 13.1 已知限制

| 限制 | 说明 | 缓解措施 |
|------|------|----------|
| **Python GIL** | 多进程而非多线程，每个 Worker 有自己的 GIL | 设计上使用 `multiprocessing.Process`，已规避 |
| **进程创建开销** | Windows 上 `spawn` 模式较慢（~1-2s） | Worker 复用策略，尽量不频繁创建/销毁 |
| **内存开销** | 每个 Worker 独立加载 LLM SDK、Hook 等 | 预计每 Worker 基础内存 ~200-500MB，通过 max_workers 控制 |
| **跨平台差异** | `fork`（Linux）vs `spawn`（Windows/macOS） | 统一使用 `spawn` 模式，确保跨平台一致性 |
| **序列化限制** | `pickle` 无法序列化某些对象（如 lambda、线程锁） | IPC 消息仅包含可序列化的基础数据，避免传递复杂对象 |
| **文件系统竞争** | 多 Worker 同时写入不同会话文件不会冲突，但同一会话不会并发写入 | 设计上保证同一会话只有一个 Worker |

### 13.2 降级方案

如果多进程模式出现问题：

1. **运行时降级**：将 `process.mode` 改为 `single`，重启服务即可回到单进程模式
2. **自动降级**：Worker 创建连续失败 N 次后，自动回退到单进程模式（配置项 `process.auto_fallback: true`）
3. **数据不丢失**：无论是单进程还是多进程模式，会话数据都持久化到磁盘，模式切换不影响已有会话
