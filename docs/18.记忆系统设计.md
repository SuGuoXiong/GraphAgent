# GraphAgent 记忆系统设计

## 1. 概述

### 1.1 设计动机

当前 GraphAgent 在每次会话中从零开始理解用户意图和任务上下文，无法利用历史交互中积累的经验。这导致：

1. **重复试错**：用户在上一次会话中纠正过的方案偏好（如"不要并行执行，改为串行"），在新会话中需要再次强调
2. **偏好丢失**：用户长期使用的习惯（如"始终返回中文"、"代码注释用英文"）在每次会话中需要重新声明
3. **经验浪费**：类似任务的成功执行方案无法为后续任务提供参考

记忆系统旨在让 GraphAgent 具备"记住用户"和"记住经验"的能力，在每次会话开始前加载用户画像和与当前任务相关的历史经验。

### 1.2 双层记忆模型

| 记忆类型 | 存储层 | 生命周期 | 相关性范围 | 注入目标 |
|---------|--------|---------|-----------|---------|
| **用户画像** (User Profile) | `user_profile.md` | 跨会话、跨主题，长期有效 | 始终相关 | Layer 2 (GuardAgent) |
| **主题记忆** (Topic Memory) | `topic/*.md` | 跨会话，按主题关联，半衰期衰减 | 按任务意图检索 | Layer 3 (PlanAgent) |

### 1.3 与现有架构的关系

```
┌──────────────────────────────────────────────────────────────────┐
│                    OrchestrationState (已有)                       │
│  user_preferences: dict | None   ← Layer 2 扩展点 (已有字段)       │
│  memory_refs: list | None        ← Layer 3 扩展点 (已有字段)       │
├──────────────────────────────────────────────────────────────────┤
│                    Context Builder (已有)                          │
│  guard_context_builder.py   ← 注入 user_preferences               │
│  plan_context_builder.py    ← 注入 memory_refs                     │
├──────────────────────────────────────────────────────────────────┤
│                    Memory System (新增)                            │
│  memory_extractor.py  — 记忆提取 Skill                            │
│  memory_retriever.py  — 主题记忆检索                               │
│  memory_manager.py    — 记忆生命周期管理                           │
│  memory_injector.py   — 记忆格式化注入                              │
└──────────────────────────────────────────────────────────────────┘
```

## 2. 记忆存储设计

### 2.1 目录结构

```
data/memory/
├── user_profile.md              # 用户画像（单文件，始终加载）
├── topic/                       # 主题记忆目录
│   ├── code_analysis.md          # 代码分析相关经验
│   ├── refactoring.md            # 重构相关经验
│   ├── project_setup.md          # 项目搭建相关经验
│   ├── data_processing.md        # 数据处理相关经验
│   └── ...
├── index.json                   # 主题记忆索引（快速检索）
└── archive/                     # 过期归档
    └── ...
```

### 2.2 用户画像文件格式

所有记忆文件使用 Markdown + YAML frontmatter 格式：

```markdown
---
updated_at: "2026-05-26T14:30:00Z"
total_extractions: 12
---

# 用户画像

## 语言与格式偏好
- 用户希望所有回复使用简体中文
- 代码注释使用英文
- 输出格式偏好 Markdown，含代码块时标注语言

## 交互风格偏好
- 用户喜欢详细的方案分析，不喜欢过于简略的回答
- 在执行危险操作前需要明确提示确认

## 工具与技能偏好
- 优先使用本地工具而非远程 MCP 工具（更快）
- 代码分析时偏好自底向上（先模块细节，后整体架构）

## 质量要求
- 代码修改后必须提供验证方案
- 涉及多文件修改时，提供修改清单
```

### 2.3 主题记忆文件格式

```markdown
---
topic: code_analysis
created_at: "2026-05-20T10:00:00Z"
updated_at: "2026-05-26T15:00:00Z"
last_accessed_at: "2026-05-26T15:00:00Z"
total_extractions: 5
keywords:
  - 代码结构分析
  - 模块依赖
  - 项目重构
---

# 代码分析

## 记忆条目

### 2026-05-26 — 大型项目代码结构分析
- **任务**: 分析 GraphAgent 项目代码结构，梳理模块职责和依赖关系
- **方案**: 采用三层分析策略——先扫描目录结构，再逐模块分析职责，最后用 grep 梳理依赖关系
- **经验**: 
  - 3 个子任务并行执行效率高于串行
  - 使用 glob + grep 组合比逐文件读取更高效
  - 最终汇总时按"模块总览 → 依赖关系图 → 关键接口"的结构组织输出
- **用户反馈**: 满意，未要求修改

### 2026-05-22 — 小型项目快速代码审查
- **任务**: 审查单个 Python 脚本的代码质量
- **方案**: 直接在 PlanAgent 中以内置知识分析，无需派发 SubAgent
- **经验**:
  - 单文件项目无需多 Agent 并行，PlanAgent 直出结果更快
  - GuardAgent 审核时重点关注安全漏洞和性能问题
- **用户反馈**: 要求增加 PEP 8 规范检查项
- **用户调整**: 后续所有代码审查任务均加入 PEP 8 检查
```

## 3. 记忆生命周期

### 3.1 三层触发机制

```
┌─────────────────────────────────────────────────────────────────┐
│                     记忆提取 = 非阻塞后台任务                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────┐    ┌──────────────┐    ┌──────────────┐           │
│  │ 实时提取  │ → │ 会话结束合并  │ → │ 新会话加载    │           │
│  │ (增量)   │    │ (去重+落盘)  │    │ (注入 State) │           │
│  └──────────┘    └──────────────┘    └──────────────┘           │
│  每条消息完成      final_answer 后     创建会话时                  │
│  后异步执行        同步执行            同步执行                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**触发层 1 — 实时提取（增量）：**

- 时机：每条用户消息处理完成后（即 `execute_turn` 返回后）
- 方式：`asyncio.create_task(_extract_memory_incremental(session_id, turn_messages))`，不阻塞后续消息
- 内容：仅提取本轮消息中可识别的偏好线索和主题关联
- 产出：写入内存暂存区（dict），标记为"待合并"

**触发层 2 — 会话结束合并（去重 + 落盘）：**

- 时机：`final_answer` 发送后，下一个 `send_message` 请求到达时（利用用户的思考间隙）
- 方式：同步等待合并完成（通常 < 2s），若超时则延迟到下次
- 内容：
  1. 从暂存区取出本轮所有增量提取结果
  2. 读取已有记忆文件
  3. 执行去重/合并/冲突解决（见 3.2）
  4. 写入正式记忆文件
  5. 更新 `index.json`

**触发层 3 — 新会话加载（注入 State）：**

- 时机：`create_session` 或 `load_session` 时
- 方式：同步读取文件并注入
- 内容：
  1. 读取 `user_profile.md` → 反序列化 → 注入 `state.user_preferences`
  2. 读取 `index.json` → 注入 `state.memory_refs` 作为可检索索引（实际检索延迟到 GuardAgent 意图分析后）

### 3.2 冲突解决与合并策略

**用户画像合并：**

```
新提取的偏好 + 已有偏好 → 逐维度对比

同一维度上的冲突：
  - 以最新表述为准（"最新优先"原则）
  - 不保留历史版本

不同维度上的追加：
  - 直接添加到对应维度下
  - 按时间倒序排列

超过 90 天未更新的维度：
  - 在条目前添加 ⚠️ 标记
  - 在注入上下文时附带"此偏好可能已过时"提示
```

**主题记忆合并：**

```
新提取的记忆 + 已有主题记忆 → 按 topic slug 分组

同一 topic 下的合并：
  1. 检查语义重复（标题和摘要相似度 > 80% 视为重复）
     - 重复时：保留信息量更大的版本
  2. 非重复时：追加为新条目，按时间倒序排列
  3. 单 topic 条目数超过 10 条时：
     - 触发 LLM 合并：将多条浓缩为综合条目
     - 合并后保留 3-5 条，保留核心经验，舍弃冗余细节
```

### 3.3 记忆衰减与过期

- **时间衰减权重**：检索匹配分 × `exp(-λt)`，t 为距今天数，λ 使得半衰期为 30 天
- **过期归档**：`last_accessed_at` 超过 90 天的主题记忆条目迁移至 `archive/` 目录
- **用户画像不过期**：用户画像是长期偏好，不自动过期，由用户显式管理（未来可增加 `/forget_preference` 命令）

### 3.4 记忆操作时序图

```
用户发送消息
  │
  ├─ [同步] state.user_preferences ← 加载 user_profile.md (Layer 2)
  ├─ [同步] state.memory_refs ← 加载 index.json 中的 topic 摘要列表 (Layer 3)
  │
  ├─ GuardAgent 意图分析 → 提取 intent 描述
  │     │
  │     └─ [同步] MemoryRetriever.search(intent, top_k=5)
  │           │
  │           ├─ 遍历 index.json 中的 topic 摘要
  │           ├─ 关键词匹配 + 时间衰减权重 → 排序
  │           ├─ 读取匹配的 topic/*.md 完整内容
  │           └─ 更新匹配条目的 last_accessed_at
  │
  ├─ state.memory_refs ← 检索到的完整主题记忆内容
  │
  ├─ PlanAgent 方案制定 → 将 memory_refs 纳入上下文 (Layer 3)
  │
  ├─ ... 编排执行 ...
  │
  ├─ final_answer 发送
  │
  ├─ [异步] _extract_memory_incremental(session_id, messages)
  │     │
  │     └─ MemoryExtractor Skill:
  │           ├─ 分析本轮用户消息 → 提取偏好线索
  │           │    - 语言/格式/风格/工具偏好
  │           │    - 与已有画像对比
  │           ├─ 分析本轮任务执行 → 提取主题经验
  │           │    - 任务描述 + 方案 + 执行过程 + 用户反馈
  │           │    - 分配 topic slug（关键词匹配已有 topic，或创建新 topic）
  │           ├─ 写入暂存区
  │           └─ 返回提取摘要
  │
  └─ [下一次 send_message 到达时]
        │
        └─ [同步] MemoryManager.consolidate(session_id)
              ├─ 从暂存区读取增量提取结果
              ├─ 读取已有 memory 文件
              ├─ 执行去重/冲突解决/合并
              ├─ 写入 user_profile.md / topic/*.md / index.json
              └─ 清空暂存区
```

## 4. 检索机制设计

### 4.1 检索策略

用户画像和主题记忆的消费模式不同，采用差异化策略：

| | 用户画像 | 主题记忆 |
|------|---------|---------|
| 数量 | 少（~10 条偏好描述） | 随使用增长（数十条记忆条目） |
| 相关性 | 始终相关 | 仅部分与当前任务相关 |
| 注入方式 | 全量注入 Layer 2 | 按需检索，注入 Layer 3 |
| 检索时机 | 会话创建时（无检索） | GuardAgent 意图分析后 |
| 检索方法 | 全量读取 | 关键词匹配 + 时间衰减排序 |

### 4.2 检索流程

```python
class MemoryRetriever:
    """主题记忆检索器。初期使用关键词匹配，后续引入向量检索。"""

    def search(self, intent: str, top_k: int = 5) -> list[dict]:
        """
        1. 从 intent 中提取关键词（中文分词 + 英文 tokenize）
        2. 遍历 index.json 中的 topic 摘要
        3. 计算匹配分：
           - 关键词命中数 / 总关键词数
           - 乘以时间衰减权重: exp(-λ × days_since_last_access)
        4. 按分数降序排序，取 top_k
        5. 读取匹配的 topic/*.md 文件
        6. 更新 last_accessed_at
        7. 返回记忆条目列表
        """
```

### 4.3 向量数据库扩展路径

当前基于 `index.json` 的关键词检索在以下场景足够：
- 主题记忆数量 < 50 条
- 主题分类明确（代码分析、重构、数据处理等）

当记忆规模增长时，按以下路径升级：

```
阶段 1 (当前): index.json + 关键词匹配 + 时间衰减
    │  存储: markdown 文件
    │  检索: 内存索引遍历（< 5ms，50 条场景）
    │
阶段 2 (未来): embedding + 余弦相似度
    │  存储: markdown 文件 + index.json 中填充 embedding 字段
    │  检索: numpy 计算余弦相似度（< 10ms，500 条场景）
    │  依赖: sentence-transformers 或 OpenAI embeddings API
    │
阶段 3 (大规模): 向量数据库（ChromaDB / Milvus Lite）
    │  存储: 向量数据库 + markdown 文件（原始内容）
    │  检索: ANN 近似最近邻（< 100ms，5000+ 条场景）
    │  依赖: chromadb 或 pymilvus
```

**关键设计原则**：`MemoryRetriever.search()` 接口保持不变。各阶段仅替换内部实现。

## 5. 记忆注入设计

### 5.1 注入层级映射

```
Layer 1: 持久化存储
    ↓ (不注入记忆，仅存储原始对话)

Layer 2: GuardAgent 战略视图  ← user_preferences
    │  context.insert(0, formatted_preferences)
    │  位置: 对话历史之前
    │  格式: "你应该知道的用户偏好：\n- ..."

Layer 3: PlanAgent 战术视图     ← memory_refs
    │  context.append(formatted_memories)
    │  位置: GuardAgent 意图分析结果之后
    │  格式: "以下是你过去处理类似任务时的经验：\n### ..."

Layer 4: SubAgent 执行视图
    (不注入记忆，SubAgent 仅关注当前子任务)
```

### 5.2 注入代码结构

```python
# guard_context_builder.py
class GuardContextBuilder:
    def build(self, history, ..., user_preferences=None):
        context = [...]  # 现有构建逻辑

        if user_preferences:
            prefs_block = self._format_user_preferences(user_preferences)
            # 插入到对话历史之前
            context.insert(0, MessageBlock(
                role="system",
                content=prefs_block,
                message_type=MessageType.SYSTEM_MEMORY_PROFILE,
            ))

        return context

    def _format_user_preferences(self, prefs: dict) -> str:
        """将用户画像 dict 格式化为 LLM 可读的提示段落。"""
        lines = ["[系统记忆] 以下是该用户的长期偏好，请在规划和回复时遵守："]
        for category, items in prefs.items():
            lines.append(f"\n## {category}")
            for item in items:
                stale = "⚠️ " if item.get("stale") else ""
                lines.append(f"- {stale}{item['content']}")
        return "\n".join(lines)


# plan_context_builder.py
class PlanContextBuilder:
    def build(self, history, ..., memory_refs=None):
        context = [...]  # 现有构建逻辑

        if memory_refs:
            memory_block = self._format_memory_refs(memory_refs)
            # 插入到 GuardAgent 意图分析之后
            context.append(MessageBlock(
                role="system",
                content=memory_block,
                message_type=MessageType.SYSTEM_MEMORY_TOPIC,
            ))

        return context

    def _format_memory_refs(self, refs: list[dict]) -> str:
        """将主题记忆列表格式化为 LLM 可读的经验提示。"""
        lines = ["[系统记忆] 以下是你过去处理类似任务的经验记录，请参考其中的成功方案和用户反馈："]
        for i, ref in enumerate(refs, 1):
            lines.append(f"\n### 经验 {i}: {ref['title']}")
            lines.append(f"- 时间: {ref['date']}")
            lines.append(f"- 任务: {ref['task']}")
            lines.append(f"- 方案: {ref['approach']}")
            lines.append(f"- 经验: {ref['lessons']}")
            if ref.get('user_feedback'):
                lines.append(f"- 用户反馈: {ref['user_feedback']}")
        return "\n".join(lines)
```

### 5.3 上下文预算控制

记忆注入占用上下文窗口，需要预算控制：

| 注入项 | 最大长度 | 说明 |
|--------|---------|------|
| user_preferences | 500 tokens | 用户画像通常很短，500 tokens 足够覆盖所有偏好维度 |
| memory_refs | 1500 tokens | 5 条主题记忆，每条约 300 tokens（标题 + 方案 + 经验） |
| **记忆注入合计** | **~2000 tokens** | 约占总上下文预算（128K）的 1.5%，可忽略 |

多条主题记忆时，按检索分排序，超过 1500 tokens 截断。

## 6. 记忆提取 Skill 设计

### 6.1 Skill 定义

```markdown
---
name: memory_extractor
description: 从对话历史中提取用户画像线索和主题经验，更新记忆存储
version: 1.0
---

# 记忆提取器

## 职责
分析一轮完整的用户对话，从中提取两类信息并更新记忆存储。

## 输入
- 本轮对话的全部消息（用户消息 + Agent 回复 + 工具调用记录）
- 已有用户画像（如存在）
- 已有主题记忆索引（如存在）

## 输出
1. 用户画像增量：新发现的偏好线索，按维度分类
2. 主题记忆增量：本轮任务的经验总结，标记 topic slug

## 提取指南

### 用户画像提取
从用户消息中识别以下维度的线索：
- 语言与格式偏好（目标语言、输出格式、代码注释语言等）
- 交互风格偏好（详细程度、确认频率、语气风格等）
- 工具与技能偏好（工具优先级、执行策略等）
- 质量要求（验证要求、测试要求、文档要求等）

### 主题记忆提取
从本轮任务执行中提取：
- 任务摘要：用户在做什么
- 方案描述：GraphAgent 制定的方案
- 执行过程：关键决策和调整点
- 经验教训：什么做得好，什么可以改进
- 用户反馈：用户满意/不满意/提出的修改要求

### 注意事项
- 只提取明确表达的信息，不要过度推断
- 与已有记忆对比，避免重复
- 如果本轮没有新的信息，输出空结果
```

### 6.2 Skill 调用方式

```python
class MemoryExtractor:
    """记忆提取器 —— 封装记忆提取 Skill 的调用逻辑。"""

    def __init__(self):
        self._skill = SkillLoader().load("memory_extractor")

    async def extract_profile(self, messages: list, existing_profile: dict | None) -> dict:
        """从本轮对话中提取用户画像增量。"""
        prompt = self._build_profile_extraction_prompt(messages, existing_profile)
        result = await self._call_llm(prompt)
        return self._parse_profile_result(result)

    async def extract_topic(self, messages: list, task_summary: str,
                            existing_topics: list) -> dict | None:
        """从本轮任务中提取主题记忆。"""
        prompt = self._build_topic_extraction_prompt(messages, task_summary, existing_topics)
        result = await self._call_llm(prompt)
        return self._parse_topic_result(result)
```

> 提取调用在后台 worker 中执行，不阻塞用户交互。

### 7.2 统一接口

```python
class MemoryManager:
    """记忆系统统一入口，对外暴露三个核心操作。"""

    def load(self, session_id: str) -> tuple[dict | None, list[dict]]:
        """会话创建时调用：加载用户画像和主题记忆索引。
        返回 (user_preferences, memory_index_list)
        """

    def search(self, intent: str, top_k: int = 5) -> list[dict]:
        """GuardAgent 意图分析后调用：检索相关主题记忆。
        返回匹配的完整主题记忆条目列表
        """

    def extract_async(self, session_id: str, messages: list, task_summary: str) -> str:
        """final_answer 后调用：提交异步提取任务。
        返回 job_id，可用于查询状态（通常不需要）
        """

    def consolidate(self, session_id: str) -> None:
        """下一次 send_message 到达时调用：合并暂存区增量到正式存储。"""
```

## 8. 数据模型

```python
# memory_types.py

from dataclasses import dataclass, field
from datetime import datetime, timezone
from typing import Optional


def _now() -> str:
    return datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ")


@dataclass
class UserPreference:
    """单条用户偏好。"""
    category: str           # 偏好维度（语言偏好、交互风格、工具偏好等）
    content: str            # 偏好内容描述
    confidence: float = 1.0 # 置信度（0-1，多次确认后提升）
    created_at: str = field(default_factory=_now)
    updated_at: str = field(default_factory=_now)
    stale: bool = False     # 超过 90 天未更新则标记为可能过时


@dataclass
class TopicMemoryEntry:
    """单条主题记忆条目。"""
    title: str              # 简短标题（如"大型项目代码结构分析"）
    task: str               # 任务描述
    approach: str           # 采用的方案
    lessons: str            # 经验教训
    user_feedback: str = "" # 用户反馈
    date: str = field(default_factory=_now)


@dataclass
class TopicMemory:
    """一个主题下的完整记忆文件。"""
    slug: str               # 文件标识（如 "code_analysis"）
    title: str              # 主题名称（如 "代码分析"）
    keywords: list[str]     # 检索关键词
    summary: str            # 一段话概述该主题覆盖的内容范围
    entries: list[TopicMemoryEntry] = field(default_factory=list)
    created_at: str = field(default_factory=_now)
    updated_at: str = field(default_factory=_now)
    last_accessed_at: str = field(default_factory=_now)


@dataclass
class MemoryIndex:
    """index.json 的数据模型。"""
    user_profile_updated_at: str = ""
    topics: list[dict] = field(default_factory=list)
    # 每个 topic dict: {slug, title, keywords, summary, ...updated_at, last_accessed_at, entry_count, embedding}
```

### 9.2 依赖与风险

| 依赖/风险 | 应对 |
|----------|------|
| 记忆提取依赖 LLM 调用（消耗 token） | 使用低成本模型（如 DeepSeek-Chat）执行提取，每次约 500 input + 200 output tokens |
| 异步提取失败（网络波动、LLM 限流） | 失败时重试 3 次，仍失败则跳过本轮提取，下次对话时自动补偿 |
| 记忆文件并发写入冲突 | 使用文件锁（`fcntl` / `msvcrt`），暂存区以 session_id 隔离 |
| 主题记忆冷启动（无历史数据） | 新用户的 `user_profile.md` 和 `index.json` 为空，记忆系统静默跳过，不影响正常功能 |
