# SubAgent DAG 并行执行设计文档

## 1. 概述

### 1.1 背景与问题

当前 GraphAgent 的 `subagent_exec_node` 采用**严格串行**的方式执行所有子任务：

```python
# node/subagent.py:235 — 当前串行执行逻辑
for task in task_plan.sub_tasks:
    if task.status != "running":
        continue
    result = _run_subagent(config, task.description, task.task_id, state)
    sub_results[task.task_id] = result
    task.status = "completed"
```

**存在的问题：**

1. **资源浪费**：多个无依赖关系的子任务被迫串行等待，即使它们可以同时执行（例如：任务A抓取网页、任务B读取本地文件——两者互不依赖却要排队）
2. **`execution_strategy` 字段形同虚设**：`TaskPlan.execution_strategy` 在 PlanAgent 的 prompt 中被要求生成（值为 `"parallel"` 或 `"sequential"`），但 `subagent_exec_node` 完全不读取该字段，所有任务一律串行处理
3. **与设计原则不符**：文档 `4.Agent编排模块设计.md` 第 1.3 节明确写了设计原则之一是"并行执行：无依赖的子任务由各自 SubAgent 并行处理"，但当前实现并未落地

### 1.2 设计目标

1. **DAG 任务编排**：PlanAgent 将任务拆解为有向无环图（DAG），通过 `dependencies` 字段表达子任务间的偏序关系
2. **一次派发、分层执行**：`_dispatch_tasks` 一次性将所有 `pending` 任务派发为 `running`；`subagent_exec_node` 按拓扑层级分组——同一层内的任务无相互依赖，并行执行；跨层任务按依赖顺序串行执行
3. **占位符延迟解析**：`{{task_id}}` 占位符从派发时解析改为执行前 JIT 解析，确保跨层数据引用始终获取最新结果
4. **向后兼容**：保持现有 `SubTask` / `TaskPlan` 数据模型不变，保持 `AskUserException` 中断恢复机制不变，保持 GuardAgent → PlanAgent → SubAgent 三层编排流程不变

### 1.3 核心思路

```
PlanAgent 生成 DAG:
  task_1 ──→ task_3 ──→ task_5
  task_2 ──→ task_4 ──→ task_5

拓扑分层：
  第0层 (并行): [task_1, task_2]
  第1层 (并行): [task_3, task_4]
  第2层 (并行): [task_5]

执行时序：
  Layer 0: task_1 ∥ task_2  ──等待完成──→
  Layer 1: task_3 ∥ task_4  ──等待完成──→
  Layer 2: task_5
```

### 2.2 新增模块：`orchestration/dag.py`

提供 DAG 编排所需的纯函数工具：

```python
"""DAG 工具模块——拓扑排序、分层分组、依赖校验。"""

from graph_agent.orchestration.state import SubTask


def build_dag(tasks: list[SubTask]) -> dict[str, set[str]]:
    """从任务列表构建邻接表（task_id -> 直接后继集合）。

    注：此函数用于可视化和调试，不参与运行时执行路径。
    """
    dag: dict[str, set[str]] = {t.task_id: set() for t in tasks}
    for t in tasks:
        for dep in t.dependencies:
            dag.setdefault(dep, set()).add(t.task_id)
    return dag


def topological_layers(tasks: list[SubTask]) -> list[list[SubTask]]:
    """将任务列表按拓扑层级分组，每层内的任务无相互依赖。

    算法（Kahn 变体）：
    1. 计算每个任务的入度——仅统计 dependencies 中**也在本批任务集内**的依赖
       （指向已完成任务或不存在任务的依赖不参与拓扑排序，避免入度虚高）
    2. 入度为 0 的任务进入当前层
    3. 执行当前层后，将其后继任务的入度减 1
    4. 入度变为 0 的后继进入下一层
    5. 重复直到所有任务分层完成

    Returns:
        [[layer_0_tasks], [layer_1_tasks], ...]
        同一层内的任务可以并行执行

    Example:
        tasks = [task_1(deps=[]), task_2(deps=[]), task_3(deps=[task_1, task_2])]
        → [[task_1, task_2], [task_3]]

        tasks = [task_3(deps=[task_1]), task_4(deps=[task_2])]
        （其中 task_1, task_2 已完成，不在 tasks 中）
        → [[task_3, task_4]]  ← 入度均为 0，归入同一层并行执行
    """
    task_ids = {t.task_id for t in tasks}
    task_map = {t.task_id: t for t in tasks}

    # 入度：仅统计 dependencies 中属于本批任务集的依赖
    in_degree: dict[str, int] = {}
    for t in tasks:
        internal_deps = [d for d in t.dependencies if d in task_ids]
        in_degree[t.task_id] = len(internal_deps)

    # 后继关系：仅记录本批任务集内部的后继
    successors: dict[str, list[str]] = {t.task_id: [] for t in tasks}
    for t in tasks:
        for dep in t.dependencies:
            if dep in task_ids:
                successors.setdefault(dep, []).append(t.task_id)

    layers = []
    current = [t for t in tasks if in_degree[t.task_id] == 0]

    while current:
        layers.append(current)
        next_layer = []
        for t in current:
            for succ_id in successors.get(t.task_id, []):
                in_degree[succ_id] -= 1
                if in_degree[succ_id] == 0:
                    next_layer.append(task_map[succ_id])
        current = next_layer

    return layers


def validate_dag(tasks: list[SubTask]) -> list[str]:
    """校验任务依赖图是否为合法的 DAG（无环、无缺失依赖）。

    Returns:
        错误信息列表，空列表表示校验通过
    """
    errors = []
    task_ids = {t.task_id for t in tasks}

    # 检查所有依赖是否指向存在的任务
    for t in tasks:
        for dep in t.dependencies:
            if dep not in task_ids:
                errors.append(f"任务 '{t.task_id}' 依赖不存在的任务 '{dep}'")

    # 检查是否存在循环依赖（DFS 三色标记法）
    WHITE, GRAY, BLACK = 0, 1, 2
    color = {t.task_id: WHITE for t in tasks}

    def dfs(task_id: str) -> bool:
        """返回 True 表示发现环。"""
        color[task_id] = GRAY
        task = next((t for t in tasks if t.task_id == task_id), None)
        if task:
            for dep in task.dependencies:
                c = color.get(dep, BLACK)
                if c == GRAY:
                    errors.append(f"检测到循环依赖: {task_id} -> {dep}")
                    return True
                if c == WHITE:
                    if dfs(dep):
                        return True
        color[task_id] = BLACK
        return False

    for t in tasks:
        if color[t.task_id] == WHITE:
            dfs(t.task_id)

    return errors


def validate_and_log(tasks: list[SubTask], logger=None) -> bool:
    """校验 DAG 合法性并在失败时记录日志。在 _generate_plan 和 _dispatch_tasks 中调用。

    Returns:
        True 表示通过校验
    """
    errors = validate_dag(tasks)
    if errors:
        for err in errors:
            if logger:
                logger.warning(f"[DAG Validation] {err}")
        return False
    return True
```

**`validate_dag` 调用时机**：

1. **PlanAgent 生成计划后**（`_generate_plan`）：校验 LLM 生成的 DAG 是否合法（无环、无缺失依赖）。若校验失败，记录告警并追加一条系统消息让 PlanAgent 重新生成
2. **任务派发前**（`_dispatch_tasks`）：兜底校验，防止非法 DAG 进入执行层。若校验失败，将所有涉及环的任务标记为 `failed`

### 2.3 PlanAgent 增强：`node/plan.py`

改动集中在 `_dispatch_tasks()` 方法。当前逻辑是逐个检查 `pending` 任务、依赖满足即派发；重构后改为**一次性批量派发**。

**核心变化：**

1. **一次性派发所有 pending 任务**：不再检查依赖是否满足，将所有 `pending` 任务全部标记为 `running`。依赖关系交由 `subagent_exec_node` 的拓扑分层机制处理
2. **占位符解析延后**：`{{task_id}}` 占位符不再在派发时解析（因为依赖任务可能尚未执行），改为在 `subagent_exec_node` 执行前 JIT 解析
3. **增加失败依赖检测**：如果前置任务已 `failed`，将当前任务也标记为 `failed` 并跳过

**为什么这样改？**

旧逻辑只派发依赖满足的任务，导致 `subagent_exec_node` 每次只收到一层任务，无法发挥拓扑分层并行的优势。改为一次派发后，`subagent_exec_node` 自行按 DAG 分层执行，减少了 PlanAgent 的调度往返次数。

```python
# node/plan.py:_dispatch_tasks() —— 改动后
def _dispatch_tasks(state: OrchestrationState) -> dict:
    task_plan = state.get("task_plan")
    if not task_plan:
        return {"phase": OrchestrationPhase.COMPLETED}

    sub_tasks = task_plan.sub_tasks
    dispatch_messages = []

    for task in sub_tasks:
        if task.status != "pending":
            continue

        # 检查是否有依赖任务失败 → 级联标记失败
        failed_deps = [
            dep_id for dep_id in task.dependencies
            if any(
                t.task_id == dep_id and t.status == "failed"
                for t in sub_tasks
            )
        ]
        if failed_deps:
            task.status = "failed"
            task.error = f"前置任务失败: {failed_deps}"
            continue

        # 匹配 SubAgent（同现有逻辑）
        candidates = _registry.find_by_skill(task.required_skill)
        if not candidates and task.required_skill:
            candidates = _registry.find_by_skill("general-purpose")
            if not candidates:
                candidates = _registry.find_by_skill("general")
            if not candidates:
                all_agents = _registry.list_all()
                if all_agents:
                    candidates = [all_agents[0]]

        if candidates:
            task.status = "running"
            task.assigned_agent = candidates[0].name
            msg = create_assistant_message(
                content=task.description,
                name="PlanAgent",
                message_type=MessageType.PLAN_TASK_DISPATCH,
                metadata={
                    "task_id": task.task_id,
                    "required_skill": task.required_skill,
                    "assigned_agent": task.assigned_agent,
                },
            )
            dispatch_messages.append(msg)
        else:
            task.status = "failed"
            task.error = f"未找到匹配技能 '{task.required_skill}' 的 SubAgent"

    # 注：不再在此处解析占位符，改为 subagent_exec_node 执行前 JIT 解析

    # 判断下一阶段
    all_terminal = all(
        t.status in ("completed", "failed") for t in sub_tasks
    )

    return {
        "phase": (
            OrchestrationPhase.RESULT_SYNTHESIS if all_terminal
            else OrchestrationPhase.TASK_EXECUTION
        ),
        "task_plan": task_plan,
        "ga_messages": dispatch_messages if dispatch_messages else [],
        "messages": (
            [agent_message_to_langchain(m) for m in dispatch_messages]
            if dispatch_messages else []
        ),
    }
```

**关键变化总结**：
- 移除 `deps_met` 检查 —— 所有 pending 任务一次性派发
- 增加 `failed_deps` 检测 —— 前置任务失败时级联标记
- 移除占位符解析调用 —— `_resolve_placeholders` 移至 `subagent_exec_node`
- dispatch 消息仍保留，用于 LangGraph 消息流追溯

### 2.4 SubAgent 执行节点重构：`node/subagent.py`

这是本次重构的核心。需要将串行 for 循环替换为拓扑分层并行执行。

#### 2.4.1 执行流程

```
subagent_exec_node(state)
  │
  ├─ 1. 从 task_plan 收集所有 running 状态的任务
  │     (首次调用时 = 全部任务，后续调用时 = 未完成的任务)
  │
  ├─ 2. 调用 topological_layers() 按依赖关系分层
  │     已完成任务的依赖被过滤，不影响入度计算
  │     同一层内的任务无相互依赖，可并行执行
  │
  ├─ 3. 逐层执行：
  │     for layer in layers:
  │        对每个任务 JIT 解析 {{task_id}} 占位符
  │        并行执行 layer 中的所有任务（ThreadPoolExecutor）
  │        收集结果，更新 task.status / sub_results
  │        检查中断 (AskUserException / _check_interrupt)
  │
  └─ 4. 返回更新后的 state
```

**与现有架构的协作**：`subagent_exec_node` 执行完当前所有层后返回 `plan` 节点。
- 若仍有 `running` 任务（理论上不应出现，除非被中断），返回 `TASK_EXECUTION` 让 `plan` 再次路由到 `sub_exec`
- 若全部完成或失败，返回 `RESULT_SYNTHESIS` 让 `plan` 进行结果汇总

#### 2.4.2 核心代码

```python
import concurrent.futures
from graph_agent.orchestration.dag import topological_layers

# 并行执行的最大线程数上限（防止 PlanAgent 拆分过多任务时创建过量线程）
_MAX_PARALLEL_WORKERS = 8


def _execute_task(
    task: SubTask,
    config: SubAgentConfig,
    local_state: dict,
) -> tuple[str, str, str | None]:
    """执行单个子任务（在工作线程中运行）。

    使用 local_state（state 浅拷贝）避免多线程竞争。
    若任务内部触发 AskUserException，直接传播到主线程处理。

    Returns:
        (task_id, result_or_empty, error_or_none)
    """
    try:
        result = _run_subagent(config, task.description, task.task_id, local_state)
        return (task.task_id, result, None)
    except AskUserException:
        raise  # 传播到主线程处理中断
    except Exception as e:
        return (task.task_id, "", str(e))


def _execute_layer_parallel(
    tasks: list[SubTask],
    state: OrchestrationState,
) -> list[dict]:
    """并行执行一层中的所有任务。

    对每个任务：
    - 使用 state 浅拷贝确保线程安全
    - 无法找到 config 的任务直接标记失败

    Returns:
        [{"task_id": str, "result": str, "error": str, "status": str}, ...]
    """
    futures_map = {}
    worker_count = min(len(tasks), _MAX_PARALLEL_WORKERS)

    # 预处理：区分有效任务和无 config 任务
    valid_tasks = []
    skipped_results = []
    for task in tasks:
        try:
            config = _registry.get(task.assigned_agent)
        except KeyError:
            config = None
        if config is None:
            skipped_results.append({
                "task_id": task.task_id,
                "result": "",
                "error": f"未找到 SubAgent 配置: {task.assigned_agent}",
                "status": "failed",
            })
        else:
            valid_tasks.append((task, config))

    with concurrent.futures.ThreadPoolExecutor(max_workers=worker_count) as executor:
        for task, config in valid_tasks:
            local_state = dict(state)
            future = executor.submit(_execute_task, task, config, local_state)
            futures_map[future] = task

        results = list(skipped_results)
        for future in concurrent.futures.as_completed(futures_map):
            task = futures_map[future]
            try:
                task_id, result, error = future.result()
                results.append({
                    "task_id": task_id,
                    "result": result if not error else "",
                    "error": error or "",
                    "status": "failed" if error else "completed",
                })
            except AskUserException:
                # 取消所有未开始的任务（已运行的任务需等 executor shutdown 完成，见 4.2 节）
                for f in futures_map:
                    f.cancel()
                # 将已完成任务的部分结果附着到异常对象，防止恢复时重复执行
                e._partial_results = results
                raise

    return results


def _resolve_placeholders_for_task(
    task: SubTask,
    sub_results: dict[str, str],
) -> None:
    """JIT 解析任务描述和输入数据中的 {{task_id}} 占位符。

    在任务执行前调用，使用当前已累积的 sub_results 进行替换。
    同时将解析后的 input_data 追加到任务描述中（同现有 _dispatch_tasks 行为）。
    """
    import re

    def _replace(match: re.Match) -> str:
        key = match.group(1).strip()
        if key.endswith(".result"):
            task_id = key[:-7]
        else:
            task_id = key
        return sub_results.get(task_id, match.group(0))

    task.description = re.sub(r'\{\{(.+?)\}\}', _replace, task.description)
    for k, v in task.input_data.items():
        if isinstance(v, str):
            task.input_data[k] = re.sub(r'\{\{(.+?)\}\}', _replace, v)

    # 将解析后的 input_data 追加到任务描述中（幂等：恢复执行时不重复追加）
    if task.input_data and "【执行参数】" not in task.description:
        param_lines = []
        for k, v in task.input_data.items():
            v_str = str(v)
            if "\n" in v_str:
                param_lines.append(f"- {k}:\n  \"\"\"\n  {v_str}\n  \"\"\"")
            else:
                param_lines.append(f"- {k}: {v_str}")
        params_block = "\n".join(param_lines)
        task.description = f"{task.description}\n\n【执行参数】\n{params_block}"


def subagent_exec_node(state: OrchestrationState) -> dict:
    """SubAgent 执行节点：按拓扑分层并行执行所有 running 状态的子任务。"""
    from graph_agent.acp.checkpoint import _check_interrupt

    task_plan = state.get("task_plan")
    if not task_plan:
        return {}

    # 收集所有 running 任务（首次调用时 = 全部任务）
    running_tasks = [t for t in task_plan.sub_tasks if t.status == "running"]
    if not running_tasks:
        return {}

    # 已完成任务的结果（用于占位符解析和依赖判断）
    sub_results = dict(state.get("sub_results", {}))

    # 拓扑分层：同一层内的任务无相互依赖，可并行执行
    layers = topological_layers(running_tasks)

    # 保护：如果 running_tasks 非空但分层为空，说明存在未检测到的 DAG 环
    if not layers:
        get_tracer().trace_phase(
            "子任务执行", "SubAgent",
            "DAG 分层失败，将所有 running 任务标记为 failed（可能为循环依赖）",
        )
        for t in running_tasks:
            t.status = "failed"
            t.error = "DAG 分层失败：可能存在循环依赖或依赖缺失"
        return {
            "phase": OrchestrationPhase.RESULT_SYNTHESIS,
            "sub_results": sub_results,
            "task_plan": task_plan,
            "ga_messages": result_messages,
            "messages": [],
        }

    get_tracer().trace_phase(
        "子任务执行", "SubAgent",
        f"DAG 拓扑分层并行执行: {len(layers)} 层, {len(running_tasks)} 个任务",
    )

    result_messages = []

    for layer_idx, layer in enumerate(layers):
        get_tracer().trace_phase(
            f"执行第 {layer_idx} 层",
            "SubAgent",
            f"并行执行 {len(layer)} 个任务: {[t.task_id for t in layer]}",
        )

        # 检查前置任务失败 → 级联标记失败，跳过执行
        executable_tasks = []
        for task in layer:
            failed_deps = [
                dep_id for dep_id in task.dependencies
                if any(
                    t.task_id == dep_id and t.status == "failed"
                    for t in task_plan.sub_tasks
                )
            ]
            if failed_deps:
                task.status = "failed"
                task.error = f"前置任务失败: {failed_deps}"
                sub_results[task.task_id] = ""
                result_messages.append(create_assistant_message(
                    content=f"前置任务失败，跳过执行: {failed_deps}",
                    name="SubAgent",
                    message_type=MessageType.SUBAGENT_TASK_RESULT,
                    metadata={"task_id": task.task_id, "status": "failed"},
                ))
            else:
                executable_tasks.append(task)

        if not executable_tasks:
            _check_interrupt(state)
            continue

        # JIT 解析本层每个任务的占位符（此时前置层的结果已就绪）
        for task in executable_tasks:
            _resolve_placeholders_for_task(task, sub_results)

        try:
            layer_results = _execute_layer_parallel(executable_tasks, state)
        except AskUserException as e:
            # 处理同层中已完成任务的部分结果（防止恢复时重复执行）
            partial_results = getattr(e, '_partial_results', [])
            for r in partial_results:
                tid = r["task_id"]
                sub_results[tid] = r["result"]
                pt = next((t for t in task_plan.sub_tasks if t.task_id == tid), None)
                if pt:
                    pt.status = r.get("status", "completed")
                    pt.result = r["result"]
                    if r.get("error"):
                        pt.error = r["error"]

            # 中断处理：保存当前状态到异常对象（同现有逻辑）
            full_state = dict(state)
            if e.state and e.state.get("messages"):
                full_state["_subagent_messages"] = e.state["messages"]
            if e.state and e.state.get("ask_user_tool_id"):
                full_state["_ask_user_tool_id"] = e.state["ask_user_tool_id"]
            rbac_token = (e.state or {}).get("_rbac_pending_escalation")
            if rbac_token:
                full_state["_rbac_pending_escalation"] = rbac_token
            full_state["task_plan"] = task_plan
            full_state["sub_results"] = sub_results
            raise AskUserException(
                question=e.question,
                options=e.options,
                require_approval=e.require_approval,
                state=full_state,
            ) from e

        # 收集本层结果
        for r in layer_results:
            tid = r["task_id"]
            sub_results[tid] = r["result"]
            task = next(t for t in task_plan.sub_tasks if t.task_id == tid)
            task.status = r.get("status", "completed")
            task.result = r["result"]
            if r["error"]:
                task.error = r["error"]

            try:
                config = _registry.get(task.assigned_agent)
                agent_name = config.name
            except KeyError:
                agent_name = task.assigned_agent or "unknown"
            msg = create_assistant_message(
                content=r["result"] or r["error"],
                name=agent_name,
                message_type=MessageType.SUBAGENT_TASK_RESULT,
                metadata={
                    "task_id": tid,
                    "agent_name": agent_name,
                    "status": task.status,
                },
            )
            result_messages.append(msg)

        _check_interrupt(state)

    # 判断下一阶段
    has_pending = any(t.status == "pending" for t in task_plan.sub_tasks)
    has_running = any(t.status == "running" for t in task_plan.sub_tasks)

    return {
        "phase": (
            OrchestrationPhase.RESULT_SYNTHESIS
            if not has_pending and not has_running
            else OrchestrationPhase.TASK_EXECUTION
        ),
        "sub_results": sub_results,
        "task_plan": task_plan,
        "ga_messages": result_messages,
        "messages": agent_messages_to_langchain(result_messages),
    }
```

## 3. 线程安全设计

### 3.1 问题分析

并行执行时，多个 SubAgent 线程同时运行，需确保以下共享资源的安全访问：

| 共享资源 | 当前状态 | 并行风险 | 解决方案 |
|---------|---------|---------|---------|
| `_agent_context` (threading.local) | 已有线程局部存储 | **无风险** — 天然线程安全 | 无需改动 |
| `ToolCenter` 工具实例 | 全局共享 | 工具如有内部可变状态则不安全 | 当前工具均为无状态纯函数；`_agent_context` 使用 thread-local 存储 Agent 身份 |
| `state` dict 写入 | 主线程集中写入 | `_run_subagent` 会修改 state（如 `_injected_messages`, `_subagent_messages`） | `_execute_task` 不直接修改 state，只返回结果 |
| `task.status` / `task.result` | 主线程写入 | 多线程同时修改同一 `SubTask` 对象 | 由主线程在收集结果时写入，工作线程只返回结果 |
| `sub_results` dict | 主线程构建 | 多线程同时写入 | 由主线程在收集结果后统一写入 |
| LLM 调用 | 每次调用创建新实例 | **无风险** — LLMFactory 创建独立实例 | 无需改动 |

### 3.2 关键决策：工作线程使用 state 浅拷贝

工作线程（`_execute_task`）接收 `local_state = dict(state)` 的浅拷贝，所有对 state 字典的读写（如 `_injected_messages`、`_subagent_messages`）只影响拷贝副本，不会污染主线程的原始 state。任务完成后，工作线程仅返回结果字符串，所有对 `task` 对象和 `sub_results` 的修改由主线程统一进行。

```python
# _execute_layer_parallel 中每个任务使用独立的 state 浅拷贝
local_state = dict(state)
future = executor.submit(_execute_task, task, config, local_state)
```

### 3.3 `_run_subagent` 的线程安全分析

`_run_subagent` 内部会读写 `state` dict 的以下 key：

- `state["_subagent_messages"]` — 恢复执行时的消息列表（读 + 写 null）
- `state["_injected_messages"]` — 注入的消息（读 + 写 null）
- `state["_rbac_pending_escalation"]` — RBAC 令牌（读）

使用浅拷贝后，这些操作安全无竞争。注意：恢复执行场景（`_subagent_messages` 非空）不会出现在并行执行中——恢复时 `subagent_exec_node` 被重新调用，中断的任务会重新执行。

### 3.4 工具实例的线程安全

`ToolCenter` 中的工具实例是全局共享的。当前设计中，每个工具被设计为无状态的纯函数（输入参数 → 返回结果），不修改实例内部状态。如果后续引入有状态工具，需要：
- 使用 `threading.local()` 为每个线程维护独立状态
- 或使用锁保护关键区域

当前 `_agent_context` 已使用 `threading.local()` 存储当前线程的 Agent 身份，工具通过 `get_current_agent_name()` 获取上下文，天然线程安全。

## 4. 异常处理与中断恢复

### 4.1 单任务异常

单个子任务执行失败（抛出普通异常）时，`_execute_task` 捕获异常并返回错误信息，不影响同层其他任务的并行执行。失败任务标记为 `status = "failed"`。

**前置任务失败时的级联处理**：

由于采用一次性派发模式，所有任务在首次调用时就被标记为 `running`。前置任务失败后，其下游任务的处理分两步：

1. **执行时检测**：`topological_layers` 将失败任务正常视为"已完成"（仅从依赖角度），其下游任务会进入后续层。在执行前，`subagent_exec_node` 检查依赖任务的最终状态，若前置任务 `failed`，将当前任务也标记为 `failed`：

```python
# 在 subagent_exec_node 中，每层执行前检查依赖状态
for task in layer:
    failed_deps = [
        dep_id for dep_id in task.dependencies
        if any(
            t.task_id == dep_id and t.status == "failed"
            for t in task_plan.sub_tasks
        )
    ]
    if failed_deps:
        task.status = "failed"
        task.error = f"前置任务失败: {failed_deps}"
        sub_results[task.task_id] = ""
        result_messages.append(create_assistant_message(...))
        continue  # 跳过该任务，不提交到线程池
```

2. **派发时检测**（保险机制）：若 `plan_router` 再次路由到 `plan` 节点，`_dispatch_tasks` 中的 `failed_deps` 检查作为兜底保护（见 2.3 节代码）。

### 4.2 AskUserException 中断

当某个正在并行执行的任务触发 `AskUserException`（用户交互工具被调用）时：

1. `_execute_task` 将异常传播到 `_execute_layer_parallel`
2. `_execute_layer_parallel` 的 `as_completed` 循环捕获异常：
   - 调用 `future.cancel()` 尝试取消同层其他尚未开始的任务
   - **将已完成任务的部分结果附着到异常对象**（`e._partial_results = results`），防止恢复时已完成的任务被重复执行
   - **重要限制**：`future.cancel()` 无法中断已在运行的线程。`ThreadPoolExecutor` 的 `__exit__` 会调用 `shutdown(wait=True)`，阻塞等待所有已运行线程完成。这意味着 AskUserException 实际传播到上层时，同层其他任务已经执行完毕（而非被立即中断）
3. 重新抛出 `AskUserException`，由 LangGraph 的中断机制处理
4. `subagent_exec_node` 在保存 state 前，先从 `e._partial_results` 提取已完成任务的结果并写入 `sub_results` 和 `task.status`，确保这些结果不会丢失
5. 恢复执行时，已完成任务的状态为 `completed`（已写入），不参与重新分层；被中断的任务状态保持 `running`，会在 `topological_layers` 重新分层时正确置于首层重新执行

**为什么接受这一限制？**
- LLM API 调用是同步阻塞的，Python 线程不支持强制中断
- SubAgent 任务执行时间有限（受 `max_iterations` 约束，通常单任务 < 60s）
- 外部效果（文件写入等）的幂等性由各工具自行保证

### 4.3 用户手动中断

用户通过 ACP 协议发送中断信号时，`_check_interrupt(state)` 会在每层执行完毕后被检查。重构后需要在层内执行期间无法响应中断的问题。

**方案**：保留每层执行后的 `_check_interrupt` 调用。对于长时间运行的层内任务，中断只能等待当前层完成。这是一个可接受的折衷——单层内的任务通常执行时间不会太长（受 `max_iterations` 限制），且当前串行模式下也是等待单个任务完成。

### 4.4 中断恢复时重新分层的保证

恢复执行时，`subagent_exec_node` 会被再次调用。此时任务状态分布：

| 任务状态 | 含义 | sub_results 中有结果？ | 恢复后行为 |
|---------|------|----------------------|-----------|
| `completed` | 中断前已执行完成 | 是 | 不参与分层，结果保留 |
| `failed` | 中断前已失败 | 否 | 不参与分层，下游任务被级联标记 |
| `running` | 中断前正在执行（被打断） | 否 | 重新参与拓扑分层，重新执行 |
| `pending` | 理论上不应出现（一次派发模式） | 否 | 进入 `_dispatch_tasks` 正常派发（检查 failed_deps 后标记为 running 或 failed） |

`topological_layers` 对 `running_tasks` 重新分层：
- 已完成任务的依赖已被第 2.2 节的逻辑过滤（只统计任务集内部的依赖）
- 被打断的任务（`running`）重新参与分层，其依赖的前置任务可能已完成（`completed`），因此入度降低
- 若前置任务也在 `running` 中（同一中断批次），则依赖关系保留，在分层中处于下游层

## 5. PlanAgent Prompt 增强

需要在 `prompts/plan/plan_generation.txt` 中增加关于 DAG 依赖编排的指导：

```markdown
## 依赖关系编排规则（DAG）
1. 仔细分析子任务间的数据依赖关系，只声明真正需要的依赖
2. 两个子任务如果没有数据依赖，不要添加依赖关系——它们可以并行执行
3. 依赖关系必须形成有向无环图（DAG），禁止循环依赖
4. 如果一个任务需要多个前置任务的结果，将它们都列在 dependencies 中
5. 使用 {{{{task_id}}}} 占位符语法引用前置任务结果，同时确保 dependencies 中包含了对应的 task_id
6. execution_strategy 字段仅用于 PlanAgent 表达整体执行意图，runtime 会根据 dependencies 自动计算拓扑分层，不会直接使用该字段做决策
```

注：prompt 模板中 `{{{{task_id}}}}` 是 Jinja2 风格的转义语法，在 `PromptLoader` 渲染后会变为 `{{task_id}}`，最终由 `_resolve_placeholders_for_task` 解析。

### 5.1 `execution_strategy` 字段的新角色

重构后，`execution_strategy` 不再是驱动串行/并行行为的开关，而是作为 PlanAgent 的**辅助信号**使用：

- **LLM 推理提示**：要求 LLM 输出该字段有助于 LLM 在生成 dependencies 时自我校验——若输出 "parallel" 但 dependencies 形成长链，则存在矛盾
- **可观测性**：`get_tracer()` 记录该字段，便于排查 PlanAgent 编排质量（例如检测 LLM 是否过度串行化独立任务）
- **未来扩展**：当 PlanAgent 误将独立任务标记为依赖关系时，可对比 `execution_strategy` 和实际 DAG 拓扑深度，触发 GuardAgent 重新审核
- **不驱动执行**：实际执行顺序完全由 `topological_layers(dependencies)` 决定，确保行为正确性

## 6. 执行时序图

```
用户输入
  → GuardAgent [意图识别]
    → PlanAgent [DAG 任务分解]
      → GuardAgent [方案审核]
        ├─ 驳回 → PlanAgent [修订]
        └─ 通过 → PlanAgent [一次性派发全部任务为 running]
                    │         (不再检查依赖，全部派发)
                    │
                    ├→ subagent_exec_node [拓扑分层]
                    │   │
                    │   ├─ 第0层: task_1 ∥ task_2  ← 并行 (无依赖)
                    │   │   ├ _execute_task(task_1)  ← Thread-1
                    │   │   └ _execute_task(task_2)  ← Thread-2
                    │   │   (等待全部完成，收集结果)
                    │   │
                    │   ├─ 第1层: task_3 ∥ task_4  ← 并行 (依赖 task_1/task_2 已完成)
                    │   │   ├ JIT 解析 task_3 中的 {{task_1}} 占位符
                    │   │   ├ JIT 解析 task_4 中的 {{task_2}} 占位符
                    │   │   ├ _execute_task(task_3)  ← Thread-1
                    │   │   └ _execute_task(task_4)  ← Thread-2
                    │   │   (等待全部完成，收集结果)
                    │   │
                    │   └─ 第2层: task_5  ← 单任务 (依赖 task_3/task_4 已完成)
                    │       ├ JIT 解析 task_5 中的 {{task_3}} {{task_4}} 占位符
                    │       └ _execute_task(task_5)  ← Thread-1
                    │
                    → PlanAgent [汇总结果]  (此时所有任务 completed/failed)
                      → GuardAgent [结果审核]
                        ├─ 驳回 → PlanAgent [修订]
                        └─ 通过 → 返回用户
```

**与旧流程的关键区别**：
- 旧流程：plan 逐批派发 → sub_exec 串行执行 → plan 再派发 → ...（多轮往返）
- 新流程：plan 一次派发 → sub_exec 分层并行执行全部任务 → plan 汇总（单次往返）

### 7.1 数据模型

`SubTask` 和 `TaskPlan` 无需任何字段变更。`dependencies` 字段已存在且被 PlanAgent 正常使用。

无需任何改动。每个 SubAgent 的 ReAct 循环保持不变，工具调用方式保持不变，RBAC 权限检查保持不变。

## 9. 风险与注意事项

1. **LLM API 并发限流**（高优先级）：并行执行时，同一层 N 个 SubAgent 同时进行 ReAct 循环，在最坏情况下（每轮都调用 LLM，max_iterations=5），单层可产生 N × 5 次并发 API 调用。应对措施：
   - `_MAX_PARALLEL_WORKERS = 8` 限制了最大并发线程数
   - 建议在 LLM provider 层加入 `asyncio.Semaphore` 或 token bucket 限流（后续版本）
   - 初期可通过环境变量控制最大并发数，根据 API quota 调整

2. **工具副作用冲突**：如果两个并行任务操作同一文件（例如都写入 `/tmp/output.md`），会产生竞态条件。PlanAgent 应在任务编排时为独立任务指定不同的输出路径，或在 task description 中明确输出文件名。

3. **任务数量膨胀**：PlanAgent 可能将任务拆分为 20+ 个子任务。`_MAX_PARALLEL_WORKERS` 确保线程数可控（最多 8 个并发），但 LLM token 总消耗会线性增长。建议在 PlanAgent prompt 中限制最大子任务数（如 ≤ 10 个）。

4. **日志与追溯**：并行执行导致日志交错输出。`get_tracer().trace_phase()` 需要支持多线程场景，建议在每个 SubAgent 的 trace 中附带 `thread_id` 和 `task_id` 标识以便区分。可在 `_execute_task` 内部设置线程名称：
   ```python
   import threading
   threading.current_thread().name = f"subagent-{task.task_id}"
   ```

5. **中断粒度**：`_check_interrupt` 仅在每层执行完毕后被调用。如果某层包含长时间运行的任务（如 `max_iterations=20`），中断响应延迟可达分钟级。这是当前设计中已知的折衷，后续可考虑在工具执行循环中注入中断检查点。

6. **`plan_router` 的路由决策**：一次派发模式下，`subagent_exec_node` 完成后所有任务应为 `completed` 或 `failed`，不会再有 `pending` 任务。`plan_router` 中 `TASK_EXECUTION` 分支的 `has_running` 检测作为兜底保护——若因中断导致部分任务仍为 `running`，路由回 `sub_exec` 继续执行。
