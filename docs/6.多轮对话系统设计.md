# GraphAgent 多轮对话系统设计文档

## 1. 概述

GraphAgent 当前仅支持单轮对话：每次 `graph.ainvoke()` 都是一个独立的无状态调用，上一轮的对话上下文不会传递到下一轮。本设计为 GraphAgent 添加多轮对话支持，核心包括四个子系统：

- **对话历史存储**：将 `List[MessageBlock]` 序列化为 JSON 持久化到磁盘
- **上下文过滤**：在注入历史上下文前，过滤掉编排流程的内部审核消息（`GUARD_PLAN_REVIEW` / `GUARD_RESULT_REVIEW`）和无效中间产物，仅保留对后续对话有语义价值的信息
- **上下文注入**：在每个新一轮调用时，将过滤后的历史消息注入到编排流程中
- **自适应压缩**：当历史消息 Token 数接近模型上下文窗口上限时，按压缩优先级分级压缩

### 1.1 核心洞察

多轮对话场景下，LLM 的注意力是稀缺资源。每轮编排流程会产生 8 条以上的内部消息（GuardAgent 审核、PlanAgent 方案、任务派发等），但其中仅有一小部分对后续对话有语义价值。若不加过滤地全部注入上下文，不仅浪费 Token 配额，更会稀释 LLM 对关键信息的注意力。

**无效信息源：**

1. **审核类消息**（`GUARD_PLAN_REVIEW` / `GUARD_RESULT_REVIEW`）—— 内容为 `{"approved": true, "feedback": "..."}` 形式的 JSON，是编排流程的内部质量管理信号，对理解对话内容毫无贡献
2. **被驳回的中间产物**（未通过审核的 `PLAN_PROPOSAL` / `PLAN_RESULT_SYNTHESIS`）—— 这些已被重新生成的过期版本，保留在上下文中只会造成混淆
3. **任务派发指令**（`PLAN_TASK_DISPATCH`）—— 内容为 "将计算结果追加写入到..." 等任务描述文本，这些指令的语义已包含在 SubAgent 的执行结果中

**关键信息源（需提升优先级）：**

4. **用户原始输入**（`USER_INPUT`）—— 对话的根本驱动力，P0 不变
5. **最终回复**（`FINAL_ANSWER`）—— Agent 呈现给用户的最终答案，是后续指代消解和追问的基础。目前它仅隐含在 `PLAN_RESULT_SYNTHESIS` 的 JSON 内部，需要显式提取为独立消息类型并提升至 P1

## 3. 核心数据结构

### 3.1 ConversationHistory

`ConversationHistory` 是管理多轮对话历史的核心类，封装 `List[MessageBlock]` 并提供添加、查询、压缩等操作。

```python
from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class ConversationHistory:
    """一次程序运行过程中的完整对话历史。"""

    session_id: str                                     # 会话唯一 ID，格式: YYYYMMDD_HHMMSS
    messages: list[MessageBlock] = field(default_factory=list)
    turn_count: int = 0                                 # 对话轮次计数
    created_at: str = ""                                # ISO 8601 UTC 时间戳
    updated_at: str = ""

    def add_user_message(self, content: str) -> MessageBlock:
        """添加用户消息，轮次 +1。"""
        ...

    def add_agent_messages(self, messages: list[MessageBlock]) -> None:
        """批量添加 Agent 侧消息（GuardAgent / PlanAgent / SubAgent 的输出）。"""
        ...

    def add_final_answer(self, content: str) -> None:
        """添加最终回复消息（P1 优先级，高保留价值）。"""
        ...

    def get_last_n_turns(self, n: int) -> list[MessageBlock]:
        """获取最近 n 轮对话的消息列表。"""
        ...

    def estimate_tokens(self) -> int:
        """估算当前历史消息的总 Token 数。"""
        ...
```

**字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `session_id` | str | 会话唯一标识，程序启动时生成，格式 `YYYYMMDD_HHMMSS` |
| `messages` | list[MessageBlock] | **全量**历史消息列表（含审核消息等内部产物），用于持久化和问题排查 |
| `turn_count` | int | 用户发言次数，每轮对话递增 |
| `created_at` | str | 会话创建时间（ISO 8601 UTC） |
| `updated_at` | str | 最后一次消息追加时间 |

**全量存储 vs 过滤注入：**

`messages` 始终保存完整的编排流程消息（含审核、派发等内部产物），确保磁盘上的历史记录具备完整的可追溯性。注入 LLM 上下文时，通过 `ContextFilter` 过滤后再转换（见第 6 节），确保 LLM 注意力不被无关信息稀释。

### 3.2 轮次标记

`MessageBlock` 本身不存储 `turn_index`。轮次边界通过 `metadata` 中的 `turn_index` 字段标记，仅用户消息（`USER_INPUT`）设置 `turn_index`，Agent 内部消息（GuardAgent / PlanAgent / SubAgent）共享同一 `turn_index`。

```
轮次 1:
  USER_INPUT           (turn_index=1)
  GUARD_INTENT_ANALYSIS(turn_index=1)
  PLAN_PROPOSAL        (turn_index=1)
  GUARD_PLAN_REVIEW    (turn_index=1)  ← 内部审核，不注入上下文
  PLAN_TASK_DISPATCH   (turn_index=1)  ← 派发指令，不注入上下文
  SUBAGENT_TASK_RESULT (turn_index=1)
  PLAN_RESULT_SYNTHESIS(turn_index=1)
  GUARD_RESULT_REVIEW  (turn_index=1)  ← 内部审核，不注入上下文
  FINAL_ANSWER         (turn_index=1)  ← 显式标记，P1 优先级

轮次 2:
  USER_INPUT           (turn_index=2)
  ...
```

## 5. Token 估算

### 5.1 设计原则

每次 LLM 调用前都做精确 Token 计算成本太高（需要调用 tokenizer）。采用**粗粒度估算**策略：

- **英文/代码**: 1 token ≈ 4 字符
- **中文**: 1 token ≈ 1.5 字符
- **混合文本**: 取 1 token ≈ 2.5 字符的平均值

```python
def estimate_tokens(text: str) -> int:
    """基于字符数的粗略 Token 估算。"""
    import re
    chinese_chars = len(re.findall(r'[一-鿿]', text))
    other_chars = len(text) - chinese_chars
    return int(chinese_chars / 1.5 + other_chars / 4)
```

### 5.2 上下文窗口阈值

从配置文件 `config/session_config.yaml` 读取：

```yaml
# 多轮对话与上下文窗口管理配置
```

## 6. 上下文过滤

### 6.1 设计动机

编排流程每轮产生 8+ 条消息，其中审核类消息（`GUARD_PLAN_REVIEW`、`GUARD_RESULT_REVIEW`）和任务派发指令（`PLAN_TASK_DISPATCH`）是编排系统的内部质量控制信号，内容为 JSON 或任务描述文本，对多轮对话的语义理解没有贡献。不加过滤地注入会浪费 Token 配额并稀释 LLM 对关键信息的注意力。

### 6.2 过滤规则

`ContextFilter` 在注入上下文前执行，按以下规则过滤：

| 消息类型 | 处理方式 | 理由 |
|---------|---------|------|
| `USER_INPUT` | **保留** | 对话驱动力，P0 优先级 |
| `GUARD_INTENT_ANALYSIS` | **保留** | 提炼的用户意图，对后续指代消解有价值 |
| `PLAN_PROPOSAL` | **保留**（仅当被审核通过） | 通过审核的方案反映了系统对用户意图的理解 |
| `GUARD_PLAN_REVIEW` | **排除** | 纯审核 JSON，无语义价值 |
| `PLAN_TASK_DISPATCH` | **排除** | 任务派发指令，语义已被 SubAgent 结果覆盖 |
| `SUBAGENT_TASK_RESULT` | **保留** | 任务执行的具体结果 |
| `PLAN_RESULT_SYNTHESIS` | **保留**（仅当被审核通过） | 通过审核的汇总包含最终结论 |
| `GUARD_RESULT_REVIEW` | **排除** | 纯审核 JSON，无语义价值 |
| `FINAL_ANSWER` | **保留** | Agent 呈现给用户的最终回复，P1 优先级 |
| `TOOL_CALL`、`TOOL_RESULT` | **排除** | 工具调用细节，语义已被 SubAgent 结果覆盖 |

### 6.3 实现

```python
# context_filter.py

from graph_agent.message.message_type import MessageType

# 需要从注入上下文中排除的消息类型（内部编排产物）
_EXCLUDED_FROM_CONTEXT: set[MessageType] = {
    MessageType.GUARD_PLAN_REVIEW,
    MessageType.GUARD_RESULT_REVIEW,
    MessageType.PLAN_TASK_DISPATCH,
    MessageType.TOOL_CALL,
    MessageType.TOOL_RESULT,
}

def is_context_eligible(msg_type: MessageType) -> bool:
    """判断消息类型是否应被包含在注入上下文中。"""
    return msg_type not in _EXCLUDED_FROM_CONTEXT
```

过滤在 `ConversationHistory.get_context_messages()` 中统一执行：

```python
def get_context_messages(self) -> list[MessageBlock]:
    """获取应注入 LLM 上下文的消息列表（已过滤内部审核/派发产物）。"""
    return [
        m for m in self.messages
        if is_context_eligible(MessageType(m.message_type))
    ]
```

**注意：** 过滤仅作用于上下文注入环节。`ConversationHistory.messages` 仍保留全量消息（含审核 JSON），确保磁盘持久化记录完整可追溯。

### 6.4 过滤效果

以典型的单轮编排流程为例：

| 阶段 | 消息类型 | 是否注入上下文 |
|------|---------|:---:|
| 用户输入 | `USER_INPUT` | ✓ |
| GuardAgent 意图分析 | `GUARD_INTENT_ANALYSIS` | ✓ |
| PlanAgent 方案制定 | `PLAN_PROPOSAL` | ✓ |
| GuardAgent 方案审核 | `GUARD_PLAN_REVIEW` | ✗ |
| PlanAgent 任务派发 | `PLAN_TASK_DISPATCH` | ✗ |
| SubAgent 执行结果 | `SUBAGENT_TASK_RESULT` | ✓ |
| PlanAgent 结果汇总 | `PLAN_RESULT_SYNTHESIS` | ✓ |
| GuardAgent 结果验收 | `GUARD_RESULT_REVIEW` | ✗ |
| 最终回复 | `FINAL_ANSWER` | ✓ |
| **合计** | **9 条** | **6 条注入（减少 33%）** |

## 7. 压缩策略

### 7.1 MessageType 与压缩优先级

压缩优先级反映消息在多轮对话中的**语义重要程度**，同时指导普通压缩时的删除顺序。

```python
class MessageType(Enum):
    # === P0: 用户侧（绝不压缩）===
    USER_INPUT = "user_input"
    USER_INTERRUPT = "user_interrupt"
    USER_FEEDBACK = "user_feedback"

    # === P1: 系统侧 + 关键决策 + 最终回复（不可丢弃）===
    SYSTEM_PROMPT = "system_prompt"
    SYSTEM_NOTIFICATION = "system_notification"
    GUARD_INTENT_ANALYSIS = "guard_intent_analysis"  # 意图分析（高语义价值）
    GUARD_PLAN_REVIEW = "guard_plan_review"           # 方案审核（内部产物，不注入上下文）
    GUARD_RESULT_REVIEW = "guard_result_review"       # 结果验收（内部产物，不注入上下文）
    FINAL_ANSWER = "final_answer"                     # 最终回复（高语义价值，供后续指代消解）

    # === P2: PlanAgent 层（可摘要但不可丢弃）===
    PLAN_PROPOSAL = "plan_proposal"
    PLAN_TASK_DISPATCH = "plan_task_dispatch"         # 任务派发（内部指令，不注入上下文）
    PLAN_RESULT_SYNTHESIS = "plan_result_synthesis"

    # === P3: SubAgent 层（可压缩）===
    SUBAGENT_TASK_RESULT = "subagent_task_result"

    # === P4: 工具侧（可高度压缩）===
    TOOL_CALL = "tool_call"                           # 不注入上下文
    TOOL_RESULT = "tool_result"                       # 不注入上下文

    # === 通用回退 ===
    AGENT_RESPONSE = "agent_response"
```

**压缩优先级速查表：**

| 优先级 | 消息类型 | 注入上下文 | 压缩行为 |
|--------|---------|:---:|------|
| **P0** | `USER_*` | ✓ | 绝不删除 |
| **P1** | `SYSTEM_*`, `GUARD_INTENT_ANALYSIS`, `FINAL_ANSWER` | ✓(大部分) | 最后删除，仅在极端情况下压缩 |
| **P1** | `GUARD_PLAN_REVIEW`, `GUARD_RESULT_REVIEW` | ✗ | 不注入上下文（全量存储于磁盘） |
| **P2** | `PLAN_PROPOSAL`, `PLAN_RESULT_SYNTHESIS` | ✓ | 保留最近 3 轮，其余可删除 |
| **P2** | `PLAN_TASK_DISPATCH` | ✗ | 不注入上下文 |
| **P3** | `SUBAGENT_TASK_RESULT` | ✓ | 保留最近 2 轮，其余可删除 |
| **P4** | `TOOL_CALL`, `TOOL_RESULT` | ✗ | 优先整组删除，不注入上下文 |

### 7.2 两级压缩机制

```
总 Token 占比        触发动作
─────────────────────────────────────────────
< 60% (39321)        不压缩，正常追加消息
  60% ~ 80%          普通压缩：按 message_type 优先级删除低价值消息
> 80% (52428)        高度压缩：LLM 摘要前 N 条消息，用摘要替换
```

### 7.3 普通压缩（Priority-Based Pruning）

依据 `get_compression_priority()` 返回的优先级（P0-P4）从低到高删除消息。

**压缩流程：**

1. 统计当前历史消息总 Token 数
2. 若未超 `normal_compression_threshold`，跳过
3. 将消息按 `compression_priority` 分组：`{4: [...], 3: [...], 2: [...], 1: [...], 0: [...]}`
4. 从 P4（最低）→ P1（次高）逐组删除：
   - 整组删除后重新估算 Token 数
   - 若 Token 数已降到阈值以下，停止
   - 若删完 P1-P4 仍未达标，保留 P0（用户消息）不动
5. 被删除的消息不进入 `messages` 列表，但可在 `metadata` 中记录 `pruned_count`

**删除规则（按优先级生效）：**

| 优先级 | 消息类型 | 处理方式 |
|--------|---------|---------|
| P4 | `TOOL_CALL`, `TOOL_RESULT` | 优先整组删除 |
| P3 | `SUBAGENT_TASK_RESULT` | 第二批次删除（保留最近 2 轮的 P3） |
| P2 | `PLAN_*` | 第三批次删除（保留最近 3 轮的 P2） |
| P1 | `GUARD_*`, `SYSTEM_*`, `FINAL_ANSWER` | 仅在极端情况下压缩（`FINAL_ANSWER` 与 `USER_INPUT` 同等保护） |
| P0 | `USER_*` | 绝不删除 |

**保留窗口（recent window protection）：**

最近 `keep_recent_turns` 轮对话的所有消息（含 P4）不参与压缩，确保当前上下文不被破坏。

### 7.4 高度压缩（LLM Summarization）

当普通压缩后 Token 数仍超过 `aggressive_compression_threshold`，或者普通压缩无法释放足够空间时，触发高度压缩。

**压缩流程：**

1. 将最近的 `keep_recent_turns` 轮标记为保护窗口，不受压缩
2. 将保护窗口之前的所有消息作为"待摘要区"
3. 构造摘要提示词，调用 LLM 生成结构化摘要
4. 用一条 `AGENT_RESPONSE` 类型的摘要消息替换待摘要区的全部消息
5. 摘要消息的 `metadata` 中记录被替换的消息数量和时间范围

**摘要提示词设计：**

```
你是一个对话摘要助手。请将以下对话历史压缩为一段简洁的结构化摘要。

## 摘要要求
1. 保留每轮用户问题的核心意图
2. 保留 Agent 最终回复的关键结果和数值
3. 忽略工具调用细节和中间推理过程
4. 按时间顺序组织

## 对话历史
{conversation_text}

## 输出格式
{{
  "summary": "一段简洁的摘要文本",
  "turns_covered": "第 1 ~ 5 轮",
  "key_results": ["结果1", "结果2"]
}}
```

**摘要替换规则：**

- 待摘要区的所有消息被替换为一条 `MessageBlock`：
  ```python
  MessageBlock(
      role="assistant",
      content=summary_text,
      name="Compressor",
      message_type=MessageType.AGENT_RESPONSE.value,
      message_id=generate_message_id(),
      metadata={
          "compression": "aggressive",
          "turns_covered": [1, 5],
          "original_message_count": 42,
      },
  )
  ```
- 摘要消息 + 保护窗口内的最新消息 = 压缩后的对话历史

```python
class SummaryCompressor:
    """基于 LLM 摘要的高度压缩器。"""

    def compress(
        self,
        history: ConversationHistory,
        llm,
    ) -> list[MessageBlock]:
        """执行高度压缩。"""
        ...
```

### 7.5 压缩触发时机

压缩检查在每轮对话**完成后**（而非进行中）触发，避免中断用户体验：

```
用户输入 → 编排执行 → 结果输出 → 压缩检查 → 持久化保存
```

## 8. 与编排流程的集成

### 8.1 整体数据流

```
┌─────────────────────────────────────────────────────────┐
│  ConversationHistory.messages (全量，含审核/派发等内部消息) │
│  ┌───┬───┬───┬───┬───┬───┬───┬───┬───┐                  │
│  │Uin│INT│PRO│REV│DIS│SUB│SYN│REV│ANS│  ...             │
│  └───┴───┴───┴───┴───┴───┴───┴───┴───┘                  │
└──────────┬──────────────────────────────────────────────┘
           │ ContextFilter.is_context_eligible()
           ▼
┌─────────────────────────────────────────────────────────┐
│  过滤后消息（仅语义有价值的信息）                           │
│  ┌───┬───┬───┬───┬───┬───┐                              │
│  │Uin│INT│PRO│SUB│SYN│ANS│  ...                         │
│  └───┴───┴───┴───┴───┴───┘                              │
└──────────┬──────────────────────────────────────────────┘
           │ 压缩检查 + LangChain 转换
           ▼
┌─────────────────────────────────────────────────────────┐
│  LangGraph State  (当前轮次 state["messages"])            │
│  ┌─────────────────────────────┬────────────────────┐   │
│  │  历史上下文（过滤+压缩后）    │ 当前轮次消息         │   │
│  └─────────────────────────────┴────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 8.2 注入方式

在每轮 `graph.ainvoke()` 调用前，将**过滤后**的历史消息作为上下文注入到 `state["messages"]` 的**前缀部分**：

```python
# debug.py 中的调用模式
history = ConversationHistory(session_id=...)
priority_compressor = PriorityCompressor(config)
summary_compressor = SummaryCompressor(config)
persistence = ConversationPersistence()

while True:
    user_input = input("你: ")

    # 1. 压缩检查（在全量历史上执行）
    if history.estimate_tokens() > config.normal_threshold_tokens:
        compressed = priority_compressor.compress(history)
        history.replace_messages(compressed)
    # ... 高度压缩同理

    # 2. 过滤 + 注入：从全量历史中筛选语义有价值的消息
    context_msgs = history.get_context_messages()
    context_lc = agent_messages_to_langchain(context_msgs)
    context_lc.append(HumanMessage(content=user_input))

    # 3. 启动编排流程
    result = await graph.ainvoke({"messages": context_lc})

    # 4. 将本轮消息追加到 history（全量存储，含审核消息）
    history.add_user_message(user_input)
    history.add_agent_messages(result.get("ga_messages", []))
    history.add_final_answer(result.get("final_answer", ""))

    # 5. 持久化（全量）
    persistence.save(history)
```

### 8.3 编排节点感知

编排节点（GuardAgent / PlanAgent）的 `_format_context()` 辅助函数已经在从 `state["messages"]` 中读取上下文并注入提示词。由于过滤在注入前完成，编排节点接收到的 `state["messages"]` 已经是净化后的上下文，无需修改即可利用多轮对话信息。

### 8.4 全量存储 vs 过滤注入的关系

```
┌──────────────────┐
│  messages (全量)  │  持久化到磁盘，包含审核 JSON、派发指令等
│  完整可追溯       │  用于问题排查、调试、audit log
└────────┬─────────┘
         │ get_context_messages() — 过滤
         ▼
┌──────────────────┐
│  context (净化)   │  注入 LLM，仅包含语义有价值的信息
│  注意力聚焦       │  USER_INPUT + FINAL_ANSWER + 关键决策
└──────────────────┘
```

**关键原则：**
- `ConversationHistory.messages` = 全量存储（持久化 = 完整记录）
- `ConversationHistory.get_context_messages()` = 过滤后（注入 = 注意力优化）
- 压缩作用于 `messages`（全量层面），不影响过滤规则
- 过滤不修改 `messages`，仅影响注入到 `state["messages"]` 的内容

## 11. 设计优势

1. **注意力优化** — 上下文过滤排除 33% 的内部审核/派发消息，让 LLM 注意力集中在用户意图和最终回复上
2. **零侵入** — 编排节点（GuardAgent / PlanAgent / SubAgent）无需修改，过滤在注入前完成，节点自动感知净化后的上下文
3. **全量可追溯** — `messages` 保留完整编排记录（含审核 JSON），磁盘持久化不丢任何信息；过滤仅影响注入环节
4. **FINAL_ANSWER P1** — 最终回复与用户输入同等级保护，为后续指代消解和追问提供可靠锚点
5. **渐进式压缩** — 两级压缩（删除低优先级消息 → LLM 摘要），逐级释放上下文空间
6. **优先级驱动** — 复用 `get_compression_priority()` 体系，压缩策略与消息类型定义解耦
7. **保护窗口** — 最近 N 轮对话不受压缩影响，确保当前对话连贯性
8. **配置化** — 所有阈值从配置文件读取，适配不同模型和场景
9. **原子持久化** — 先写临时文件再原子替换，避免写入崩溃导致数据损坏
