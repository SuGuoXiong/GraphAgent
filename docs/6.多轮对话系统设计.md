# GraphAgent 多轮对话系统设计文档

## 1. 概述

GraphAgent 当前仅支持单轮对话：每次 `graph.ainvoke()` 都是一个独立的无状态调用，上一轮的对话上下文不会传递到下一轮。本设计为 GraphAgent 添加多轮对话支持，核心包括三个子系统：

- **对话历史存储**：将 `List[MessageBlock]` 序列化为 JSON 持久化到磁盘
- **上下文注入**：在每个新一轮调用时，将历史消息注入到编排流程中
- **自适应压缩**：当历史消息 Token 数接近模型上下文窗口上限时，按压缩优先级分级压缩

## 3. 核心数据结构

### 3.1 ConversationHistory

`ConversationHistory` 是管理多轮对话历史的核心类，封装 `List[MessageBlock]` 并提供添加、查询、压缩等操作。

```python
from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class ConversationHistory:
    """一次程序运行过程中的完整对话历史。"""

    session_id: str                                     # 会话唯一 ID，格式: YYYYMMDD_HHMMSS
    messages: list[MessageBlock] = field(default_factory=list)
    turn_count: int = 0                                 # 对话轮次计数
    created_at: str = ""                                # ISO 8601 UTC 时间戳
    updated_at: str = ""

    def add_user_message(self, content: str) -> MessageBlock:
        """添加用户消息，轮次 +1。"""
        ...

    def add_agent_messages(self, messages: list[MessageBlock]) -> None:
        """批量添加 Agent 侧消息（GuardAgent / PlanAgent / SubAgent 的输出）。"""
        ...

    def get_last_n_turns(self, n: int) -> list[MessageBlock]:
        """获取最近 n 轮对话的消息列表。"""
        ...

    def estimate_tokens(self) -> int:
        """估算当前历史消息的总 Token 数。"""
        ...
```

**字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `session_id` | str | 会话唯一标识，程序启动时生成，格式 `YYYYMMDD_HHMMSS` |
| `messages` | list[MessageBlock] | 全量历史消息列表，按时间顺序追加 |
| `turn_count` | int | 用户发言次数，每轮对话递增 |
| `created_at` | str | 会话创建时间（ISO 8601 UTC） |
| `updated_at` | str | 最后一次消息追加时间 |

**轮次标记：**

`MessageBlock` 本身不存储 `turn_index`。轮次边界通过 `metadata` 中的 `turn_index` 字段标记，仅用户消息（`USER_INPUT`）设置 `turn_index`，Agent 内部消息（GuardAgent / PlanAgent / SubAgent）共享同一 `turn_index`。

```
轮次 1:
  USER_INPUT       (turn_index=1)
  GUARD_INTENT     (不设 turn_index，继承上一条)
  PLAN_PROPOSAL    (不设 turn_index)
  GUARD_REVIEW     (不设 turn_index)
  PLAN_DISPATCH    (不设 turn_index)
  SUBAGENT_RESULT  (不设 turn_index)
  PLAN_SYNTHESIS   (不设 turn_index)
  GUARD_REVIEW     (不设 turn_index)

轮次 2:
  USER_INPUT       (turn_index=2)
  ...
```

## 5. Token 估算

### 5.1 设计原则

每次 LLM 调用前都做精确 Token 计算成本太高（需要调用 tokenizer）。采用**粗粒度估算**策略：

- **英文/代码**: 1 token ≈ 4 字符
- **中文**: 1 token ≈ 1.5 字符
- **混合文本**: 取 1 token ≈ 2.5 字符的平均值

```python
def estimate_tokens(text: str) -> int:
    """基于字符数的粗略 Token 估算。

    中文字符权重更高 (1 token ≈ 1.5 字符)，
    英文字符权重较低 (1 token ≈ 4 字符)。
    """
    import re
    chinese_chars = len(re.findall(r'[一-鿿]', text))
    other_chars = len(text) - chinese_chars
    return int(chinese_chars / 1.5 + other_chars / 4)
```

### 5.2 上下文窗口阈值

从配置文件 `config/session_config.yaml` 读取：

```yaml
# 多轮对话与上下文窗口管理配置
session:
  # 模型上下文窗口（Token 数）
  context_window: 65536

  # 普通压缩触发阈值（占上下文窗口的比例，0.0 ~ 1.0）
  normal_compression_threshold: 0.6

  # 高度压缩触发阈值
  aggressive_compression_threshold: 0.8

  # 每次压缩后保留的最近 N 轮对话（不被压缩）
  keep_recent_turns: 3
```

## 6. 压缩策略

### 6.1 两级压缩机制

```
总 Token 占比        触发动作
─────────────────────────────────────────────
< 60% (39321)        不压缩，正常追加消息
  60% ~ 80%          普通压缩：按 message_type 优先级删除低价值消息
> 80% (52428)        高度压缩：LLM 摘要前 N 条消息，用摘要替换
```

### 6.2 普通压缩（Priority-Based Pruning）

依据 `get_compression_priority()` 返回的优先级（P0-P4）从低到高删除消息。

**压缩流程：**

1. 统计当前历史消息总 Token 数
2. 若未超 `normal_compression_threshold`，跳过
3. 将消息按 `compression_priority` 分组：`{4: [...], 3: [...], 2: [...], 1: [...], 0: [...]}`
4. 从 P4（最低）→ P1（次高）逐组删除：
   - 整组删除后重新估算 Token 数
   - 若 Token 数已降到阈值以下，停止
   - 若删完 P1-P4 仍未达标，保留 P0（用户消息）不动
5. 被删除的消息不进入 `messages` 列表，但可在 `metadata` 中记录 `pruned_count`

**删除规则（按优先级生效）：**

| 优先级 | 消息类型 | 处理方式 |
|--------|---------|---------|
| P4 | `TOOL_CALL`, `TOOL_RESULT` | 优先整组删除 |
| P3 | `SUBAGENT_TASK_RESULT` | 第二批次删除（保留最近 2 轮的 P3） |
| P2 | `PLAN_*` | 第三批次删除（保留最近 3 轮的 P2） |
| P1 | `GUARD_*`, `SYSTEM_*` | 仅在极端情况下压缩（保留 GUARD 的裁决字段） |
| P0 | `USER_*` | 绝不删除 |

**保留窗口（recent window protection）：**

最近 `keep_recent_turns` 轮对话的所有消息（含 P4）不参与压缩，确保当前上下文不被破坏。

```python
class PriorityCompressor:
    """基于消息类型优先级的普通压缩器。"""

    def __init__(self, config: SessionConfig):
        self._config = config

    def compress(
        self,
        history: ConversationHistory,
    ) -> list[MessageBlock]:
        """执行普通压缩，返回压缩后的消息列表。

        1. 估算总 Token 数
        2. 隔离最近 keep_recent_turns 轮（保护窗口）
        3. 对窗口外的消息按优先级删除
        4. 合并保护窗口与被保留的旧消息
        """
        ...
```

### 6.3 高度压缩（LLM Summarization）

当普通压缩后 Token 数仍超过 `aggressive_compression_threshold`，或者普通压缩无法释放足够空间时，触发高度压缩。

**压缩流程：**

1. 将最近的 `keep_recent_turns` 轮标记为保护窗口，不受压缩
2. 将保护窗口之前的所有消息作为"待摘要区"
3. 构造摘要提示词，调用 LLM 生成结构化摘要
4. 用一条 `AGENT_RESPONSE` 类型的摘要消息替换待摘要区的全部消息
5. 摘要消息的 `metadata` 中记录被替换的消息数量和时间范围

**摘要提示词设计：**

```
你是一个对话摘要助手。请将以下对话历史压缩为一段简洁的结构化摘要。

## 摘要要求
1. 保留每轮用户问题的核心意图
2. 保留关键的执行结果和数值
3. 忽略工具调用细节和中间推理过程
4. 按时间顺序组织

## 对话历史
{conversation_text}

## 输出格式
{{
  "summary": "一段简洁的摘要文本",
  "turns_covered": "第 1 ~ 5 轮",
  "key_results": ["结果1", "结果2"]
}}
```

**摘要替换规则：**

- 待摘要区的所有消息被替换为一条 `MessageBlock`：
  ```python
  MessageBlock(
      role="assistant",
      content=summary_text,
      name="Compressor",
      message_type=MessageType.AGENT_RESPONSE.value,
      message_id=generate_message_id(),
      metadata={
          "compression": "aggressive",
          "turns_covered": [1, 5],
          "original_message_count": 42,
          "approximate_tokens_before": 60000,
          "approximate_tokens_after": 500,
      },
  )
  ```
- 摘要消息 + 保护窗口内的最新消息 = 压缩后的对话历史

```python
class SummaryCompressor:
    """基于 LLM 摘要的高度压缩器。"""

    def compress(
        self,
        history: ConversationHistory,
        llm,  # LLM 实例（复用 LLMFactory）
    ) -> list[MessageBlock]:
        """执行高度压缩。

        1. 隔离保护窗口
        2. 提取待摘要区的消息文本
        3. 调用 LLM 生成摘要
        4. 用摘要消息替换待摘要区
        """
        ...
```

### 6.4 压缩触发时机

压缩检查在每轮对话**完成后**（而非进行中）触发，避免中断用户体验：

```
用户输入 → 编排执行 → 结果输出 → 压缩检查 → 持久化保存
```

编排执行过程中，节点内部通过 `_format_context()` 注入到提示词的仅是当前轮次的消息（`state["messages"]`），不涉及历史消息的压缩。历史消息的注入在 `graph.ainvoke()` 调用之前完成（见第 7 节）。

## 7. 与编排流程的集成

### 7.1 整体数据流

```
┌─────────────────────────────────────────────────────┐
│  ConversationHistory (List[MessageBlock])            │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐        │
│  │ T1  │ T1  │ T1  │ T2  │ T2  │ T2  │ ... │        │
│  │User │Agen1│Agen2│User │Agen1│Agen2│     │        │
│  └─────┴─────┴─────┴─────┴─────┴─────┴─────┘        │
└──────────┬──────────────────────────────────────────┘
           │ 压缩后注入
           ▼
┌─────────────────────────────────────────────────────┐
│  LangGraph State  (当前轮次 state["messages"])       │
│  ┌──────────────────────────┬──────────────────┐    │
│  │  历史压缩摘要消息         │ 当前轮次消息       │    │
│  │  (HistoryContextMessage) │ (本轮编排产生的)   │    │
│  └──────────────────────────┴──────────────────┘    │
└─────────────────────────────────────────────────────┘
```

### 7.2 注入方式

在每轮 `graph.ainvoke()` 调用前，将压缩后的历史消息作为上下文注入到 `state["messages"]` 的**前缀部分**：

```python
# debug.py 中的调用模式（修改后）
history = ConversationHistory(session_id=...)
compressor = PriorityCompressor(config)
summary_compressor = SummaryCompressor(config)
persistence = ConversationPersistence()

while True:
    user_input = input("你: ")
    user_msg = create_user_message(user_input)

    # 1. 将历史消息注入 state（压缩后）
    compressed_history = _prepare_context(history, compressor,
                                          summary_compressor, llm)

    # 2. 启动编排流程
    result = await graph.ainvoke({
        "messages": compressed_history + [HumanMessage(content=user_input)],
    })

    # 3. 将本轮消息追加到 history
    history.add_user_message(user_input)
    history.add_agent_messages(extract_agent_messages(result))

    # 4. 压缩检查
    if history.estimate_tokens() > threshold:
        history.messages = compressor.compress(history, ...)

    # 5. 持久化
    persistence.save(history)
```

### 7.3 编排节点感知

编排节点（GuardAgent / PlanAgent）的 `_format_context()` 辅助函数已经在从 `state["messages"]` 中读取上下文并注入提示词。历史消息注入后，这些节点会自动感知到对话历史：

```
GuardAgent 系统提示词:
  ...
  ## 当前对话上下文
  {conversation_context}
```

`_format_context()` 读取 `state["messages"]` 中的历史消息（已注入），取最近 10 条格式化后作为 `{conversation_context}` 的值。这使得 GuardAgent 无需修改即可利用多轮上下文进行意图分析和质量把关。

### 7.4 压缩与注入的关系

```
┌──────────────┐
│  全量历史     │  ConversationHistory.messages (完整列表)
└──────┬───────┘
       │ compression check: estimate_tokens() vs thresholds
       ▼
┌──────────────┐
│  压缩后历史   │  compressor.compress()  /  summary_compressor.compress()
└──────┬───────┘
       │ inject as prefix to state["messages"]
       ▼
┌──────────────┐
│  注入后的     │  [HistoryContext] + [HumanMessage(current)]
│  state        │
└──────────────┘
```

**关键原则：** 压缩作用于 `ConversationHistory` 层面（持久化时），注入作用于每次 `graph.ainvoke()` 之前。压缩后的历史仍然保存在 `ConversationHistory.messages` 中（替换原始消息），持久化保存的是压缩后版本。

## 10. 设计优势

1. **零侵入** — 编排节点（GuardAgent / PlanAgent / SubAgent）无需修改，历史消息通过 `state["messages"]` 前缀注入，节点内已有的 `_format_context()` 自动感知
2. **渐进式压缩** — 两级压缩（删除低优先级消息 → LLM 摘要），逐级释放上下文空间，避免一次性截断丢失关键信息
3. **优先级驱动** — 复用 `get_compression_priority()` 体系，压缩策略与消息类型定义解耦，新增消息类型自动获得合理的压缩优先级
4. **保护窗口** — 最近 N 轮对话不受压缩影响，确保当前对话连贯性
5. **配置化** — 所有阈值（上下文窗口大小、压缩触发比例、保护轮次）均从配置文件读取，适配不同模型和场景
