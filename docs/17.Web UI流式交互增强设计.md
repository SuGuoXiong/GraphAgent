# Web UI 流式交互增强设计

## 1. 现状分析

### 1.1 当前行为

用户提问后，Web UI 的展示内容仅为：

- **阶段变化事件卡片**（`phase_changed`）：蓝色小药丸，显示 "GuardAgent: 意图分析"、"PlanAgent: 方案制定" 等
- **工具调用事件卡片**（`tool_call` / `tool_result`）：黄色小药丸
- **最终结果**（`final_answer`）：一条不可变的 Agent 聊天气泡

### 1.2 核心问题

LLM 在每个阶段的实时输出（逐字流式内容）虽然在 `streamBuffer` 中短暂可见，但在阶段切换时**被丢弃**——仅显示一条 "XxxAgent 输出完成" 的事件卡片，实际内容丢失。用户无法回溯 GuardAgent 意图分析的结果、PlanAgent 的任务计划、或 SubAgent 的执行过程。

### 1.3 现有基础设施（可复用）

| 能力 | 状态 | 说明 |
|------|------|------|
| `llm_stream_start` 事件 | 已有 | 携带 `agent_name`，标记流式开始 |
| `llm_stream_chunk` 事件 | 已有 | 携带 `token`，逐字推送 |
| `llm_stream_end` 事件 | 已有 | 携带 `total_tokens`，标记流式结束 |
| `phase_changed` 事件 | 已有 | 携带 `phase` + `agent_name` + `detail` |
| `final_answer` 事件 | 已有 | 最终答案 |
| 前端 `streamBuffer` | 已有 | 当前仅用于实时显示，阶段切换后丢弃 |
| `currentStreamAgent` | 已有 | 跟踪当前流式内容的来源 Agent |

**结论：后端无需改动，所有必要数据已在 SSE 通道中。改造完全集中在前端。**

### 1.4 当前事件处理逻辑（`handleSSEEvent`，`index.html:515-598`）

```
llm_stream_start → 如果 streamBuffer 非空，输出事件卡片 "XxxAgent 输出完成"，
                    然后重置 buffer，创建新的 typing indicator
llm_stream_chunk → 追加 token 到 streamBuffer，更新 typing indicator 显示
llm_stream_end   → 空操作（注释："内容保留在 typing 气泡中，等待下一阶段"）
final_answer     → 调用 onFinalAnswer，提交为不可变聊天气泡
```

## 2. 设计目标

### 2.1 用户体验目标

1. **实时可见**：用户提问后，每个阶段 LLM 的输出逐字打印在界面上
2. **阶段折叠**：每当一个阶段完成（进入下一阶段），当前阶段的内容自动折叠，只显示阶段标题 + 状态图标
3. **可回溯**：用户可以点击折叠的阶段标题，展开查看该阶段的完整输出
4. **最终结果突出**：所有阶段完成后，最终结果始终展开显示，其他阶段折叠
5. **状态可视化**：运行中的阶段显示旋转动画，已完成的阶段显示勾选标记

### 3.3 事件处理流程改造

#### 3.3.1 `phase_changed` 事件

```
phase_changed 到达时：
  1. 如果存在 currentPhaseId（前一个阶段正在运行）：
     a. 将当前 phaseSection.status 设为 "completed"
     b. 更新 phase-header 中的状态图标（旋转 → ✅）
     c. 折叠该阶段的 phase-body（添加 collapsed class）
     d. 将 streamBuffer 内容计入该阶段的 content 字段
  2. 创建新的 phaseSection：
     a. 构建 DOM 元素（phase-section > phase-header + phase-body）
     b. 状态设为 "running"，显示旋转动画
     c. 插入到聊天消息容器中（在最终结果之前，如果存在的话）
     d. 自动展开该阶段（移除 collapsed class）
  3. 更新 currentPhaseId = 新阶段 ID
  4. 重置 streamBuffer = ""
```

#### 3.3.2 `llm_stream_start` 事件

```
llm_stream_start 到达时：
  1. 不创建新的 phaseSection（阶段容器由 phase_changed 管理）
  2. 更新 currentStreamAgent
  3. 重置 streamBuffer = ""
  4. 如果 currentPhaseId 对应 phaseSection 存在，在 phase-body 中显示 "正在生成..." 占位
```

#### 3.3.3 `llm_stream_chunk` 事件

```
llm_stream_chunk 到达时：
  1. 追加 token 到 streamBuffer
  2. 找到 currentPhaseId 对应的 phaseSection
  3. 实时更新其 phase-body 中的 phase-content：
     phaseSection.content = streamBuffer
     phaseSection.bodyElement.innerHTML = simpleMarkdown(streamBuffer)
  4. 自动滚动到底部
```

#### 3.3.4 `llm_stream_end` 事件

```
llm_stream_end 到达时：
  1. 不做 DOM 操作，内容已通过 chunk 实时渲染
  2. 可选：在 phase-header 旁短暂显示 "完成" 提示（tooltip）
```

#### 3.3.5 `final_answer` 事件

```
final_answer 到达时：
  1. 如果存在 currentPhaseId：
     a. 将其状态设为 "completed"，折叠
  2. 创建最终结果区块（不折叠，特殊样式高亮）
     - 使用 .final-result 类名
     - 显示 🤖 头像 + "最终结果" 标题
     - 内容渲染为 markdown
  3. 清除 currentPhaseId = null
```

### 3.4 折叠 / 展开交互

```javascript
function togglePhase(phaseId) {
  var section = findPhaseSection(phaseId);
  if (!section) return;
  var body = section.querySelector('.phase-body');
  var toggle = section.querySelector('.phase-toggle');
  body.classList.toggle('collapsed');
  toggle.textContent = body.classList.contains('collapsed') ? '▶' : '▼';
}
```

- 已完成阶段默认折叠（`collapsed` class）
- 当前运行阶段始终展开
- 点击 `phase-header` 触发折叠/展开
- CSS transition 控制高度动画（`max-height: 0 → 2000px`，`transition: max-height 0.3s ease`）

### 3.5 特殊阶段处理：任务执行（TASK_EXECUTION）

任务执行阶段可能涉及多个 SubAgent 并行输出。处理方式：

1. **单一流式区域**：所有 SubAgent 的工具调用和输出在同一 `phase-body` 中累积
2. **工具调用内联**：`tool_call` 和 `tool_result` 事件不再创建独立的事件卡片，而是以缩进代码块形式追加到当前阶段的 `phase-body` 末尾
3. **Agent 切换提示**：当 `llm_stream_start` 的 `agent_name` 变化时（SubAgent A → SubAgent B），在 phase-body 中插入一个分割线 "--- **SubAgent-B** 开始执行 ---"

### 3.6 事件卡片降级

当前独立的事件卡片（`addEvent` 生成的 `.event-card`）在阶段容器模式下应降级或移除：

- `phase_changed` 事件卡片：**移除**（由 phase-section 替代）
- `tool_call` / `tool_result` 事件卡片：**降级为内联**（追加到 phase-body 内）
- `execution_complete` / `execution_paused` / `execution_resumed` 事件卡片：**保留**（作为轮次级状态提示）
- `error` 事件卡片：**保留**

### 3.7 最终结果区块

最终结果是用户最关心的内容，需要特殊处理：

- **始终展开**，不可折叠
- **视觉区分**：与阶段容器不同的背景色 / 边框（如淡蓝色左侧边框）
- **Markdown 渲染**：完整支持代码高亮、表格、列表
- **位置**：在所有阶段容器之后、下一条用户消息之前

## 5. 数据流示意

```
SSE 事件流
    │
    ├── phase_changed ──→ 创建 phase-section DOM，collapse 上一个
    │
    ├── llm_stream_start ──→ 标记 currentStreamAgent，初始化 buffer
    │
    ├── llm_stream_chunk ──→ buffer += token
    │                         phaseSection.body.innerHTML = markdown(buffer)
    │
    ├── llm_stream_end ──→ (no-op, 内容已实时渲染)
    │
    ├── tool_call ──→ 追加内联工具卡片到当前 phase-body
    │
    ├── tool_result ──→ 追加内联结果卡片到当前 phase-body
    │
    ├── final_answer ──→ collapse 当前阶段
    │                    创建 final-result 区块（始终展开）
    │
    └── execution_complete ──→ 事件卡片
```

总预估：约 300-350 行增量，在一个文件内完成。

### 6.3 兼容性

- **服务端**：零改动。所有必要事件已通过现有 SSE 通道推送。
- **中断恢复**：`resumeSession` 使用相同的 SSE 事件处理路径，天然兼容。
- **AskUser 交互**：`ask_user` 事件处理保持不变，AskUser 卡片插入在最后阶段之后。
- **多轮对话**：每轮独立创建阶段容器，轮次间通过用户消息自然分隔。

### 6.4 风险点

| 风险 | 应对 |
|------|------|
| `phase_changed` 和 `llm_stream_start` 到达顺序不确定 | 在 `llm_stream_start` 中检查 `currentPhaseId` 是否已存在，未存在则创建默认容器 |
| 某阶段无 LLM 输出（如纯工具调用阶段） | `phase-body` 显示 "该阶段无文本输出" 占位 |
| 阶段内容超长影响性能 | `phase-body` 设置 `max-height: 2000px` + `overflow-y: auto`，内容过长时内部滚动 |
| 并行 SubAgent 流式内容交错 | 所有 SubAgent 输出合并到同一 `TASK_EXECUTION` phase-body，用分割线区分 |

## 8. 初始阶段创建时机

编排流程的第一个阶段是 `INTENT_ANALYSIS`。在 `sendMessage()` 中，收到第一个 `phase_changed` 事件之前，需要在发送 HTTP 请求后立即创建一个 "初始分析" 占位容器，或者等待第一个 `phase_changed` 事件。

**推荐方案**：在 `sendMessage()` 发送请求后，立即创建一个状态为 "running" 的 phase-section，label 为 "正在分析..."，待第一个 `phase_changed` 到达后更新其 label 和 agentName。如果 `phase_changed` 在流式开始之前到达，则在 `phase_changed` 处理中创建容器。
