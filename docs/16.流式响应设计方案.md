# GraphAgent 流式响应设计方案

## 1. 现状分析

### 1.1 当前响应流程（批量模式）

```
Client POST /acp/message → 202 accepted
  └─ Server: asyncio.create_task(_execute_and_push())
       └─ execute_turn()
            ├─ 安装 ACPEventCollector（线程安全 queue.Queue）
            ├─ graph.ainvoke(initial_state)  ← 阻塞等待整个图执行完毕
            │    ├─ guard_node → _call_llm() → llm.invoke()  ← 同步阻塞
            │    ├─ plan_node  → _call_llm() → llm.invoke()  ← 同步阻塞
            │    └─ sub_exec   → _run_subagent() → llm.invoke() × N  ← 同步阻塞
            ├─ collector.drain()  ← 执行完毕后才取出所有事件
            └─ return events
  └─ _execute_and_push: 将 events 批量推入所有 SSE 队列
       └─ SSE generator 收到事件 → 推送给前端
```

**核心问题**：用户在 Web UI 发送消息后，在 `graph.ainvoke()` 完成前看不到任何反馈。体验表现为"点击发送 → 长时间无响应 → 突然全部出现"。

### 1.2 已具备的基础设施

| 组件 | 现状 | 流式支持潜力 |
|------|------|-------------|
| **ACP 协议** | `PushEvent` 已定义 `LLM_STREAM_START`、`LLM_STREAM_CHUNK`、`LLM_STREAM_END` | 已就绪，从未被使用 |
| **SSE 传输** | `HTTPSSETransport._event_queues` 维护 `asyncio.Queue` | 支持实时推送，但目前只在执行完毕后收到事件 |
| **ACPEventCollector** | 使用 `queue.Queue` 收集事件，`drain()` 事后倾泻 | 需改造为实时广播 |
| **LangChain Provider** | `ChatOpenAI` / `ChatAnthropic` / `ChatOllama` 均原生支持 `streaming=True` | 底层能力就绪，未被利用 |
| **_HookEnabledChatModel** | 只包装了 `invoke()`，`__getattr__` 代理其他方法 | 需增加 `astream()` 包装 |
| **前端** | 已有 typing 动画（三点跳动）、`updateTyping()` 替换内容 | 需新增 token 级增量追加 |

### 1.3 关键约束

1. **LangGraph 节点是同步函数**：`guard_node`、`plan_node`、`subagent_exec_node` 均为普通 `def`，在 `graph.ainvoke()` 内部同步执行。无法直接使用 `await llm.astream()`。

2. **同步代码运行在事件循环内**：`execute_turn` 通过 `asyncio.wait_for(graph.ainvoke(), timeout)` 执行，同步节点在事件循环的同一线程中运行。`asyncio.Queue.put_nowait()` 仅操作内部 `deque`，不涉及事件循环调度，可以安全调用。但 SSE 消费者需要等待 `invoke()` 返回控制权给事件循环后才能取出数据，因此 token 将以 burst 形式送达（每次 `invoke()` 结束后一批），而非逐 token 实时送达。

3. **多节点 LLM 调用**：一次请求可能涉及 3~10+ 次 LLM 调用（GuardAgent × 2~3 + PlanAgent × 2 + SubAgent × 1~N），需要标识每次流式响应的来源。

4. **并发会话安全**：多个 SSE 连接可能同时存在，流式回调需与会话绑定，不能使用全局单例。

5. **SubAgent 并行执行线程安全**：`_execute_layer_parallel` 使用 `ThreadPoolExecutor` 并行执行同层 SubAgent。多个工作线程各自调用 `_run_subagent` → `llm.invoke()` → callback。不能共享同一个 callback 对象（`_agent_name` 会竞争），必须每次 LLM 调用创建独立 callback 实例。

6. **并行 SubAgent 的 token 延迟**：worker 线程中的 `put_nowait()` 向 `asyncio.Queue` 写入数据是安全的（GIL 保护），但 SSE 消费者在事件循环中运行，只有在 `ThreadPoolExecutor` 的 `as_completed()` 迭代 yield 时（即一个 worker 任务完成）才能处理队列中的事件。因此同一层多个 SubAgent 的 token 会在各自任务完成后才 burst 送达前端，非实时交织。

## 2. 方案设计

### 2.1 总体架构

```
┌──────────┐   SSE    ┌──────────────────┐  live_push()  ┌─────────────────────┐
│  Web UI  │ ◄────── │ HTTPSSETransport │ ◄─────────── │  ACPLiveBroadcaster  │
│          │          │  _event_queues   │               │  (替代 EventCollector) │
└──────────┘          └──────────────────┘               └──────────┬──────────┘
                                                                   │
                                           ┌────────────────────────┤
                                           │ graph 同步调用栈内       │ graph 同步调用栈内
                                           ▼                        ▼
                                  ┌──────────────┐        ┌──────────────┐
                                  │ graph nodes  │        │ LLM Callback │
                                  │ trace_phase  │        │ on_llm_new_  │
                                  │ trace_decision│       │ token()      │
                                  └──────────────┘        └──────────────┘
```

**核心思路**：

1. **传输层提供 `live_push` 回调**：`HTTPSSETransport` 创建一个广播函数，将 ACP 事件实时推入**所有**现有 SSE 队列，通过 `execute_turn` → `ACPLiveBroadcaster` 传递。事件在发生时立即入队，而非事后倾泻。

2. **通过 state 传递 `_live_push`，每次调用创建独立 callback**：`execute_turn` 将广播函数注入 `initial_state["_live_push"]`。各节点的 `_call_llm` 从 state 中读取 `_live_push`，每次 LLM 调用创建独立的 `ACPStreamingCallback` 实例。这样天然支持并发会话隔离，同时也避免了 SubAgent 并行执行时多线程共享同一 callback 对象的 agent_name 竞争条件。

3. **LangChain Callback 实现 Token 流式**：利用 `BaseCallbackHandler.on_llm_new_token()`，在 LLM 配置 `streaming=True` 后，即使是同步 `invoke()` 调用，每个 token 也会触发该回调。

4. **避免事件重复**：当流式 callback 激活时，`_ACPTracerHook` 跳过 `LLM_STREAM_START/END` 事件的收集（它们由 `ACPStreamingCallback` 独家负责）。

### 2.2 变更分层

| 层 | 文件 | 变更内容 |
|----|------|---------|
| **LLM 配置** | `llm/base.py`、`openai_provider.py`、`anthropic_provider.py`、`ollama_provider.py` | 增加 `streaming=True` 配置项 |
| **LLM 包装** | `llm/base.py` `_HookEnabledChatModel` | `invoke()` 中注入 `config["callbacks"]` |
| **流式回调** | `acp/streaming_callback.py` (**新建**) | `ACPStreamingCallback(BaseCallbackHandler)` |
| **事件广播** | `acp/server.py` | 新增 `ACPLiveBroadcaster`；修改 `execute_turn` 签名；修改 `_ACPTracerHook` 避免与 callback 重复 |
| **传输层** | `acp/transports/http_sse.py` | `_execute_and_push` / `_resume_and_push` 传递 `live_push` 回调 |

### 3.2 Callback 层 — ACPStreamingCallback

**新建文件**：`src/graph_agent/acp/streaming_callback.py`

```python
"""ACP 流式回调 —— 将 LLM token 转换为 ACP 流式事件。"""

import logging
from langchain_core.callbacks import BaseCallbackHandler
from graph_agent.acp.protocol import ACPMessage, PushEvent

_logger = logging.getLogger(__name__)


class ACPStreamingCallback(BaseCallbackHandler):
    """LangChain 流式回调：将 on_llm_new_token 转换为 LLM_STREAM_CHUNK 事件。

    配合 streaming=True 使用，在同步 invoke() 调用期间由 LangChain
    内部逐个 token 触发。所有方法对 push_fn 做了异常保护，
    确保回调异常不会中断 LLM 调用。
    """

    def __init__(self, push_fn, agent_name: str = ""):
        super().__init__()
        self._push = push_fn
        self._agent_name = agent_name
        self._token_count = 0

    def on_llm_start(self, serialized, prompts, **kwargs):
        self._token_count = 0
        self._safe_push(ACPMessage.event(PushEvent.LLM_STREAM_START, {
            "agent_name": self._agent_name,
        }))

    def on_llm_new_token(self, token: str, **kwargs):
        self._token_count += 1
        self._safe_push(ACPMessage.event(PushEvent.LLM_STREAM_CHUNK, {
            "agent_name": self._agent_name,
            "token": token,
            "index": self._token_count,
        }))

    def on_llm_end(self, response, **kwargs):
        self._safe_push(ACPMessage.event(PushEvent.LLM_STREAM_END, {
            "agent_name": self._agent_name,
            "total_tokens": self._token_count,
        }))

    def on_llm_error(self, error, **kwargs):
        self._safe_push(ACPMessage.event(PushEvent.LLM_STREAM_END, {
            "agent_name": self._agent_name,
            "total_tokens": self._token_count,
            "error": str(error)[:200],
        }))

    def _safe_push(self, msg: ACPMessage) -> None:
        """异常保护的推送：回调中抛出异常可能被 LangChain 吞没。"""
        try:
            self._push(msg)
        except Exception:
            _logger.warning("ACPStreamingCallback: push failed", exc_info=True)
```

**关键设计决策**：`ACPStreamingCallback` **独家负责** `LLM_STREAM_START`、`LLM_STREAM_CHUNK`、`LLM_STREAM_END` 三个事件。`ACPLiveBroadcaster.collect_llm_start/end` 保留但仅在非流式场景使用（见 3.4 节）。

### 3.3 事件广播层 — ACPLiveBroadcaster

**文件**：`src/graph_agent/acp/server.py`

```python
class ACPLiveBroadcaster:
    """实时事件广播器 —— 事件产生时立即推入 SSE 队列。

    替代 ACPEventCollector 的 post-hoc drain 模式。
    所有 collect_* 方法立即将事件通过 _push 回调广播。
    """

    def __init__(self, session_id: str, push_fn):
        self.session_id = session_id
        self._push = push_fn
        self._phase_count = 0
        self._tool_count = 0

    # collect_phase / collect_decision / collect_tool_call / collect_tool_result
    # 签名与 ACPEventCollector 完全一致，内部直接调用 self._push()
    # ...（实现同 ACPEventCollector，但将 self.push() 替换为 self._push()）

    def collect_llm_start(self, agent_name, prompt_preview):
        """非流式场景：LLM 调用开始（当无 ACPStreamingCallback 时使用）。"""
        self._push(ACPMessage.event(PushEvent.LLM_STREAM_START, {
            "agent_name": agent_name,
            "prompt_preview": prompt_preview[:300],
        }))

    def collect_llm_end(self, agent_name, token_usage=None):
        """非流式场景：LLM 调用结束（当无 ACPStreamingCallback 时使用）。"""
        self._push(ACPMessage.event(PushEvent.LLM_STREAM_END, {
            "agent_name": agent_name,
            "token_usage": token_usage or {},
        }))

    def drain(self):
        """兼容旧接口：实时模式下始终返回空列表。"""
        return []
```

### 3.4 传输层 — 连接 SSE 队列与执行引擎

**文件**：`src/graph_agent/acp/transports/http_sse.py`

核心变更：不再创建独立执行队列，而是创建一个**广播函数**，将事件推入所有现有 SSE 队列。

```python
async def _execute_and_push(self, session_id, content, request_id):
    # 执行开始前对当前 SSE 队列取快照。SubAgent 通过 ThreadPoolExecutor
    # 在 worker 线程中执行，_broadcast 从 worker 线程调用。
    # dict.items() 迭代期间若队列被增删会触发 RuntimeError，
    # 取快照（list of tuples）可避免此问题。
    queues_snapshot = list(self._event_queues.items())

    def _broadcast(msg: ACPMessage) -> None:
        prefix = f"{session_id}_"
        for key, q in queues_snapshot:
            if key.startswith(prefix):
                try:
                    q.put_nowait(msg)
                except asyncio.QueueFull:
                    pass

    try:
        events = await self._server.execute_turn(
            session_id, content,
            live_push=_broadcast,
        )
        # 执行完毕，回到事件循环线程，直接访问 _event_queues 安全
        prefix = f"{session_id}_"
        for key, q in self._event_queues.items():
            if key.startswith(prefix):
                for event in events:
                    try:
                        q.put_nowait(event)
                    except asyncio.QueueFull:
                        pass
    finally:
        self._executing.discard(session_id)


async def _resume_and_push(self, session_id, request_id):
    queues_snapshot = list(self._event_queues.items())

    def _broadcast(msg: ACPMessage) -> None:
        prefix = f"{session_id}_"
        for key, q in queues_snapshot:
            if key.startswith(prefix):
                try:
                    q.put_nowait(msg)
                except asyncio.QueueFull:
                    pass

    try:
        events = await self._server.resume_session(
            session_id,
            live_push=_broadcast,
        )
        prefix = f"{session_id}_"
        for key, q in self._event_queues.items():
            if key.startswith(prefix):
                for event in events:
                    try:
                        q.put_nowait(event)
                    except asyncio.QueueFull:
                        pass
    finally:
        self._executing.discard(session_id)
```

**为什么这样设计**：

1. `acp_events` 端点创建的 SSE 队列 key 为 `{session_id}_{random_uuid}`。如果 `_execute_and_push` 创建独立队列，事件无法送达 SSE 监听者
2. 通过 `session_id` 前缀过滤，避免跨会话事件泄露
3. 同一 session 的多个 SSE 连接（多终端观看）均能收到事件
4. **线程安全**：`_broadcast` 被传入 SubAgent 的 `ThreadPoolExecutor` worker 线程（通过 state → `_live_push` → callback → `_safe_push`）。`queues_snapshot`（list of tuples）在 worker 线程中迭代安全，不会因 event loop 线程增删队列而触发 `RuntimeError: dictionary changed size during iteration`

### 3.5 执行引擎 — 注入流式上下文

**文件**：`src/graph_agent/acp/server.py` — `execute_turn` 内部

```python
# 创建广播器
if live_push is not None:
    broadcaster = ACPLiveBroadcaster(session_id, live_push)
else:
    broadcaster = ACPEventCollector(session_id)  # 回退：非 SSE 场景

# 安装 tracer hook（常规事件 → broadcaster）
hook = _ACPTracerHook(broadcaster, self._tracer,
                      streaming_active=(live_push is not None))
hook.install()

# 将广播函数注入 state（供节点层每次 LLM 调用时创建独立 callback）
if live_push is not None:
    initial_state["_live_push"] = live_push

try:
    result = await asyncio.wait_for(
        self._graph.ainvoke(initial_state),
        timeout=timeout,
    )
finally:
    hook.uninstall()

# 正常完成：收集终端事件
# 注意：broadcaster.drain() 在实时模式下返回 []，终端事件
# （final_answer、execution_complete 等）通过函数返回值传递，
# 由传输层的 _execute_and_push 负责推送。
```

**关键变更**：将 `_live_push`（广播函数）而非 callback 实例注入 state。各节点在每次 LLM 调用时自行创建 `ACPStreamingCallback`，从而天然解决并行 SubAgent 的 agent_name 竞争问题。

### 3.6 节点层 — 每次 LLM 调用创建独立 callback

**设计原则**：不在 state 中存储 callback 实例，而是存储 `_live_push` 函数。每次 LLM 调用（`_call_llm` 或 `_run_subagent` 的 ReAct 循环内）按需创建独立的 `ACPStreamingCallback` 实例。

**为什么这样做**：
1. `_run_subagent` 通过 `ThreadPoolExecutor` 并行执行，多个线程共享同一个 callback 对象会导致 `_agent_name` 竞争（线程 A 设置 name 后，线程 B 覆盖，导致 A 的输出被标为 B 的名称）
2. Callback 是轻量对象（仅存储 `push_fn` 引用 + `agent_name` + 计数），每次 LLM 调用创建无性能负担
3. 无需在 checkpoint 序列化/反序列化时处理 callback

**文件**：`src/graph_agent/node/guard.py`、`plan.py` 中的 `_call_llm`

```python
def _call_llm(system_prompt: str, user_text: str,

              message_type: MessageType = MessageType.AGENT_RESPONSE) -> MessageBlock:
    from graph_agent.llm import LLMFactory
    from graph_agent.session.persistence import sanitize_text
    from langchain_core.messages import SystemMessage, HumanMessage

    provider = LLMFactory.create_from_env()
    llm = provider.get_chat_model()

    live_push = state.get("_live_push")

        HumanMessage(content=sanitize_text(user_text)),
    ]
    config = {"run_name": name}
    if callbacks:
        config["callbacks"] = callbacks
    response = llm.invoke(messages, config=config)
    return ...
```

### 3.7 Tracer Hook 适配 — 避免事件重复

**文件**：`src/graph_agent/acp/server.py` — `_ACPTracerHook`

核心问题：如果 `_ACPTracerHook` 仍然注册 `before_llm_call` / `after_llm_call` hook 来发射 `LLM_STREAM_START/END`，而 `ACPStreamingCallback` 也在 `on_llm_start/end` 中发射同样的事件，前端会收到**重复的 START/END 事件**。

解决方案：`_ACPTracerHook` 增加 `streaming_active` 参数。当流式激活时，**跳过 LLM hook 注册**（token 级事件由 callback 独家负责），只保留 tool 和 phase 事件的收集。

```python
class _ACPTracerHook:
    def __init__(self, collector, tracer, streaming_active: bool = False):
        self._collector = collector
        self._tracer = tracer
        self._streaming_active = streaming_active
        # ... 原有初始化 ...

    def install(self) -> None:
        # ... trace_phase / trace_decision 钩入逻辑不变 ...

        # tool call hook 始终注册
        register.add_session_hook(_collect_tool_call, "before_tool_call", 600, ...)
        register.add_session_hook(_collect_tool_result, "after_tool_call", 600, ...)

        # LLM hook 仅在非流式模式下注册
        if not self._streaming_active:
            register.add_session_hook(_collect_llm_start, "before_llm_call", 600, ...)
            register.add_session_hook(_collect_llm_end, "after_llm_call", 600, ...)

        self._installed = True
```

**事件来源决策表**：

| 事件 | 流式模式（有 callback） | 非流式模式（无 callback） |
|------|----------------------|------------------------|
| `phase_changed` | `_ACPTracerHook` → `collect_phase` | 同左 |
| `tool_call` / `tool_result` | `_ACPTracerHook` → `collect_tool_call/result` | 同左 |
| `llm_stream_start` | **`ACPStreamingCallback.on_llm_start`** | `_ACPTracerHook` → `collect_llm_start` |
| `llm_stream_chunk` | **`ACPStreamingCallback.on_llm_new_token`** | 不发射 |
| `llm_stream_end` | **`ACPStreamingCallback.on_llm_end`** | `_ACPTracerHook` → `collect_llm_end` |

### 3.8 OrchestrationState 扩展

**文件**：`src/graph_agent/orchestration/state.py`

```python
_live_push: Any | None = None  # Callable[[ACPMessage], None]，由 execute_turn 注入
```

**说明**：存储的是广播函数而非 callback 实例。每次 LLM 调用时节点自行创建 `ACPStreamingCallback`。这避免了线程竞争，也无需在 checkpoint 序列化时特殊处理（`serialize_checkpoint` 不会包含此字段，恢复时值为默认 `None`）。

### 3.9 流式事件总览

一轮完整对话的 SSE 事件时序（带流式）：

```
1. phase_changed: phase="意图分析", agent_name="GuardAgent"
2. llm_stream_start:  agent_name="GuardAgent"          ← 来自 ACPStreamingCallback
3. llm_stream_chunk:  agent_name="GuardAgent", token="{"  ← Token 以 burst 推送
4. llm_stream_chunk:  agent_name="GuardAgent", token=""intent""
   ... (N 个 chunk，在 invoke() 执行期间累积，invoke() 返回后 burst 送达)
5. llm_stream_end:    agent_name="GuardAgent"             ← 来自 ACPStreamingCallback

6. phase_changed: phase="方案制定", agent_name="PlanAgent"
7. llm_stream_start:  agent_name="PlanAgent"
8. llm_stream_chunk × N ...
9. llm_stream_end:    agent_name="PlanAgent"

   ... (方案审核、SubAgent 执行、结果汇总等类似)

N.  final_answer:     content="..."                       ← 最终文本回复
N+1. execution_complete: turn_count=N
```

**关于 token 送达时机**：由于同步 `invoke()` 内部触发 `on_llm_new_token` 时事件循环被阻塞，token 事件在 `asyncio.Queue` 中累积，在一次 `invoke()` 返回、LangGraph 在节点间 yield 时 burst 送达前端。前端收到后逐个渲染 `llm_stream_chunk`，形成快速打字机效果。每个 Agent 的 LLM 输出间隔送达（非所有 Agent 一次性全出），相比当前"全黑盒等待"已有质的提升。

## 4. 前端 UI 层

### 4.1 新增 `llm_stream_*` 事件处理

```javascript
case 'llm_stream_start':
  streamBuffer = '';
  currentStreamAgent = msg.payload.agent_name || '';
  if (!typingMsg) {
    typingMsg = addTypingIndicator();
  }
  // 在 typing 气泡上方添加 Agent 来源标注
  updateTyping(typingMsg, '');
  break;

case 'llm_stream_chunk':
  streamBuffer += (msg.payload.token || '');
  if (typingMsg) {
    // 渲染时附带当前 Agent 名称作为前缀
    var display = currentStreamAgent
      ? '**[' + currentStreamAgent + ']**\n' + streamBuffer
      : streamBuffer;
    updateTyping(typingMsg, display);
  }
  break;

case 'llm_stream_end':
  // 流式完成，内容保留在 typing 气泡中
  // 等待下一个 llm_stream_start（新 Agent 开始）或 final_answer 来提交
  break;
```

### 4.2 Agent 切换时的流式内容处理

当 GuardAgent 完成输出后 PlanAgent 开始输出时，前端收到新的 `llm_stream_start`。此时应提交前一个 Agent 的流式内容为轻量事件卡片：

```javascript
case 'llm_stream_start':
  // 提交前一个 Agent 的流式输出
  if (streamBuffer && currentStreamAgent) {
    addEvent('llm', currentStreamAgent + ' 输出完成');
  }
  streamBuffer = '';
  currentStreamAgent = msg.payload.agent_name || '';
  // ...
```

### 4.3 `agent_message` 去重

流式回调已实时呈现 LLM 输出，`agent_message` 中的完整内容不应覆盖流式结果：

```javascript
case 'agent_message':
  // 流式场景：忽略 agent_message（内容已通过 llm_stream_chunk 呈现）
  if (streamBuffer) break;
  if (msg.payload.content) updateTyping(typingMsg, msg.payload.content);
  break;
```


在每次 `sendMessage` 开始时重置。

## 5. 兼容性设计

### 5.1 debug.py 兼容

`debug.py` 通过 `graph.ainvoke(initial_state)` 直接调用，不经过 SSE 传输层：

- `live_push=None`，`execute_turn` 不会注入 `_live_push` 到 state
- `_call_llm` 从 state 读取 `_live_push` 得到 `None`，走原有 `config={"run_name": name}` 路径
- LLM `streaming=True` 仍生效但无 callback 注入，token 不在终端打印（避免日志刷屏）

### 5.2 非流式 LLM Provider 兼容

设置 `LLM_STREAMING=false` 后：
- `on_llm_new_token` 不会被调用，前端收不到 `llm_stream_chunk`
- `_ACPTracerHook` 在非流式模式下注册 LLM hook，发射 `llm_stream_start/end`（不含 chunk）
- 前端看到 "LLM 调用开始 → 等待 → 调用结束"，而不是打字机效果

### 5.3 中断/暂停兼容

`InterruptException` 和 `AskUserException` 场景：
- 已推送的实时事件不受影响（SSE 连接保持）
- 恢复执行时：`_resume_and_push` 创建新的 `live_push` 闭包 → `execute_turn` 重新注入 `_live_push` 到 state
- 前端 streamBuffer 在收到新的 `llm_stream_start` 时被重置

### 5.4 Callback `config["callbacks"]` 合并安全

LangChain 的 `config["callbacks"]` 可能为 `list[BaseCallbackHandler]` 或 `BaseCallbackManager`。节点层 `_call_llm` 直接赋值为 `[streaming_cb]`（新建 list），不尝试与现有 callbacks 合并。这是因为：

1. `_call_llm` 每次都创建新的 `provider.get_chat_model()`，不存在复用场景
2. `streaming_cb` 是唯一的 callback 需求方
3. 避免 `BaseCallbackManager` 合并的复杂度

## 7. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **`asyncio.Queue` 在同步上下文中调用** | `put_nowait` 在同步 `invoke()` 栈中调用 | `asyncio.Queue` 内部使用 `deque`，仅在同一线程中作为事件循环所在线程时安全，已满足条件。防御：`_safe_push` 异常捕获 |
| **streaming=True 兼容性** | 部分 LLM 后端不支持流式 | 环境变量 `LLM_STREAMING=false` 可禁用；此时 tracer hook 自动接管 LLM START/END 事件 |
| **事件量爆炸** | 大模型输出长文本时 SSE 事件量激增 | `asyncio.Queue` maxsize；前端使用 `requestAnimationFrame` 限频渲染 |
| **并发会话 callback 冲突** | 全局单例 callback 导致事件串到其他会话 | 通过 `state["_live_push"]` 传递广播函数，每个 graph 执行 + 每次 LLM 调用创建独立 callback |
| **并行 SubAgent agent_name 竞争** | 多线程共享 callback 对象，_agent_name 被覆盖 | 每次 `invoke()` 创建独立 `ACPStreamingCallback` 实例，天然线程安全 |
| **Worker 线程访问 _event_queues** | `_broadcast` 在 worker 线程迭代 dict，event loop 增删队列可能触发 RuntimeError | 执行前取 `queues_snapshot`（list of tuples），worker 线程迭代快照安全 |
| **并行 SubAgent token 延迟** | 同层多个 SubAgent 并发时，token 在各自任务完成后才送达，不实时交织 | 单层单 Agent 场景不受影响；未来可将节点 async 化实现实时交织 |
| **LangChain 版本兼容** | `on_llm_new_token` 签名可能因版本而异 | `ACPStreamingCallback._safe_push` 异常保护；锁定 langchain 版本 |
| **Hook 系统冲突** | Callback 与现有 `before_llm_call` hook 同时发射事件 | `_ACPTracerHook` 流式模式下跳过 LLM hook；仅 callback 发射 LLM 流式事件 |
