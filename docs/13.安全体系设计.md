# 13. 安全体系设计

## 一、概述

GraphAgent 在多 Agent 编排运行过程中会调用多种工具（文件读写、命令执行、MCP 服务器、Web 请求等），涉及大量文件操作和外部资源访问。为避免文件误删、敏感信息泄露等安全风险，本设计建立一套完整的安全体系，包含两大核心模块：

- **RBAC 安全鉴权体系**：基于"对象-动作-资源-效果"四元组的策略模型，在工具调用前进行权限校验。策略配置 `effect: allow` 的操作直接放行，`effect: deny` 的操作直接拒绝，未匹配任何策略的操作触发用户授权流程。
- **审计日志体系**：对每次工具调用记录完整的审计信息，包括调用来源、操作、对象、风险等级、时间戳和参数等，支持事后追溯和合规审查。

两者通过 Hook 机制无缝集成到现有架构中，不与业务逻辑耦合。

---

## 二、RBAC 安全鉴权体系

### 2.1 核心概念

#### RBAC 策略四元组

RBAC 策略由四个属性构成一个完整的权限判定规则：

| 属性 | 含义 | 示例 |
|------|------|------|
| **Subject（对象）** | 操作的发起者，即 Agent 身份标识 | `plan_agent`、`subagent:code_reviewer` |
| **Action（动作）** | 被调用的工具名称，支持 glob 通配 | `read_file`、`mcp__github__*`、`*` |
| **Resource（资源）** | 工具操作的目标实体。仅对文件和命令工具从入参提取，其他工具使用通配符 `*` | `file:/tmp/data.txt`、`cmd:rm*`、`*` |
| **Effect（效果）** | 授权判定结果 | `allow`、`deny` |

> **核心设计决策**：Action 直接使用工具名而非抽象分类（如 "read"/"write"），消除了 `_infer_action` 映射表的维护成本。Resource 仅在"操作了什么实体"有安全意义时才从工具入参提取——当前仅 `file_tools.py` 和 `cmd_tools.py` 需要此能力，其余工具的 Resource 统一为 `*`。

##### Effect 与授权判定逻辑

Effect 是 RBAC 策略的授权判定结果，仅有两种取值：

| Effect | 含义 | RBAC Hook 行为 |
|--------|------|---------------|
| `allow` | 显式允许 | 直接放行，无需用户授权 |
| `deny` | 显式禁止 | 硬拒绝（ABORT），不可升级 |

**当没有任何策略匹配时**：未命中任何 `allow` 或 `deny` 策略的操作视为"未授权"，触发用户授权流程（AskUser），用户同意后执行。这体现了"默认询问"的安全原则。

判定结果汇总：

| 引擎返回值 | 含义 | RBAC Hook 行为 |
|-----------|------|---------------|
| `allow` | 策略显式允许（`effect: allow`） | 放行（CONTINUE） |
| `deny` | 策略显式禁止（`effect: deny`） | 硬拒绝（ABORT），不可升级 |
| `need_escalation` | 无任何策略匹配 | 抛 AskUserException，用户可批准 |

#### 工具风险等级（与 RBAC 解耦）

`risk_level` 是标注在工具上的属性，**不参与 RBAC 策略匹配**，仅影响工具的执行方式：

| 等级 | 值 | 含义 | 执行方式 |
|------|-----|------|---------|
| `low` | 1 | 只读操作，不会修改系统状态 | **直接执行**（当前进程内） |
| `medium` | 2 | 修改操作，可能影响系统状态 | **进程隔离**（独立子进程执行） |
| `high` | 3 | 破坏性操作或外部不可信工具 | **Docker 沙箱**（专用容器内执行） |

> **设计原则**：RBAC 负责"能不能做"（授权），risk_level 负责"怎么做"（执行方式）。两者独立运作，互不干扰。

#### Subject 命名规范

```
# 内置 Agent
guard_agent                          # GuardAgent
plan_agent                           # PlanAgent

# Skill 驱动的 SubAgent
subagent:<skill_name>                # 例如 subagent:code_reviewer

# 通配符
*                                    # 匹配所有 subject
```

#### Resource 命名规范

Resource 仅在"被操作的实体是什么"有安全区分意义时才从工具入参提取。当前只有 `file_tools.py` 和 `cmd_tools.py` 中的工具需要提取 Resource，其余工具的 Resource 统一为 `*`（通配符）。

```
# 文件路径（file_tools.py 中的工具）
file:<absolute_path>                 # 例如 file:/home/user/data.txt

# 命令（cmd_tools.py 中的工具，提取命令名及其参数）
cmd:<command_line>                   # 例如 cmd:ls -la、cmd:rm -rf /

# 通配符（所有其他工具，含 MCP、Web、JSON 等）
*                                    # Resource 无安全区分意义，用于策略匹配
```

**Resource 提取的可扩展性**：需要提取 Resource 的工具在 `_extract_resource()` 中按工具名显式匹配。如需为其他工具（如 `json_load`）启用路径级 Resource 控制，在匹配列表中增加对应工具名即可。

#### Action 命名规范

Action 直接使用工具注册名称，支持 glob 通配：

```
# 内建工具（精确匹配）
read_file                            # 读取文件
write_file                           # 写入文件
run_command                          # 执行命令
search_web                           # 搜索网页

# MCP 工具（格式为 mcp__<server>__<tool>）
mcp__github__list_issues             # GitHub MCP: 列出 Issues
mcp__production_db__query            # 数据库 MCP: 执行查询

# glob 通配（策略配置中批量匹配）
mcp__*                               # 匹配所有 MCP 工具
read_*                               # 匹配 read_file 等读取类工具
*                                    # 匹配所有工具
```

#### Agent 身份传播（Subject 来源）

RBAC 鉴权需要知道"谁在调用工具"，即当前执行上下文中的 Agent 身份。该身份通过 `HookContext.agent_name` 字段在系统中传播：

```
SubAgent ReAct 循环
       │
       ├─ 设置 thread_local.current_agent_name = config.name
       │
       ├─ LLM 决定调用工具
       │
       ├─ AgentTool.run() 构造 HookContext
       │     hook_context.agent_name = get_current_agent_name()
       │
       └─ RBAC Hook 读取 ctx.agent_name 作为 subject
```

**HookContext 扩展**：

```python
@dataclass
class HookContext:
    # ... 现有字段 ...
    agent_name: str | None = None  # 新增：当前 Agent 身份标识
```

**SubAgent 执行器中设置身份**：

```python
# src/graph_agent/node/subagent.py

import threading
from contextlib import contextmanager

_agent_context = threading.local()

def get_current_agent_name() -> str | None:
    return getattr(_agent_context, 'name', None)

def set_current_agent_name(name: str):
    _agent_context.name = name

@contextmanager
def agent_execution_context(agent_name: str):
    """在 SubAgent 工具执行期间设置 Agent 身份上下文。
    
    用法：
        with agent_execution_context(config.name):
            # SubAgent ReAct 循环在此执行
            # 所有工具调用自动携带此 agent_name
            ...
    """
    old = get_current_agent_name()
    set_current_agent_name(agent_name)
    try:
        yield
    finally:
        if old is not None:
            set_current_agent_name(old)
        else:
            delattr(_agent_context, 'name')
```

使用线程局部存储（`threading.local`）是因为当前架构中每个会话的执行是单线程、同步的，避免了分布式上下文传播的复杂性。

#### Resource 跨平台说明

`Resource` 中的路径（`file:`、`dir:` 前缀）使用 `os.path.abspath()` 进行规范化，而非 `Path.resolve()`：

- **Windows**：根目录为驱动器号（如 `C:\`），路径使用反斜杠，但策略配置中统一使用正斜杠（`/`），匹配引擎内部做归一化处理。
- **Linux/macOS**：根目录为 `/`，路径区分大小写。
- **策略中的通配路径**（如 `dir:/etc`）同时适用于各平台的实际系统目录。编写策略时应考虑目标部署平台的实际路径结构。

路径匹配时统一使用 `os.path.normpath()` 处理平台差异，确保 `file:C:\Users\admin\data.txt` 在归一化后（转为 `file:C:/Users/admin/data.txt`）能匹配策略中的 `file:C:/Users/**`。

针对根目录的策略（如禁止在根目录执行删除）需要根据部署平台配置多条规则：

```yaml
# Linux/macOS
- subject: "*"
  resource: "dir:/"
  action: "delete_directory"
  effect: deny

# Windows（每个驱动器根路径单独配置）
- subject: "*"
  resource: "dir:C:\\"
  action: "delete_directory"
  effect: deny
```

### 2.2 工具风险等级标注

#### 扩展 @tool 装饰器

在现有 `@tool(name, description)` 装饰器基础上增加 `risk_level` 参数（仅影响执行方式，不参与 RBAC 决策）：

```python
# src/graph_agent/tools/base.py

def tool(name: str, description: str, risk_level: str = "medium"):
    """工具装饰器。
    
    Args:
        name: 工具名称
        description: 工具功能描述
        risk_level: 风险等级，取值为 "low" / "medium" / "high"，默认 "medium"
                    low   → 直接执行
                    medium → 进程隔离
                    high  → Docker 沙箱
    """
    def decorator(func: Callable[..., str]) -> AgentTool:
        sig = signature(func)
        parameters = []
        for param in sig.parameters.values():
            parameters.append({
                "name": param.name,
                "type": str(param.annotation) if param.annotation != param.empty else "Any"
            })
        return AgentTool(name, description, func, parameters, risk_level=risk_level)
    return decorator
```

#### AgentTool 扩展

在现有字段基础上增加 `risk_level`，用于决定工具的执行方式。

```python
class AgentTool:
    name: str
    description: str
    parameters: list[dict[str, str]]
    func: Callable[..., str]
    risk_level: str          # 新增：工具风险等级（决定执行方式，不参与 RBAC）

    def __init__(self, name, description, func, parameters, 
                 risk_level="medium"):
        self.name = name
        self.description = description
        self.func = func
        self.parameters = parameters
        self.risk_level = risk_level
```

#### 工具执行方式

`AgentTool.run()` 的完整流程为：先执行 `before_tool_call` Hook 链（含 RBAC 校验），通过后再根据 `risk_level` 选择执行方式调用实际的工具函数，最后执行 `after_tool_call` Hook 链。以下仅展示执行方式分发部分：

```python
# src/graph_agent/tools/base.py

class AgentTool:
    def run(self, tool_args: dict) -> str:
        # 1. HookExecutor.execute("before_tool_call") — 含 RBAC 校验
        # ...
        
        # 2. RBAC 通过后，根据风险等级选择执行方式
        if self.risk_level == "low":
            result = self._run_direct(tool_args)
        elif self.risk_level == "medium":
            result = self._run_isolated(tool_args)
        elif self.risk_level == "high":
            result = self._run_in_sandbox(tool_args)
        else:
            result = self._run_direct(tool_args)
        
        # 3. HookExecutor.execute("after_tool_call")
        # ...
        return result

    def _run_direct(self, tool_args: dict) -> str:
        """直接在当前进程中执行（低风险工具）。"""
        return self.func(**tool_args)

    def _run_isolated(self, tool_args: dict) -> str:
        """在独立子进程中执行（中风险工具）。

        使用 multiprocessing 启动子进程，设置资源限制（内存、CPU 时间），
        超时或资源超限时终止子进程并返回错误。

        注意：_worker 必须是模块级函数（非闭包/内嵌函数），
        因为 multiprocessing 默认使用 pickle 序列化，
        闭包和实例方法无法被 pickle。实际实现中应将 _worker
        定义为模块级函数，通过参数传递 func 和 args。
        """
        import multiprocessing
        import json

        # _worker 实际应定义在模块级别，此处仅为示意
        def _worker(func, args_json, result_queue):
            try:
                args = json.loads(args_json)
                result = func(**args)
                result_queue.put(("ok", result))
            except Exception as e:
                result_queue.put(("error", str(e)))

        result_queue = multiprocessing.Queue()
        args_json = json.dumps(tool_args)
        # 注意：实际实现中 func 也必须是模块级可 pickle 的函数
        p = multiprocessing.Process(
            target=_worker, args=(self.func, args_json, result_queue)
        )
        p.start()
        p.join(timeout=30)  # 30 秒超时
        if p.is_alive():
            p.terminate()
            p.join()
            return "Error: 工具执行超时（30s）"
        if not result_queue.empty():
            status, result = result_queue.get()
            if status == "ok":
                return result
            return f"Error: {result}"
        return "Error: 工具执行异常"

    def _run_in_sandbox(self, tool_args: dict) -> str:
        """在 Docker 沙箱中执行（高风险工具）。

        启动临时 Docker 容器，挂载只读文件系统，限制网络和系统调用，
        执行完成后销毁容器。Docker 不可用时降级到进程隔离模式并发出警告。
        """
        import subprocess
        import json
        import tempfile
        import os

        # 将参数写入临时文件供容器内脚本读取
        with tempfile.NamedTemporaryFile(
            mode='w', suffix='.json', delete=False
        ) as f:
            json.dump({"tool": self.name, "args": tool_args}, f)
            input_file = f.name

        try:
            result = subprocess.run([
                "docker", "run", "--rm",
                "--network", "none",           # 禁用网络
                "--memory", "256m",            # 内存限制
                "--cpus", "1",                 # CPU 限制
                "--read-only",                 # 只读根文件系统
                "--tmpfs", "/tmp:noexec",      # 临时文件系统（禁止执行）
                "-v", f"{input_file}:/input.json:ro",
                "graphagent-sandbox:latest",
                "python", "-m", "sandbox_executor",
                "--input", "/input.json",
            ], capture_output=True, text=True, timeout=60)
            return result.stdout if result.returncode == 0 else f"Sandbox Error: {result.stderr}"
        except subprocess.TimeoutExpired:
            return "Error: 沙箱执行超时（60s）"
        except FileNotFoundError:
            # Docker 不可用时，降级到进程隔离并发出警告
            import warnings
            warnings.warn("Docker 不可用，高风险工具降级为进程隔离执行")
            return self._run_isolated(tool_args)
        finally:
            try:
                os.unlink(input_file)
            except OSError:
                pass
```

#### MCP 工具风险等级

MCP 工具在运行时动态发现，无法在编码期预判其行为，因此 **MCP 工具默认风险等级为 `high`**（Docker 沙箱执行）。用户在验证某个 MCP 工具的安全性后，可在 `mcp_servers.json` 中为指定工具降低风险等级：

```json
{
  "mcpServers": {
    "github": {
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "risk_overrides": {
        "list_issues": "low",
        "search_code": "low",
        "create_pr": "medium"
      }
    },
    "production_db": {
      "transport": "stdio",
      "command": "python",
      "args": ["mcp_db_server.py"],
      "risk_overrides": {
        "query": "low",
        "migrate": "high"
      }
    }
  }
}
```

`risk_overrides` 中的 key 是 MCP 工具名（不含 `mcp__<server>__` 前缀），value 是覆盖后的风险等级。未在 `risk_overrides` 中列出的 MCP 工具保持默认的 `high`。

加载时，MCP 模块将覆盖后的风险等级应用到生成的 `AgentTool` 对象上。完整的 MCP 工具名格式为 `mcp__<server>__<tool>`，匹配时按 server 分组查找对应的 `risk_overrides`。

#### Skill 脚本工具风险等级

Skill 通过 `register_script_tools()` 注册的脚本工具使用 `shell=True` 执行任意命令，风险高于普通内建工具。**脚本工具默认风险等级为 `high`**：

```python
# src/graph_agent/orchestration/subagent.py

def register_script_tools(skill_def: SkillDef, tool_center) -> list[str]:
    for script_path in skill_def.script_files:
        tool = AgentTool(
            name=tool_name,
            description=f"执行 Skill '{skill_def.meta.name}' 的定制脚本: {tool_name}",
            func=make_script_func(script_path),
            parameters=[...],
            risk_level="high",          # 脚本工具默认高风险（Docker 沙箱执行）
        )
        tool_center.register(tool)
```

### 2.3 授权判定规则

#### 默认行为

| Agent 类型 | 说明 |
|------------|------|
| `guard_agent` | GuardAgent 仅做推理审核，不调用工具。通过策略 `effect: deny` 做纵深防御。 |
| `plan_agent` | PlanAgent 仅做规划调度，不调用工具。通过策略 `effect: deny` 做纵深防御。 |
| `subagent:*` | SubAgent 的行为由策略控制。无匹配策略时触发用户授权。 |

> **防御性策略说明**：`guard_agent` 和 `plan_agent` 在架构上不直接调用工具（它们仅做 LLM 推理），对应的 `effect: deny` 策略为"纵深防御"措施——即使未来代码演进导致它们获得了工具调用能力，策略也会在 Hook 层拦截。

#### 授权检查流程

```
subject 发起 tool 调用:
  1. 解析 action = tool.name（工具名称，如 "read_file"、"mcp__github__list_issues"）
  2. 提取 resource：
       file_tools.py → file:<path>
       cmd_tools.py  → cmd:<command>
       其他工具      → * （通配符）
  3. 从 ctx.agent_state 检查是否存在临时授权令牌（escalation token）
       → 令牌有效且匹配当前调用 → 放行（跳过策略查询）
  4. 按优先级顺序遍历策略列表（高→低），首个匹配的策略生效：
       → effect: allow → 放行
       → effect: deny  → 拒绝（ABORT）
  5. 若无匹配策略：
       → 触发 ask_user 授权（need_escalation）
```

#### 权限升级（Escalation）重试机制

当 RBAC 判定为 `need_escalation` 时，执行以下流程：

```
RBAC Hook 判定 need_escalation
       │
       ├─→ 1. 生成一次性授权令牌（escalation_token）
       │      token = {tool_name, resource, expires_at}
       │      expires_at = now + 60s（60 秒内有效，防止重放）
       │
       ├─→ 2. 将令牌暂存到 ctx.agent_state["_rbac_pending_escalation"]
       │
       ├─→ 3. 抛出 AskUserException（携带 tool/ resource 信息）
       │
       └─→ 4. 用户在前端确认
                 │
                 ├─ 批准 → agent_state 中令牌转为已批准状态
                 │          → 工具调用重试 → RBAC Hook 检查令牌 → 放行
                 │          → 执行完成后令牌自动清除
                 │
                 └─ 拒绝 → 令牌清除 → 工具调用被跳过
```

```python
# RBAC Hook 中的 escalation 逻辑（完整 RBAC Hook 实现见 2.5 节）

# 在 engine.evaluate() 返回 "need_escalation" 后：

import uuid, time

token = {
    "id": uuid.uuid4().hex[:8],
    "tool_name": tool_name,          # 工具名（即 action）
    "resource": resource,
    "created_at": time.time(),
    "expires_at": time.time() + 60,  # 60 秒有效
    "approved": False,
    "denied": False,
}
if ctx.agent_state is None:
    ctx.agent_state = {}
ctx.agent_state["_rbac_pending_escalation"] = token

raise AskUserException(
    question=f"Agent '{subject}' 请求执行操作:\n"
             f"  工具: {tool_name}\n"
             f"  资源: {resource}\n"
             f"是否授权?",
    require_approval=True,
    state=ctx.agent_state,
)
```

用户回复后，`agent_state["_rbac_pending_escalation"]["approved"]` 被设为 `True`，工具调用重试时令牌校验通过，调用继续执行。执行完成后令牌在 `after_tool_call` 审计 Hook 中清理。

### 2.4 RBAC 策略配置

#### 配置文件

#### 策略匹配算法

**按列表顺序遍历，首次匹配即生效：**

```
遍历 policies 列表（从上到下）：
  若当前策略同时满足以下三个条件，则匹配成功：
    1. subject 匹配：精确匹配（如 "subagent:code_reviewer"）> 通配符 "*"
    2. action 匹配：使用 fnmatch glob 匹配工具名称
       - "read_file" 精确匹配
       - "read_*" 批量匹配所有读取类工具
       - "mcp__*" 匹配所有 MCP 工具
       - "*" 匹配所有工具
    3. resource 匹配：
       - file:/path 使用 glob 匹配（** 匹配任意层级，* 匹配单层）
       - dir:/path 前缀匹配（匹配该目录及所有子目录）
       - cmd:command 前缀匹配（匹配命令及任意参数）
       - * 匹配所有资源（大多数工具的 Resource 即为此值）

  匹配成功 → 返回该策略的 effect（allow / deny）
  匹配失败 → 继续检查下一条策略

所有策略均不匹配 → 返回 need_escalation（触发用户授权）
```

**判定结果汇总**：

| 引擎返回值 | 含义 | RBAC Hook 行为 |
|-----------|------|---------------|
| `allow` | 策略显式允许（`effect: allow`） | 放行（CONTINUE） |
| `deny` | 策略显式禁止（`effect: deny`） | 硬拒绝（ABORT），不可升级 |
| `need_escalation` | 无任何策略匹配 | 抛 AskUserException，用户可批准 |

**匹配设计原则**：更具体的策略排在更前面。因为采用首次匹配即生效，策略的排列顺序决定了优先级。例如策略 3（`plan_agent` + `action: "read_file"` + `file:**/*.env`）在策略 4（`*` + `action: "*"` + `file:**/*.env`）之前，确保 `plan_agent` 的特权先被匹配到。

#### RBACEngine 实现框架

```python
# src/graph_agent/security/rbac_engine.py

import fnmatch
from typing import Literal

EngineDecision = Literal["allow", "deny", "need_escalation"]

class RBACEngine:
    """RBAC 策略匹配引擎。"""

    def __init__(self, config: "RBACConfig"):
        self.config = config

    def evaluate(self, subject: str, action: str, resource: str) -> EngineDecision:
        """评估一次工具调用是否被允许。

        Args:
            subject: Agent 身份标识（如 "subagent:code_reviewer"）
            action: 工具名称（如 "read_file"、"mcp__github__list_issues"）
            resource: 操作目标（如 "file:/tmp/a.txt"、"cmd:rm -rf /"、"*"）

        Returns:
            "allow": 允许执行（策略 effect: allow 匹配）
            "deny": 硬拒绝（策略 effect: deny 匹配）
            "need_escalation": 无策略匹配，需用户授权
        """
        for policy in self.config.policies:
            if not self._match_subject(policy, subject):
                continue
            if not self._match_action(policy, action):
                continue
            if not self._match_resource(policy, resource):
                continue
            # 匹配成功 → 返回该策略的 effect
            return policy.get("effect", "deny")

        # 无任何策略匹配 → 触发用户授权
        return "need_escalation"

    def _match_subject(self, policy: dict, subject: str) -> bool:
        expected = policy.get("subject", "")
        if expected == "*":
            return True
        return expected == subject

    def _match_action(self, policy: dict, action: str) -> bool:
        """匹配工具名称，支持 glob 通配。

        "read_file" 精确匹配， "read_*" 批量匹配读取工具， "*" 匹配所有。
        """
        expected = policy.get("action", "")
        if expected == "*":
            return True
        return fnmatch.fnmatch(action, expected)

    def _match_resource(self, policy: dict, resource: str) -> bool:
        expected = policy.get("resource", "")
        if expected == "*":
            return True

        # 统一路径分隔符为正斜杠
        norm_resource = resource.replace("\\", "/")
        norm_expected = expected.replace("\\", "/")

        # 按 resource 前缀分发匹配器
        prefix, _, _ = norm_expected.partition(":")

        if prefix == "file":
            # 文件路径：glob 匹配（支持 ** 递归匹配）
            return _glob_match(norm_resource, norm_expected)
        elif prefix == "dir":
            # 目录路径：前缀匹配
            return norm_resource == norm_expected or \
                   norm_resource.startswith(norm_expected.rstrip("/") + "/")
        elif prefix == "cmd":
            # 命令行：fnmatch 匹配（如 "cmd:rm*" 匹配 "cmd:rm -rf /"）
            return fnmatch.fnmatch(norm_resource, norm_expected)

        return norm_resource == norm_expected


def _glob_match(text: str, pattern: str) -> bool:
    \"\"\"支持 ** 递归匹配的 glob 匹配。

    与 fnmatch 不同，** 匹配任意层级目录（包括零层级），
    * 仅匹配单层（不跨越 /）。

    >>> _glob_match("file:/etc/hosts", "file:/etc/**")
    True
    >>> _glob_match("file:/etc/subdir/hosts", "file:/etc/**")
    True
    \"\"\"
    import re
    parts = []
    i = 0
    while i < len(pattern):
        if pattern[i:i+2] == "**":
            parts.append(".*")
            i += 2
        elif pattern[i] == "*":
            parts.append("[^/]*")
            i += 1
        elif pattern[i] == "?":
            parts.append("[^/]")
            i += 1
        elif pattern[i] in ".^$+{}()[]|\\":
            parts.append("\\\\" + pattern[i])
            i += 1
        else:
            parts.append(pattern[i])
            i += 1
    return re.match("^" + "".join(parts) + "$", text) is not None
```

Note: `evaluate()` 不再接受 `risk_level` 参数。RBAC 引擎仅关注 subject/action/resource 的策略匹配，不关心工具的风险等级。

### 2.5 架构集成

#### 整体流程

```
Agent 发起工具调用
       │
       ▼
  AgentTool.run()
       │
       ├─ 1. 构造 HookContext（含 agent_name、tool_name、tool_args）
       │
       ├─ 2. HookExecutor.execute("before_tool_call")
       │     │
       │     ├─→ [优先 10] AskUserInterceptor (CONTROL)
       │     │     └─ 拦截 ask_user 工具调用，抛出 AskUserException
       │     │
       │     ├─→ [优先 15] AuditPreRecord (OBSERVE)
       │     │     └─ 记录调用意图到 agent_state["_audit_prerecord"]
       │     │
       │     ├─→ [优先 20] RBACHook (CONTROL)
       │     │     │
       │     │     ├─ 1. 检查临时授权令牌（escalation token）
       │     │     │     └─ 令牌有效且已批准 → 放行
       │     │     ├─ 2. 解析 subject (ctx.agent_name) / action (tool名称) / resource
       │     │     ├─ 3. 策略引擎按序匹配（首次匹配即生效）
       │     │     ├─ 4. 判定：allow / deny / need_escalation
       │     │     │
       │     │     ├─ allow → CONTINUE → 继续执行
       │     │     ├─ deny  → ABORT (返回拒绝信息)
       │     │     └─ need_escalation → 生成一次性授权令牌 → 抛出 AskUserException
       │     │           │
       │     │           └─→ 用户在 ACP 前端确认/拒绝
       │     │                 │
       │     │                 ├─ 确认 → 令牌设为已批准 → 重试工具调用
       │     │                 └─ 拒绝 → 令牌设为已拒绝 → 工具调用被跳过
       │     │
       │     └─→ [优先 500] TracerHook (OBSERVE)
       │
       ├─ 3. RBAC 通过后，根据 risk_level 选择执行方式：
       │      low    → _run_direct()     当前进程直接执行
       │      medium → _run_isolated()   独立子进程隔离执行
       │      high   → _run_in_sandbox() Docker 容器沙箱执行
       │
       ├─ 4. HookExecutor.execute("after_tool_call")
       │     │
       │     └─→ [优先 590] AuditRecord (OBSERVE)
       │            │
       │            └─ 更新审计记录状态（allowed / escalated / error）
       │
       └─ 5. 返回执行结果
```

#### RBACHook 实现

```python
# src/graph_agent/hook/builtin/rbac_hook.py

@hook(
    checkpoint="before_tool_call",
    priority=20,           # 在所有其他 Hook 之前（仅晚于 audit_prerecord 的 15 和 ask_user_interceptor 的 10）
    hook_type=HookType.CONTROL,
    description="RBAC 权限校验"
)
def rbac_check(ctx: HookContext) -> HookDecision:
    """在工具调用前进行 RBAC 权限校验。"""
    engine = get_rbac_engine()

    # 从 HookContext 获取当前 Agent 身份
    subject = ctx.agent_name or "unknown"
    tool_name = ctx.tool_name
    tool_args = ctx.tool_args or {}

    # 查找工具元数据
    tool = get_tool_center().get_tool(tool_name) if tool_name else None
    if not tool:
        return HookDecision(action=HookAction.CONTINUE)

    action = tool_name              # action 直接使用工具名称
    resource = _extract_resource(tool_name, tool_args)

    # 优先检查临时授权令牌（见 2.3 节权限升级机制）
    pending = (ctx.agent_state or {}).get("_rbac_pending_escalation")
    if pending and _token_matches(pending, tool_name, resource):
        if pending.get("approved") and not _token_expired(pending):
            return HookDecision(action=HookAction.CONTINUE)
        elif pending.get("denied"):
            # 用户拒绝 → 写入审计记录并清理令牌
            prerecord = (ctx.agent_state or {}).get("_audit_prerecord")
            if prerecord:
                record = AuditRecord.from_dict(prerecord)
                record.status = "denied"
                record.escalated = True
                get_audit_logger().write(record)
                del ctx.agent_state["_audit_prerecord"]
            ctx.agent_state.pop("_rbac_pending_escalation", None)
            return HookDecision(action=HookAction.ABORT, reason="用户拒绝授权")

    # 策略判定（不再传入 risk_level）
    decision = engine.evaluate(subject, action, resource)

    if decision == "deny":
        # 直接写入审计记录（后续 OBSERVE Hook 会被 CONTROL ABORT 阻断）
        prerecord = (ctx.agent_state or {}).get("_audit_prerecord")
        if prerecord:
            record = AuditRecord.from_dict(prerecord)
            record.status = "denied"
            get_audit_logger().write(record)
            del ctx.agent_state["_audit_prerecord"]
        return HookDecision(
            action=HookAction.ABORT,
            reason=f"RBAC 拒绝: subject={subject}, resource={resource}, "
                   f"action={action}"
        )

    if decision == "need_escalation":
        _create_escalation_token(ctx, subject, tool_name, resource)
        raise AskUserException(
            question=f"Agent '{subject}' 请求执行操作:\n"
                     f"  工具: {tool_name}\n"
                     f"  对象: {resource}\n"
                     f"是否授权?",
            require_approval=True,
            state=ctx.agent_state,
        )

    return HookDecision(action=HookAction.CONTINUE)
```

**辅助函数**：

```python
# src/graph_agent/hook/builtin/rbac_hook.py

import time

def _token_matches(token: dict, tool_name: str, resource: str) -> bool:
    """检查令牌是否匹配当前工具调用。

    令牌中的 resource 支持前缀匹配，例如令牌中的 "file:/workspace/"
    可以匹配具体的 "file:/workspace/subdir/file.txt"。
    """
    if token.get("tool_name") != tool_name:
        return False
    token_resource = token.get("resource", "")
    return resource == token_resource or resource.startswith(token_resource)


def _token_expired(token: dict) -> bool:
    """检查令牌是否已过期。"""
    return time.time() > token.get("expires_at", 0)


def _create_escalation_token(ctx: HookContext, subject: str, tool_name: str,
                              resource: str) -> dict:
    """生成一次性授权令牌并存储到 agent_state。"""
    import uuid
    token = {
        "id": uuid.uuid4().hex[:8],
        "subject": subject,
        "tool_name": tool_name,
        "resource": resource,
        "created_at": time.time(),
        "expires_at": time.time() + 60,
        "approved": False,
        "denied": False,
    }
    if ctx.agent_state is None:
        ctx.agent_state = {}
    ctx.agent_state["_rbac_pending_escalation"] = token
    # 审计预记录中标记为 escalated
    prerecord = ctx.agent_state.get("_audit_prerecord")
    if prerecord:
        prerecord["escalated"] = True
    return token
```

**关键变化**：`engine.evaluate()` 不再接受 `risk_level` 参数，因为 risk_level 已从 RBAC 策略属性中移除，改为仅决定工具的执行方式。

```python
import os

def _extract_resource(tool_name: str, tool_args: dict) -> str:
    """从工具名称和参数中提取操作资源标识。

    仅对 file_tools.py 和 cmd_tools.py 中的工具提取 Resource，
    其余工具的 Resource 统一返回 "*"（无安全区分意义）。

    注意：使用 os.path.abspath() 而非 Path.resolve()，
    因为 Path.resolve() 在 POSIX 上要求路径存在，使用 abspath 更健壮。
    """
    # file_tools.py：resource 是文件或目录路径
    if tool_name in ("read_file", "write_file", "append_file", 
                     "delete_file", "file_exists"):
        path = tool_args.get("file_path", "")
        return f"file:{os.path.abspath(path) if path else ''}"

    if tool_name in ("list_directory", "delete_directory"):
        path = tool_args.get("dir_path", "")
        return f"dir:{os.path.abspath(path) if path else ''}"

    # cmd_tools.py：resource 是命令行
    if tool_name == "run_command":
        command = tool_args.get("command", "")
        return f"cmd:{command.strip() if command else ''}"

    # 其他所有工具：无实体资源，使用通配符
    return "*"
```

### 3.1 审计记录模型

```python

from dataclasses import dataclass, field
from datetime import datetime, timezone
from typing import Any

@dataclass
class AuditRecord:
    """单条审计日志记录。"""
    # 时间信息
    timestamp: str = ""              # ISO 8601 格式 UTC 时间
    call_sequence: int = 0           # 本次会话中的调用序号

    tool_name: str = ""              # 工具名称
    action: str = ""                 # 工具名称（即 RBAC 中的 action）
    resource: str = ""               # 操作对象
    risk_level: str = ""             # 风险等级（low/medium/high，记录执行方式）

    # 调用参数（脱敏后）
    parameters: dict[str, Any] = field(default_factory=dict)

    # 执行结果
    status: str = ""                 # "allowed" / "denied" / "escalated" / "error"
    result_summary: str = ""         # 结果摘要（前200字符，避免日志膨胀）
    error_message: str = ""          # 错误信息（如有）

    # 授权信息
    escalated: bool = False          # 是否经过用户授权升级
    escalation_approved: bool = False  # 用户是否批准
    escalation_reason: str = ""      # 授权理由

    def to_dict(self) -> dict:
        return {
            "timestamp": self.timestamp,
            "call_sequence": self.call_sequence,
            "subject": self.subject,
            "session_id": self.session_id,
            "tool_name": self.tool_name,
            "action": self.action,
            "resource": self.resource,
            "risk_level": self.risk_level,
            "parameters": self.parameters,
            "status": self.status,
            "result_summary": self.result_summary,
            "error_message": self.error_message,
            "escalated": self.escalated,
            "escalation_approved": self.escalation_approved,
            "escalation_reason": self.escalation_reason,
        }

    @staticmethod
    def from_ctx(ctx: HookContext, subject: str, session_id: str,
                 risk_level: str, action: str, resource: str,
                 status: str, **kwargs) -> "AuditRecord":
        """从 HookContext 构造审计记录。"""
        return AuditRecord(
            timestamp=datetime.now(timezone.utc).isoformat(),
            subject=subject,
            session_id=session_id or "",
            tool_name=ctx.tool_name or "",
            action=action,
            resource=resource,
            risk_level=risk_level,
            parameters=AuditRecord._sanitize_params(ctx.tool_args or {}),
            status=status,
            result_summary=AuditRecord._truncate(str(ctx.tool_result or ""), 200),
            error_message=ctx.tool_error or "",
            **kwargs,
        )

    @staticmethod
    def from_dict(data: dict) -> "AuditRecord":
        """从字典反序列化 AuditRecord（与 to_dict() 互为逆操作）。"""
        return AuditRecord(
            timestamp=data.get("timestamp", ""),
            call_sequence=data.get("call_sequence", 0),
            subject=data.get("subject", ""),
            session_id=data.get("session_id", ""),
            tool_name=data.get("tool_name", ""),
            action=data.get("action", ""),
            resource=data.get("resource", ""),
            risk_level=data.get("risk_level", ""),
            parameters=data.get("parameters", {}),
            status=data.get("status", ""),
            result_summary=data.get("result_summary", ""),
            error_message=data.get("error_message", ""),
            escalated=data.get("escalated", False),
            escalation_approved=data.get("escalation_approved", False),
            escalation_reason=data.get("escalation_reason", ""),
        )

    @staticmethod
    def _sanitize_params(params: dict) -> dict:
        """脱敏参数中的敏感字段。"""
        sanitized = {}
        sensitive_keys = {"password", "token", "api_key", "secret", "auth"}
        for k, v in params.items():
            if any(sk in k.lower() for sk in sensitive_keys):
                sanitized[k] = "***REDACTED***"
            elif isinstance(v, str) and len(v) > 500:
                sanitized[k] = v[:500] + "..."
            else:
                sanitized[k] = v
        return sanitized

    @staticmethod
    def _truncate(text: str, max_len: int) -> str:
        if len(text) <= max_len:
            return text
        return text[:max_len] + "..."
```

### 3.2 审计日志存储

#### 存储格式

采用 **JSON Lines** 格式（每行一条完整 JSON 记录），便于流式写入和按行解析：

```jsonl
{"timestamp":"2026-05-22T10:30:15.123Z","call_sequence":1,"subject":"subagent:code_reviewer",...,"status":"allowed"}
{"timestamp":"2026-05-22T10:30:18.456Z","call_sequence":2,"subject":"subagent:data_engineer",...,"status":"escalated","escalated":true,"escalation_approved":true}
```

#### 存储路径与轮转策略

```
data/audit/
├── audit_2026-05-22.jsonl        # 按天分文件
├── audit_2026-05-21.jsonl
└── ...
```

轮转策略：
- 按天切换日志文件（UTC 0 点轮转）
- 单文件超过 100MB 时提前轮转（附加 `.2` 后缀）
- 保留最近 30 天的审计日志，超过的自动归档或删除（可配置）

#### AuditLogger 实现

采用**缓冲写入**策略：审计记录先写入内存缓冲区，达到批量阈值或会话结束时再刷新到磁盘，避免每次工具调用都产生同步磁盘 I/O。

```python
# src/graph_agent/security/audit.py

import json
import threading
from datetime import datetime, timezone
from pathlib import Path

class AuditLogger:
    """审计日志写入器（线程安全，缓冲写入）。"""

    _instance = None
    _lock = threading.Lock()

    def __new__(cls, log_dir: str = "data/audit"):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._initialized = False
        return cls._instance

    def __init__(self, log_dir: str = "data/audit"):
        if self._initialized:
            return
        self._log_dir = Path(log_dir)
        self._log_dir.mkdir(parents=True, exist_ok=True)
        self._max_size = 100 * 1024 * 1024  # 100MB
        self._retention_days = 30
        self._buffer: list[AuditRecord] = []
        self._buffer_max = 50              # 缓冲区达到 50 条时自动刷新
        self._sequence = 0                  # 全局自增序列号（用于完整性校验）
        self._initialized = True

    def _get_log_path(self) -> Path:
        """获取当天的日志文件路径（考虑轮转）。"""
        today = datetime.now(timezone.utc).strftime("%Y-%m-%d")
        base = self._log_dir / f"audit_{today}.jsonl"

        if base.exists() and base.stat().st_size >= self._max_size:
            i = 2
            while True:
                alt = self._log_dir / f"audit_{today}.{i}.jsonl"
                if not alt.exists() or alt.stat().st_size < self._max_size:
                    return alt
                i += 1
        return base

    def write(self, record: AuditRecord):
        """写入一条审计记录（缓冲模式，线程安全）。"""
        with self._lock:
            self._sequence += 1
            record.call_sequence = self._sequence
            self._buffer.append(record)
            if len(self._buffer) >= self._buffer_max:
                self._flush_locked()

    def flush(self):
        """强制刷新缓冲区（会话结束或轮转时调用）。"""
        with self._lock:
            self._flush_locked()

    def _flush_locked(self):
        """刷新缓冲区到磁盘（需持有 _lock）。"""
        if not self._buffer:
            return
        path = self._get_log_path()
        with open(path, "a", encoding="utf-8") as f:
            for record in self._buffer:
                f.write(json.dumps(record.to_dict(), ensure_ascii=False) + "\n")
        self._buffer.clear()

    def cleanup(self):
        """清理过期日志（> retention_days 天）。"""
        self.flush()  # 清理前先刷新
        cutoff = datetime.now(timezone.utc).timestamp() - self._retention_days * 86400
        for f in self._log_dir.glob("audit_*.jsonl*"):
            if f.stat().st_mtime < cutoff:
                f.unlink()
```

### 3.3 审计 Hook 集成

通过两个 Hook 覆盖审计的完整生命周期。

由于 `HookExecutor` 中 CONTROL Hook 返回 SKIP/ABORT 后会 `break` 中断循环，后续低优先级 OBSERVE Hook 不会被触发，因此审计采用**两段式记录**：

```python
# src/graph_agent/hook/builtin/audit_hook.py

@hook(
    checkpoint="before_tool_call",
    priority=15,            # 在 RBAC (20) 之前运行，确保被拒绝的调用也能被记录
    hook_type=HookType.OBSERVE,
    description="审计预记录：在 RBAC 之前保存调用意图"
)
def audit_prerecord(ctx: HookContext):
    """在所有 CONTROL Hook 之前记录调用意图。

    将初始 AuditRecord (status="pending") 暂存到 agent_state，
    后续在 after_tool_call 或 RBAC 拒绝时更新状态并写入日志。
    此 Hook 为 OBSERVE 类型，不会影响后续 Hook 的执行。
    """
    subject = ctx.agent_name or "unknown"
    tool = get_tool_center().get_tool(ctx.tool_name) if ctx.tool_name else None
    tool_name = ctx.tool_name or ""

    record = AuditRecord(
        timestamp=datetime.now(timezone.utc).isoformat(),
        subject=subject,
        session_id=ctx.session_id or "",
        tool_name=tool_name,
        action=tool_name,            # action 直接使用工具名称
        resource=_extract_resource(tool_name, ctx.tool_args or {}),
        risk_level=tool.risk_level if tool else "medium",
        parameters=AuditRecord._sanitize_params(ctx.tool_args or {}),
        status="pending",
    )
    # 暂存到 agent_state，稍后由 audit_record 或 RBAC 内部更新写入
    if ctx.agent_state is None:
        ctx.agent_state = {}
    ctx.agent_state["_audit_prerecord"] = record.to_dict()


@hook(
    checkpoint="after_tool_call",
    priority=590,
    hook_type=HookType.OBSERVE,
    description="审计日志：写入已通过的调用结果"
)
def audit_record(ctx: HookContext):
    """在 after_tool_call 更新审计记录并写入。

    仅当工具调用通过 RBAC 检查后才会执行到此处（CONTROL 已放行）。
    从 agent_state 读取预记录的数据，更新 result 和 status，写入日志。
    """
    prerecord = (ctx.agent_state or {}).get("_audit_prerecord")
    if not prerecord:
        return

    record = AuditRecord.from_dict(prerecord)
    record.status = "error" if ctx.tool_error else "allowed"
    record.result_summary = AuditRecord._truncate(str(ctx.tool_result or ""), 200)
    record.error_message = ctx.tool_error or ""

    # 若经过用户授权升级，标记 escalated 和批准状态
    # 注意：重试时 audit_prerecord 会覆盖之前的预记录，因此 escalated 需在此处补设
    # 同时检查令牌未过期，避免将已过期降级为策略放行的调用误标为 escalation
    pending = (ctx.agent_state or {}).get("_rbac_pending_escalation")
    if pending and pending.get("approved") and not _token_expired(pending):
        record.escalated = True
        record.escalation_approved = True
        record.escalation_reason = f"用户批准: {pending.get('id', '')}"

    # 清理暂存
    ctx.agent_state.pop("_audit_prerecord", None)
    ctx.agent_state.pop("_rbac_pending_escalation", None)

    get_audit_logger().write(record)
```

**RBAC 拒绝时的审计记录**（在 RBAC Hook 内部直接写入，因为后续 Hook 会被 CONTROL ABORT 阻断）：

```python
# 在 rbac_check() 中，当判定为 deny 时：
```

### 3.4 审计日志完整性

#### 序列号与间隙检测

每条审计记录包含全局自增的 `call_sequence`，用于检测日志是否被篡改或截断：

```python
# 完整性校验工具
def verify_log_integrity(log_dir: str = "data/audit") -> dict:
    """检查审计日志的序列号连续性。"""
    issues = []
    for log_file in sorted(Path(log_dir).glob("audit_*.jsonl")):
        sequences = []
        with open(log_file, "r", encoding="utf-8") as f:
            for line in f:
                record = json.loads(line)
                sequences.append(record.get("call_sequence", 0))

        if not sequences:
            continue
        expected = list(range(min(sequences), max(sequences) + 1))
        missing = set(expected) - set(sequences)
        if missing:
            issues.append(f"{log_file.name}: 缺失序列号 {sorted(missing)}")

    return {"intact": len(issues) == 0, "issues": issues}
```

#### 存储安全

- `data/audit/` 目录的读写权限应限制为仅 GraphAgent 进程用户。
- 对于合规要求较高的场景，建议在生产部署中采用**仅追加日志后端**（如 SQLite WAL 模式、Elasticsearch、或系统 syslog），而非纯 JSONL 文件。
- 当前设计满足开发和小规模部署的需求；生产环境的防篡改需求（HMAC 签名链、加密存储）应在第二阶段实施时根据实际合规要求补充。

```python
class RBACConfig:

        """热加载配置（无需重启）。"""
        self._load()
```

## 七、附录：关键设计决策

### A. 为什么使用 Hook 而非直接修改 AgentTool.run()

Hook 机制天然支持关注点分离。RBAC 鉴权和审计日志是横切关注点，不应侵入业务工具代码。通过 Hook 的优先级机制，可以精确控制 RBAC 校验在所有其他 Hook 之前执行，且不影响现有的 ask_user、tracer 等 Hook。

### B. 为什么 RBAC Hook 是 CONTROL 类型而非 MODIFY

CONTROL 类型可以返回 ABORT 直接阻断不安全调用，也可以抛出 AskUserException 触发用户授权流程。MODIFY 类型只能修改参数，无法阻断调用。

### C. 为什么审计日志使用 OBSERVE Hook

审计日志不应影响工具调用的正常流程（即使工具执行失败也应记录），因此使用 OBSERVE 类型，静默执行不干扰主流程。

### D. 为什么参数脱敏在审计记录层而非工具层

工具参数是工具正常执行的必要数据，不应在工具感知不到的情况下被修改（否则工具执行结果可能异常）。脱敏仅影响审计日志中的存储内容，不应影响工具执行的参数。

### E. ask_user 工具本身的安全考虑

`ask_user` 工具的风险等级设为 `low`，因为该工具本质是信息请求而非系统操作。其权限升级机制（当 RBAC 校验失败后抛出 AskUserException）位于 Hook 层，与 ask_user 工具的语义不同——前者是安全机制，后者是业务交互。

### F. 为什么策略匹配采用首次匹配即生效而非异常列表

早期设计使用 `exceptions` 字段实现"默认禁止 + 例外放行"的模式（如禁止所有 agent 读取 `.env`，但允许 `plan_agent`）。这引入了两级匹配逻辑，增加了引擎复杂度。改为有序列表 + 首次匹配即生效后：

- 更具体的策略（如 `subject: "plan_agent" + resource: "file:**/*.env"`）排在前面
- 更宽泛的禁止策略（如 `subject: "*" + resource: "file:**/*.env"`）排在后面
- 引擎只需按序遍历，匹配即停止，逻辑更简单，行为更可预测

### G. 为什么临时授权令牌有效期设为 60 秒

权限升级时生成的临时令牌在 60 秒后自动失效。这是为了防止：
- 重放攻击：攻击者截获令牌后无法在过期后使用
- 会话劫持：如果用户离开终端，未处理的升级请求不会永久有效
- 令牌泄漏：即使 `agent_state` 被序列化到磁盘（会话持久化），过期令牌也会在下次会话恢复时被清理

60 秒对用户来说是合理的交互时间，对攻击者来说窗口足够短。

### H. 为什么脚本工具默认风险等级为 high

Skill 脚本通过 `shell=True` 在子进程中执行任意命令（`orchestration/subagent.py:137`），攻击面显著大于参数受限的 `run_command`。因此脚本工具默认标记为 `high`（Docker 沙箱执行）。这确保了新增 Skill 不会在无安全审查的情况下以低隔离级别执行。

### I. 审计日志的缓冲写入策略

每次工具调用都同步写磁盘会产生显著的 I/O 开销（ReAct 循环中可能调用数十个工具）。缓冲策略以 50 条为批次刷新，将会话期间的小写入合并为大块写入。在极端情况下（进程崩溃），最多丢失最后 49 条未刷新的记录。对于需要零丢失的场景，可将 `buffer_max_size` 设为 1（即时写入）。

### J. 为什么 Action 使用工具名而非抽象分类

早期设计将 Action 定义为四种抽象操作类型（read/write/delete/execute），需要维护 `_infer_action` 映射表从工具名推断 action。改为直接使用工具名后：

- **消除映射表维护成本**：每新增一个工具不再需要更新 `_infer_action` 的硬编码映射。
- **策略粒度更细**：可以直接在策略中区分 `read_file` 和 `search_web`，而不是笼统地归为同一个 `read` action。
- **支持 glob 批量匹配**：策略中的 action 支持 glob（如 `read_*` 匹配所有读取工具，`mcp__*` 匹配所有 MCP 工具），保持批量配置能力。
- **降低概念复杂度**：不再需要向读者解释"为什么 `search_web` 和 `run_command` 是不同的 action 类型（read vs execute）但都来自非文件工具"，每个工具名即其 action。

### K. 为什么 Resource 大多数工具使用通配符

Resource 提取的本质目的是让策略能区分"操作了哪个实体"——比如写入 `/tmp/test.txt` 和写入 `/etc/passwd` 需要不同的安全判定。但对于 `search_web`、`json_load`、`safe_calculator` 等工具，其入参本身的安全意义不足以通过路径匹配表达（URL 模式、JSON 文件路径等），强行提取只会让提取逻辑碎片化。仅在以下场景提取 Resource：

- **file_tools.py**：操作的文件/目录路径 — 有明确的路径级权限区分需求
- **cmd_tools.py**：执行的命令 — 可以区分 `ls` vs `rm -rf`

其余工具的 Resource 统一为 `*`，配合 Action 精确匹配已能实现有效权限控制。

### L. 为什么将 risk_level 从 RBAC 策略中分离

早期设计中 `risk_level` 作为 RBAC 策略的第四个属性，策略通过 `risk_levels: [low, medium]` 控制允许的风险等级范围。这种设计将两个独立概念耦合在一起：

- **授权决策**（谁能做什么）是策略层面的二元问题：允许或不允许。
- **执行方式**（怎么做）是工具属性层面的问题：如何安全地执行。

分离后：
- RBAC 策略只关心 effect（allow/deny），语义清晰，"未匹配策略 = 询问用户"的默认行为简单且安全。
- risk_level 回归工具属性，专门控制执行隔离级别（direct / isolated / sandbox），职责单一。
- MCP 工具的 risk_level 覆盖从 RBAC 配置文件移至 `mcp_servers.json`，因为它是工具的属性覆盖而非安全策略配置。

### M. 为什么未匹配策略时默认询问用户而非默认拒绝

默认拒绝的策略虽然最安全，但在实际使用中会导致大量工具调用被阻断，用户需要为每个预期内的操作逐一配置 `effect: allow` 策略，运维成本高。默认询问用户的设计在安全性和可用性之间取得平衡：

- 关键危险操作（删除系统文件、执行危险命令等）通过 `effect: deny` 策略主动阻断，不可绕过。
- 常规操作用户仅需确认一次（60 秒令牌有效期内同类操作自动放行），体验可接受。
- 部署者可以根据自身需求逐步增加 `effect: allow` 策略来减少询问频率，而不是一开始就需要配置大量策略。
