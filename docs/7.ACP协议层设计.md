# GraphAgent ACP（Agent Client Protocol）协议层设计文档

## 1. 概述

### 1.1 背景与动机

GraphAgent 当前通过 `debug.py` 中的 `while True: input()` 循环与用户交互，这是一种紧耦合的 CLI 模式。为支持 Web UI、IDE 插件、REST API 等多种交互方式，需要在 UI 层与 GraphAgent 编排引擎之间引入一个标准化的协议层 —— ACP（Agent Client Protocol）。

**核心问题：**

- 不同 UI 形态（Web、CLI、IDE、API）有不同的通信方式（HTTP、WebSocket、stdio），但需要共享同一套 Agent 能力
- 编排流程是异步、多阶段的（意图分析 → 方案制定 → 审核 → 执行 → 汇总），UI 需要感知中间状态
- 工具调用在多轮 ReAct 循环中执行，UI 可能需要展示工具调用/返回的细节
- 多轮对话需要会话管理（创建、加载、持久化、压缩），不能每个 UI 各自实现

### 1.2 ACP 的定位

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ Web UI   │  │ CLI 工具  │  │ IDE 插件  │  │ REST API │
│(Browser) │  │(Terminal) │  │(VSCode)  │  │(curl/..) │
└────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
     │              │              │              │
     │   HTTP/SSE   │   stdio     │   HTTP/SSE   │   HTTP/SSE
     │              │              │              │
     └──────────────┴──────────────┴──────────────┘
                         │
                         ▼
              ┌──────────────────────┐
              │     ACP Server       │  ← 协议编解码 + 会话管理
              │  (Transport Adapter) │
              └──────────┬───────────┘
                         │
                         ▼
              ┌──────────────────────┐
              │   GraphAgent 编排引擎  │  ← 不变的业务核心
              │  (Orchestration Graph)│
              └──────────────────────┘
```

ACP 不是 RPC 框架，而是**消息驱动的 Agent 交互协议**。它定义了 UI 客户端与 Agent 服务端之间交换消息的标准格式和通信语义。

### 1.3 设计目标

| 目标 | 说明 |
|------|------|
| **传输无关** | 同一套协议消息可通过 HTTP/SSE、WebSocket、stdio 等多种传输层承载 |
| **流式优先** | 编排流程的阶段变化、工具调用、LLM 输出均以流式事件推送 |
| **会话隔离** | 每个 UI 连接绑定一个 `session_id`，服务端管理完整的 `ConversationHistory` |
| **双向通信** | 客户端发送用户消息/控制指令，服务端推送事件/响应/错误 |
| **增量交付** | 最终回复与阶段性状态分开发送，UI 可以渐进式渲染 |
| **可扩展** | 消息类型通过 `event_type` 字段区分，新增事件类型不破坏兼容性 |

## 2. 协议规范

### 2.1 消息模型

所有 ACP 消息均遵循统一的 JSON 信封格式：

```json
{
  "version": "1.0",
  "type": "request | response | event",
  "event": "send_message",
  "id": "req_abc123",
  "timestamp": "2026-05-18T15:30:00Z",
  "payload": { }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `version` | string | 协议版本，当前 `"1.0"` |
| `type` | string | 消息方向：`"request"`（客户端→服务端）、`"response"`（服务端→客户端，单次回复）、`"event"`（服务端→客户端，流式推送） |
| `event` | string | 事件类型名称（见 §2.2） |
| `id` | string | 消息唯一标识；请求与对应的响应共享相同 ID |
| `timestamp` | string | ISO 8601 UTC 时间戳 |
| `payload` | object | 事件类型相关的负载，结构由各事件定义 |

### 2.2 事件类型总览

#### 2.2.1 客户端 → 服务端（Request）

| 事件 | 说明 | 关键 payload 字段 |
|------|------|-------------------|
| `send_message` | 发送用户消息 | `session_id`, `content` |
| `create_session` | 创建新会话 | `config`（可选，覆盖默认配置） |
| `load_session` | 加载已有会话 | `session_id` |
| `interrupt` | 中断当前执行 | `session_id` |
| `configure` | 更新会话配置 | `session_id`, `config` |
| `list_sessions` | 列出所有已持久化的会话 | — |
| `delete_session` | 删除指定会话 | `session_id` |

#### 2.2.2 服务端 → 客户端（Response / Event）

| 事件 | 类型 | 说明 | 关键 payload 字段 |
|------|------|------|-------------------|
| `ack` | response | 请求已接收 | `request_id`, `status` |
| `error` | response | 请求处理失败 | `request_id`, `code`, `message` |
| `phase_changed` | event | 编排阶段切换 | `phase`, `agent_name`, `detail` |
| `agent_message` | event | Agent 产生的消息（非最终） | `message`（`MessageBlock` JSON） |
| `llm_stream_start` | event | LLM 调用开始 | `agent_name`, `prompt_preview` |
| `llm_stream_chunk` | event | LLM 输出增量 | `delta` |
| `llm_stream_end` | event | LLM 调用完成 | `agent_name`, `token_usage` |
| `tool_call` | event | 工具被调用 | `tool_name`, `input_args` |
| `tool_result` | event | 工具返回结果 | `tool_name`, `output_preview` |
| `final_answer` | event | 最终回复 | `content` |
| `session_created` | response | 会话创建成功 | `session_id` |
| `session_loaded` | response | 会话加载成功 | `session_id`, `messages`, `turn_count` |
| `session_list` | response | 会话列表 | `sessions[]` |
| `session_deleted` | response | 会话已删除 | `session_id` |
| `execution_complete` | event | 本轮执行结束 | `session_id`, `turn_count` |
| `heartbeat` | event | 心跳保活 | — |

#### 场景 1：新会话 + 单轮对话


### 2.4 流式传输说明

`llm_stream_chunk` 事件用于 LLM 输出的增量传输。当底层 LLM provider 支持 streaming 时，ACP Server 将 chunk 逐个推送给客户端，实现逐字/逐词渲染效果。

```
Client                     ACP Server
  │                            │
  │◀── llm_stream_start ──────│  (开始生成)
  │◀── llm_stream_chunk ─────│  ("根据")
  │◀── llm_stream_chunk ─────│  ("您的")
  │◀── llm_stream_chunk ─────│  ("需求")
  │◀── llm_stream_chunk ─────│  ("...")
  │◀── llm_stream_end ───────│  (生成完成，含 token 统计)
```

## 3. 传输层适配

### 3.1 传输抽象

ACP 定义传输层接口 `ACPTransport`，所有具体传输方式实现该接口：

```python
class ACPTransport(ABC):
    """ACP 传输层抽象。"""

    @abstractmethod
    async def start(self) -> None:
        """启动传输层，开始监听连接。"""
        ...

    @abstractmethod
    async def send(self, connection_id: str, message: ACPMessage) -> None:
        """向指定连接发送消息。"""
        ...

    @abstractmethod
    async def broadcast(self, message: ACPMessage) -> None:
        """向所有连接广播消息。"""
        ...

    @abstractmethod
    async def on_message(self, connection_id: str, handler: Callable) -> None:
        """注册消息处理器。"""
        ...
```

### 3.2 内置传输实现

#### 3.2.1 HTTP + SSE（Server-Sent Events）

- **适用场景**：Web UI、REST API 客户端
- **请求通道**：HTTP POST `/acp/message` 发送 JSON 请求
- **事件通道**：HTTP GET `/acp/events?session_id=xxx` 建立 SSE 长连接，接收流式事件
- **框架选择**：FastAPI + sse-starlette

```
POST /acp/message          → 发送请求，返回 ack
GET  /acp/events?session=  → SSE 流，推送 phase_changed / final_answer 等事件
```

#### 3.2.2 WebSocket

- **适用场景**：需要全双工低延迟的 Web UI、实时协作
- **双向通道**：单连接同时承载请求和事件
- **消息格式**：文本帧承载 JSON 消息，与 ACP 信封格式完全一致

```
ws://host/acp/ws?session_id=xxx
  → Client: {"type":"request","event":"send_message",...}
  ← Server: {"type":"event","event":"phase_changed",...}
```

#### 3.2.3 stdio（标准输入输出）

- **适用场景**：CLI 工具、IDE 插件（LSP 风格）
- **输入通道**：stdin，每行一个 JSON 消息
- **输出通道**：stdout，每行一个 JSON 消息
- **控制/日志**：stderr 用于日志和调试

```
< stdin:  {"type":"request","event":"send_message",...}
> stdout: {"type":"response","event":"ack",...}
> stdout: {"type":"event","event":"phase_changed",...}
```

### 3.3 传输选择指南

| UI 形态 | 推荐传输 | 理由 |
|---------|---------|------|
| Web UI (React/Vue) | HTTP + SSE | 浏览器原生支持 SSE，实现简单 |
| Web UI (高级) | WebSocket | 全双工，支持中断和双向实时通信 |
| CLI 工具 | stdio | 零依赖，适合脚本和管道 |
| IDE 插件 | stdio | 与 LSP/DAP 等协议风格一致 |
| 移动端 | WebSocket | 全双工，省电省流量 |
| 第三方集成 | HTTP + SSE | 最广泛的兼容性 |

### 4.2 ACPServer 核心设计

```python
class ACPServer:
    """ACP 服务端 —— 传输无关的 Agent 服务。

    职责：
    1. 管理多个并发会话（ConversationHistory）
    2. 接收来自传输层的请求并分派到编排引擎
    3. 将编排引擎的输出转换为 ACP 事件推送给客户端
    4. 管理会话生命周期（创建、加载、压缩、持久化）
    """

    def __init__(self, config: ACPConfig):
        self._session_manager = SessionManager(config)
        self._transport: ACPTransport | None = None
        self._event_bus = EventBus()  # 内部事件总线

    async def bind_transport(self, transport: ACPTransport) -> None:
        """绑定传输层并注册消息处理器。"""
        ...

    async def handle_request(
        self, connection_id: str, request: ACPMessage
    ) -> None:
        """处理客户端请求。"""
        ...

    async def execute_turn(
        self, session_id: str, user_input: str
    ) -> AsyncIterator[ACPMessage]:
        """执行一轮对话并流式产出 ACP 事件。"""
        ...
```

```python
        """从磁盘加载已有会话。"""

        """获取会话上下文（含过滤后的注入消息）。"""

        """持久化会话到磁盘。"""
        ...

        ...
```

### 4.4 与编排引擎的集成

ACP Server 调用编排引擎的方式与 `debug.py` 完全一致，但增加了事件流式产出：

```python
async def execute_turn(
    self, session_id: str, user_input: str
) -> AsyncIterator[ACPMessage]:
    """执行一轮对话，流式产出 ACP 事件。"""
    ctx = await self._session_manager.get_context(session_id)

    # 1. 压缩检查
    comp_result = self._session_manager.compress_if_needed(session_id)
    if comp_result.did_compress:
        yield ACPMessage.event("phase_changed", {
            "phase": "compression",
            "detail": f"压缩完成: {comp_result.before_count} → {comp_result.after_count} 条消息",
        })

    # 2. 准备上下文
    ctx.history.add_user_message(user_input)
    context_msgs = ctx.history.get_context_messages()
    context_lc = agent_messages_to_langchain(context_msgs)
    context_lc.append(HumanMessage(content=user_input))

    # 3. 安装事件收集回调
    collector = ACPEventCollector(session_id)
    tracer = get_tracer()
    original_callback = tracer.get_llm_callback()
    tracer._llm_callback = ACPLLMCallback(collector, original_callback)

    # 4. 执行编排图
    try:
        result = await self._graph.ainvoke({"messages": context_lc})

        # 5. 广播阶段事件
        for event in collector.phase_events:
            yield event

        # 6. 广播工具调用事件
        for event in collector.tool_events:
            yield event

        # 7. 提取最终回复
        final_answer = result.get("final_answer", "")
        if not final_answer:
            final_answer = result["messages"][-1].content
        ctx.history.add_final_answer(final_answer)

        # 8. 广播 final_answer
        yield ACPMessage.event("final_answer", {"content": final_answer})

        # 9. 持久化
        await self._session_manager.save_session(session_id)

        # 10. 结束
        yield ACPMessage.event("execution_complete", {
            "session_id": session_id,
            "turn_count": ctx.history.turn_count,
        })

    except Exception as e:
        yield ACPMessage.event("error", {
            "code": "EXECUTION_ERROR",
            "message": str(e),
        })
    finally:
        # 恢复原始 callback
        tracer._llm_callback = original_callback
```

### 4.5 事件收集机制

ACP 利用现有的 `OrchestrationTracer` 和 `LLMCallbackHandler` 拦截编排流程中的关键节点，将内部可观测性事件转换为 ACP 事件：

```
OrchestrationTracer.trace_phase()
        │
        ▼
  ACPLLMCallback (decorator)
        │
        ├── 调用原始 LLMCallbackHandler（日志输出到 stderr）
        └── 将事件推入 ACPEventCollector
                │
                ▼
          ACPServer 异步迭代器
                │
                ▼
          ACPTransport.send() → 客户端
```

**事件映射表：**

| 内部可观测事件 | ACP 事件 | payload 处理 |
|---------------|----------|-------------|
| `trace_phase(name, agent, detail)` | `phase_changed` | `{phase: name, agent_name: agent, detail}` |
| `trace_decision(agent, decision, reason)` | `phase_changed` | `{phase: "decision", detail: decision}` |
| `on_llm_start` | `llm_stream_start` | `{agent_name, prompt_preview}` |
| streaming chunk | `llm_stream_chunk` | `{delta}` |
| `on_llm_end` | `llm_stream_end` | `{agent_name, token_usage}` |
| `on_tool_start` | `tool_call` | `{tool_name, input_args}` |
| `on_tool_end` | `tool_result` | `{tool_name, output_preview}` |
| `trace_error(agent, error)` | `error` | `{code, message}` |

## 5. 客户端 SDK 设计

### 5.1 SDK 定位

ACP 客户端 SDK 封装了传输层连接管理和消息序列化，为 UI 开发者提供简洁的异步 API，屏蔽底层协议细节。

### 5.2 核心 API

```python
class ACPClient:
    """ACP 客户端 —— UI 侧的协议封装。

    使用示例::

        async with ACPClient.http("http://localhost:8080") as client:
            session_id = await client.create_session()
            async for event in client.send_message(session_id, "你好"):
                if event.event == "final_answer":
                    print(event.payload["content"])
    """

    @classmethod
    def http(cls, base_url: str) -> "ACPClient":
        """通过 HTTP + SSE 连接到 ACP Server。"""
        ...

    @classmethod
    def websocket(cls, url: str) -> "ACPClient":
        """通过 WebSocket 连接到 ACP Server。"""
        ...

    @classmethod
    def stdio(cls) -> "ACPClient":
        """通过 stdio 连接到 ACP Server（用于 CLI / IDE 插件）。"""
        ...

    async def create_session(self, config: dict | None = None) -> str:
        """创建新会话，返回 session_id。"""
        ...

    async def load_session(self, session_id: str) -> SessionInfo:
        """加载已有会话，返回会话信息（含历史消息）。"""
        ...

    async def send_message(
        self, session_id: str, content: str
    ) -> AsyncIterator[ACPMessage]:
        """发送用户消息，返回事件流异步迭代器。"""
        ...

    async def list_sessions(self) -> list[SessionInfo]:
        """列出所有已持久化的会话。"""
        ...

    async def delete_session(self, session_id: str) -> bool:
        """删除指定会话。"""
        ...

    async def interrupt(self, session_id: str) -> None:
        """中断当前正在执行的对话。"""
        ...

    async def close(self) -> None:
        """关闭连接。"""
        ...

@dataclass
class SessionInfo:
    session_id: str
    created_at: str
    updated_at: str
    turn_count: int
    message_count: int
    preview: str  # 首条用户消息的前 100 字符
```

### 5.3 多语言 SDK

| 语言 | 传输方式 | 用途 |
|------|---------|------|
| Python | HTTP+SSE / WebSocket / stdio | CLI 工具、Python 脚本 |
| TypeScript | HTTP+SSE / WebSocket | Web UI (React/Vue)、Node.js 服务 |
| 其他 | HTTP+SSE | 通过标准 HTTP 库即可集成 |

TypeScript SDK 的 API 与 Python 镜像，差异仅在于异步模型（`async/await` → `Promise`，`AsyncIterator` → `EventSource` / `Observable`）。

## 7. 错误处理

### 7.1 错误码体系

| 错误码 | HTTP 状态 | 说明 |
|--------|----------|------|
| `INVALID_REQUEST` | 400 | 消息格式无效或缺少必填字段 |
| `SESSION_NOT_FOUND` | 404 | 指定的 session_id 不存在 |
| `SESSION_EXPIRED` | 410 | 会话已过期并被清理 |
| `SESSION_LIMIT` | 429 | 并发会话数超过上限 |
| `EXECUTION_ERROR` | 500 | 编排引擎执行异常 |
| `COMPRESSION_ERROR` | 500 | 压缩过程失败（降级到不压缩继续执行） |
| `LLM_ERROR` | 502 | LLM provider 调用失败 |
| `TRANSPORT_ERROR` | 500 | 传输层内部错误 |
| `TIMEOUT` | 504 | 编排执行超时 |

### 7.2 错误事件格式

```json
{
  "type": "response",
  "event": "error",
  "id": "req_abc123",
  "timestamp": "2026-05-18T15:30:00Z",
  "payload": {
    "code": "SESSION_NOT_FOUND",
    "message": "会话 '20260518_xxx' 不存在或已被删除",
    "detail": {},
    "recoverable": true
  }
}
```

| payload 字段 | 说明 |
|-------------|------|
| `code` | 机器可读错误码（§7.1） |
| `message` | 人类可读错误描述 |
| `detail` | 可选的错误详情（堆栈、上下文等，生产环境应裁剪） |
| `recoverable` | 是否可恢复（true 表示客户端可重试或继续使用会话） |

## 8. 与现有模块的关系

### 8.1 复用关系

```
ACP 层
 ├── 复用 session/history.py       → ConversationHistory（会话状态）
 ├── 复用 session/persistence.py   → ConversationPersistence（磁盘 IO）
 ├── 复用 session/compressor.py    → Priority/Summary Compressor（上下文压缩）
 ├── 复用 session/context_filter.py → is_context_eligible（上下文过滤）
 ├── 复用 message/convert.py       → agent_messages_to_langchain（格式转换）
 ├── 复用 message/base.py          → MessageBlock（消息序列化已有）
 ├── 复用 tracer/                  → OrchestraionTracer（事件拦截）
 ├── 复用 orchestration/graph.py   → build_orchestration_graph（编排图）
 └── 复用 llm/                     → LLMFactory（LLM 实例化）
```

### 8.2 零侵入原则

ACP 层是 GraphAgent 编排引擎的**外层包装**，不修改编排引擎的任何内部代码。编排引擎继续以 `graph.ainvoke({"messages": ...})` 方式运行，ACP 仅负责：

1. 将客户端请求转换为 `graph.ainvoke()` 的输入
2. 拦截编排节点的阶段/LLM/工具事件并转换为 ACP 事件
3. 管理 `ConversationHistory` 的生命周期

### 8.3 debug.py 迁移

现有的 `debug.py` 将重构为使用 ACP stdio 客户端：

```python
# 新的 debug.py（简化版）
async def main():
    async with ACPClient.stdio() as client:
        session_id = await client.create_session()
        while True:
            user_input = input("你: ")
            async for event in client.send_message(session_id, user_input):
                if event.event == "final_answer":
                    print(f"Agent: {event.payload['content']}")
                elif event.event == "phase_changed":
                    print(f"[{event.payload['phase']}]")
```

CLI 的完整格式化输出（带框线、分隔符等）仍由 debug.py 负责，ACP 只传递结构化事件数据。

## 11. 设计优势

1. **传输无关** —— 同一套协议消息可通过 HTTP/SSE、WebSocket、stdio 三种传输方式承载，适配不同 UI 形态
2. **流式原生** —— 协议设计从第一性原理支持流式事件，编排阶段变化、工具调用、LLM 输出渐进式推送
3. **零侵入** —— ACP 层作为编排引擎的外层包装，不修改任何现有编排节点代码，复用现有的 Tracer 事件拦截机制
4. **会话完整管理** —— 复用已有的 `ConversationHistory` + `ConversationPersistence` + 压缩器，ACP 只需做调度编排
5. **复用 MessageBlock 序列化** —— `persistence.py` 中已有的 `_serialize_message` / `_deserialize_message` 直接用于 ACP 消息的 JSON 编解码
6. **渐进式实现** —— 先完成核心协议 + HTTP/SSE 传输即可验证整体流程，WebSocket/stdio/TypeScript SDK 等按需追加
7. **事件类型可扩展** —— 新增 ACP 事件类型不影响已有客户端，只需双方协商新的 `event` 名称
8. **多语言支持** —— 协议使用 JSON 明文，任何语言只需实现 HTTP + SSE 客户端即可集成
