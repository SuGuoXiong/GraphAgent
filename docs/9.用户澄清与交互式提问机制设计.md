# 9. 用户澄清与交互式提问机制设计

## 1. 概述

### 1.1 背景

当前 GraphAgent 的意图分析是一次性的：GuardAgent 分析用户输入后直接进入任务规划，即便用户的描述模糊、缺少关键信息、存在歧义，系统也不会追问，而是基于不完整的信息强行推进。这导致最终回答质量低、偏离用户真实意图，甚至因关键参数缺失而执行失败。

### 1.2 目标

- **主动识别歧义**：GuardAgent 在意图分析阶段检测用户描述是否清晰、信息是否充分
- **向用户提问**：当信息不足时，生成结构化的澄清问题，必要时附带可选选项供用户点选
- **多轮澄清**：支持多轮问答，直到信息充分后再推进编排流程
- **协议支持**：通过 ACP 协议暴露澄清事件，Web UI 提供交互式澄清体验
- **无缝集成**：澄清过程不影响后续的编排执行，澄清结果作为上下文注入

### 1.3 非目标

- 不支持执行过程中（PlanAgent / SubAgent 阶段）的澄清（仅意图分析阶段）
- 不替代 PlanAgent 的 `guard_feedback` 回退循环（那是方案审核，不是需求澄清）
- 不支持离线场景（用户离线时自动选择默认选项）

## 2. 澄清流程设计

### 2.1 整体流程

```
用户发送消息
    │
    ▼
GuardAgent 意图分析（含澄清上下文）
    │
    ├── 问题清晰 → 继续编排（现有流程不变）
    │
    └── 需要澄清 → 生成提问 + 选项
            │
            ▼
        推送 CLARIFICATION_NEEDED 事件
        会话状态 → AWAITING_CLARIFICATION
            │
            ▼
        Web UI 展示提问卡片（含可点击选项/输入框）
            │
            ▼
        用户提交澄清回复
            │
            ▼
        GuardAgent 再次分析（附带历史澄清上下文）
            │
            ├── 仍不清晰 → 再次提问（最多 3 轮）
            │
            └── 已清晰 → 继续编排
```

### 2.2 澄清轮次限制

为防止无限循环，澄清最多 3 轮。第 3 轮后若仍不清晰，GuardAgent 基于已有信息"尽最大努力"继续执行并附注不确定项。

### 2.3 状态转换

在现有会话状态机中新增 `AWAITING_CLARIFICATION` 状态：

```
IDLE → (send_message) → RUNNING → (needs_clarification) → AWAITING_CLARIFICATION
                                       │
AWAITING_CLARIFICATION → (provide_clarification) → RUNNING → ...
AWAITING_CLARIFICATION → (send_message) → RUNNING（放弃澄清，重新开始）
```

## 4. GuardAgent 改造

### 4.1 意图分析提示词扩展

扩展 `prompts/guard/intent_analysis.txt`，增加澄清职责：

```
你是一个 GuardAgent（质量把关者），负责准确理解用户需求并提炼核心意图。

## 职责
1. 分析用户输入，识别用户真正想要完成的目标
2. 判断任务复杂度：简单任务可直接给出答案，复杂任务需要制定执行方案
3. 提炼意图为一句话描述，传递给 PlanAgent 进行任务分解
4. 检测用户描述是否清晰，若信息不足则主动提问澄清

## 澄清判断标准
以下情况应标记 needs_clarification=true 并生成澄清问题：
- 核心目标模糊：不知道用户到底想要完成什么
- 关键参数缺失：如文件路径、日期范围、目标系统等必要参数
- 存在歧义：用户描述有多种合理理解
- 范围过大：用户请求过于宽泛，需要限定范围
- 依赖不明确：用户引用了当前上下文无法确定的内容

以下情况不应追问：
- 用户已明确拒绝提供更多信息
- 已有 3 轮及以上澄清（此时应基于现有信息尽最大努力执行）
- 简单问候、闲聊

## 输出格式
{{
  "intent": "用户意图的一句话描述（即使不清晰，也给出最佳猜测）",
  "complexity": "simple | medium | complex",
  "key_requirements": ["需求点1", "需求点2"],
  "notes": "补充说明（如有）",
  "needs_clarification": false,
  "clarification_question": "向用户提出的澄清问题（清晰简洁，一句话）",
  "clarification_options": ["选项A", "选项B"],
  "uncertain_points": ["不确定的具体内容"]
}}
```

### 4.2 _analyze_intent 逻辑改造

```python
def _analyze_intent(state: OrchestrationState) -> dict:
    """意图识别阶段 —— 含澄清检测。"""
    get_tracer().trace_phase("意图分析", "GuardAgent", "分析用户输入")

    context_msgs = state.get("messages", [])
    user_text = context_msgs[-1].content if context_msgs else ""

    # 构建提示词，注入历史澄清上下文
    clarification_rounds = state.get("clarification_rounds", 0)
    max_rounds = state.get("max_clarification_rounds", 3)

    system_prompt = _prompt_loader.load_with_context(
        "guard", "intent_analysis",
        conversation_context=_format_context(context_msgs),
    )
    # 非首轮分析时追加提示
    if clarification_rounds > 0:
        system_prompt += (
            f"\n\n【注意】这是第 {clarification_rounds} 轮澄清。"
            f"最多 {max_rounds} 轮。若本轮后信息仍不足，请基于现有信息尽最大努力执行。"
        )

    msg = _call_llm(system_prompt, user_text, state,
                    name="GuardAgent", message_type=MessageType.GUARD_INTENT_ANALYSIS)
    result = _parse_json_response(msg.content if isinstance(msg.content, str) else "")

    needs_clarification = result.get("needs_clarification", False)
    rounds = clarification_rounds

    if needs_clarification and rounds < max_rounds:
        return {
            "phase": OrchestrationPhase.AWAITING_CLARIFICATION,
            "intent": result.get("intent", user_text),
            "needs_clarification": True,
            "clarification_question": result.get("clarification_question", ""),
            "clarification_options": result.get("clarification_options", []),
            "clarification_rounds": rounds + 1,
            "ga_messages": [msg],
            "messages": [agent_message_to_langchain(msg)],
        }
    else:
        # 不再追问：信息充分，或已达最大轮次
        if needs_clarification and rounds >= max_rounds:
            notes = result.get("notes", "")
            result["notes"] = (notes + " [注意：已达最大澄清轮次，基于现有信息执行]").strip()

        return {
            "phase": OrchestrationPhase.PLAN_GENERATION,
            "intent": result.get("intent", user_text),
            "needs_clarification": False,
            "clarification_rounds": rounds,
            "ga_messages": [msg],
            "messages": [agent_message_to_langchain(msg)],
        }
```

def guard_node(state: OrchestrationState) -> dict:
    from graph_agent.acp.checkpoint import _check_interrupt

    phase = state.get("phase", OrchestrationPhase.INTENT_ANALYSIS)

    if phase == OrchestrationPhase.INTENT_ANALYSIS:

## 5. 编排图改造

### 5.1 图结构

编排图结构不变（仍是 guard / plan / sub_exec 三个节点），澄清通过 `guard_router` 返回 `__end__` 实现。`AWAITING_CLARIFICATION` 阶段不增加新节点，避免图复杂度膨胀。

### 5.2 恢复执行

用户提交澄清后，ACPServer 的 `execute_turn()` 被再次调用。此时会话状态为 `AWAITING_CLARIFICATION`，且 `clarification_rounds > 0`。`execute_turn()` 正常构建 initial_state（含新增的澄清消息），注入已有状态字段（`clarification_rounds`, `intent` 等），GuardAgent 在 `INTENT_ANALYSIS` 阶段重新分析。

```
execute_turn 改造（新增逻辑）：

1. 若 ctx.status == AWAITING_CLARIFICATION:
   - 用户在澄清阶段发送的消息作为澄清回复
   - initial_state 注入已有 clarification_rounds / intent 等字段
   - phase 设置为 INTENT_ANALYSIS（重新进入意图分析）
2. 其余逻辑不变
```

## 9. 提示词设计原则

### 9.1 澄清问题的质量要求

- **具体**：不能问"请提供更多信息"，要问"您希望分析哪个目录下的代码？"
- **限定选项**：尽量提供 2-4 个可选选项，降低用户输入负担
- **一次一问**：每轮只问一个问题，避免用户困惑
- **暗示默认**：若有合理默认值，在选项中标注"（推荐）"

### 9.2 何时不应追问

- 用户明确拒绝提供更多信息（如说"随便"、"无所谓"、"你决定"）
- 已经 3 轮澄清（配置项 `max_clarification_rounds`）
- 用户进行简单闲聊/问候
- 缺失的信息有合理默认值（如"今天的日期"）

## 12. 风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 对澄清判断不准确 | 不该追问时追问或该追问时不追问 | 提示词中增加正反例；max 3 轮硬限制 |
| 用户不理解澄清机制 | 用户困惑为什么不直接回答 | 澄清卡片中说明原因（"因为缺少以下信息..."） |
| 澄清轮次过多 | 用户体验下降、token 消耗增加 | 硬限制 3 轮；鼓励提供默认选项 |
| 澄清上下文过长 | 压缩后可能丢失关键信息 | 澄清消息使用 P1 优先级，不易被压缩 |
| 与现有中断机制冲突 | AWAITING_CLARIFICATION 与 PAUSED 语义重叠 | 两者互斥：PAUSED 是用户主动暂停，AWAITING_CLARIFICATION 是系统请求输入 |
