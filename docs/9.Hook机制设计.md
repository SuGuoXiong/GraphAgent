# 9. Hook 机制设计

## 1. 概述

### 1.1 背景

当前 GraphAgent 的扩展点依赖 LangChain Callback（仅能观测 LLM/工具调用，无法修改数据或控制流程）和手动埋点（tracer）。随着系统演进，出现了三类需求是现有机制无法满足的：

- **数据校验与转换**：工具调用前对参数做合法性校验或自动修正，LLM 返回后对结果做格式规范化
- **流程控制**：工具调用前询问用户是否允许本次操作，用户拒绝则跳过调用
- **非侵入式观测**：在工具/LLM 调用前后记录审计日志、上报指标

这些需求具有共性：它们都希望在特定的执行检查点介入，但介入的目的和权限不同。Hook 机制将这三类需求统一建模，提供结构化的扩展点。

此外，现有的 `OrchestrationTracer` 模块（参见 [5.可观测性系统设计](5.可观测性系统设计.md)）中，`LLMCallbackHandler` 通过 LangChain Callback 拦截 LLM 和工具调用事件以输出终端日志——其本质就是一个 Type 3（纯观测）Hook。将其迁移到 Hook 框架下，可以统一扩展机制、消除 LangChain Callback 的间接依赖。

### 1.2 目标

- **检查点埋点**：在工具调用前/后、LLM 调用前/后四个检查点插入 Hook 触发逻辑
- **三类 Hook**：数据修改型（可变更输入/输出）、流程控制型（可中断/跳过执行）、纯观测型（不可修改数据和流程）
- **声明式注册**：通过 `@hook` 注解声明 Hook 方法，自动发现并注册
- **有序编排**：同一检查点可注册多个 Hook，按优先级顺序执行
- **上下文传递**：通过 `HookContext` 统一传递检查点相关信息
- **Tracer 迁移**：将 `LLMCallbackHandler` 的 LLM/工具调用追踪功能重构为内置 Type 3 Hook，统一到 Hook 框架下

### 1.3 非目标

- 不支持异步 Hook（Hook 方法均为同步调用，避免引入复杂度）
- 不支持 Hook 间的依赖编排（仅支持优先级排序，不构建 DAG）
- 不支持运行时动态加载/卸载 Hook（注册在启动时完成）
- 不将 OrchestrationTracer 的阶段性功能（`trace_phase`、`trace_decision`）迁移到 Hook（阶段切换事件与工具/LLM 调用检查点属不同抽象层级，保持独立）

## 2. 核心概念

### 2.1 检查点

| 检查点 | 触发时机 | 携带数据 |
|--------|---------|---------|
| `before_tool_call` | 工具执行前，参数已解析完毕 | 工具名、工具参数、当前状态 |
| `after_tool_call` | 工具执行后，结果已返回 | 工具名、工具参数、工具结果、状态 |
| `before_llm_call` | LLM 调用前，消息已组装完毕 | 消息列表、模型名、调用者名称 |
| `after_llm_call` | LLM 调用后，响应已返回 | 消息列表、响应内容、token 用量、调用者名称 |

### 2.2 Hook 类型

| 类型 | 能力 | 典型场景 |
|------|------|---------|
| **Type 1 — 数据修改** | 读取并修改 `HookContext` 中的数据字段 | 参数校验与修正、结果格式化、敏感信息脱敏 |
| **Type 2 — 流程控制** | 返回 `HookDecision` 决定是否继续执行 | 用户确认（human-in-the-loop）、安全拦截、权限检查 |
| **Type 3 — 纯观测** | 只读访问 `HookContext`，不可修改数据或控制流程 | 审计日志、指标上报、终端调试输出（Tracer 迁移目标） |

三种类型的权限矩阵：

| 操作 | Type 1 | Type 2 | Type 3 |
|------|--------|--------|--------|
| 读取 HookContext | ✅ | ✅ | ✅ |
| 修改 HookContext 数据字段 | ✅ | ❌ | ❌ |
| 返回 HookDecision 控制流程 | ❌ | ✅ | ❌ |

### 2.3 HookContext

统一的数据载体，在检查点触发时构建并依次传递给各 Hook：

```python
@dataclass
class HookContext:
    """Hook 执行上下文，携带检查点的相关信息。"""

    # 检查点标识
    checkpoint: str  # "before_tool_call" | "after_tool_call" | "before_llm_call" | "after_llm_call"

    # 工具相关（before_tool_call / after_tool_call 时填充）
    tool_name: str | None = None
    tool_args: dict | None = None
    tool_result: str | None = None        # 仅 after_tool_call
    tool_error: str | None = None         # 仅 after_tool_call（异常时）

    # LLM 相关（before_llm_call / after_llm_call 时填充）
    llm_messages: list | None = None      # 发送给 LLM 的消息列表
    llm_model: str | None = None          # 模型名称
    llm_caller: str | None = None         # 调用者名称（如 "GuardAgent", "PlanAgent"）
    llm_response: str | None = None       # 仅 after_llm_call
    llm_token_usage: dict | None = None   # 仅 after_llm_call

    # 编排状态（所有检查点均可用）
    session_id: str | None = None
    agent_state: dict | None = None       # 当前 OrchestrationState 的快照
```

### 2.4 HookDecision

Type 2 Hook 通过返回 `HookDecision` 来控制执行流程：

```python
class HookAction(Enum):
    CONTINUE = "continue"    # 正常继续执行
    SKIP = "skip"            # 跳过本次调用（工具调用时返回预设占位结果，LLM 调用时返回空响应）
    ABORT = "abort"          # 中断整个编排流程，抛出异常

@dataclass
class HookDecision:
    action: HookAction
    reason: str = ""                        # 决策原因（用于日志/审计）
    fallback_result: str | None = None      # action=SKIP 时的回退结果
```

### 3.2 核心组件

#### 3.2.1 Hook 函数签名

不同类型对应不同的函数签名：

```python
# Type 1 — 数据修改型：接收并返回 HookContext
def type1_hook(ctx: HookContext) -> HookContext:
    ...

# Type 2 — 流程控制型：接收 HookContext，返回 HookDecision
def type2_hook(ctx: HookContext) -> HookDecision:
    ...

# Type 3 — 纯观测型：接收 HookContext，无返回值
def type3_hook(ctx: HookContext) -> None:
    ...
```

#### 3.2.2 @hook 装饰器

```python
def hook(
    checkpoint: str,            # 检查点标识
    priority: int = 100,        # 优先级（数字越小越先执行）
    hook_type: HookType = HookType.OBSERVE,  # Hook 类型
    description: str = "",      # 描述信息
):
    """声明一个方法为 Hook。

    被装饰的函数会被 HookRegister 自动发现并注册。

    Args:
        checkpoint: 检查点，取值为 "before_tool_call" | "after_tool_call"
                    | "before_llm_call" | "after_llm_call"
        priority: 优先级，数字越小越先执行，默认 100
        hook_type: Hook 类型
        description: Hook 描述
    """
```

使用示例：

```python
from graph_agent.hook import hook, HookContext, HookDecision, HookAction, HookType

# Type 1: 工具参数校验与修正
@hook("before_tool_call", priority=10, hook_type=HookType.MODIFY,
      description="校验文件路径，防止路径穿越")
def validate_file_path(ctx: HookContext) -> HookContext:
    if ctx.tool_name == "read_file":
        path = ctx.tool_args.get("path", "")
        if ".." in path:
            ctx.tool_args["path"] = path.replace("..", "")
    return ctx

# Type 2: 工具调用前请求用户确认
@hook("before_tool_call", priority=50, hook_type=HookType.CONTROL,
      description="高危命令执行前请求用户确认")
def confirm_dangerous_cmd(ctx: HookContext) -> HookDecision:
    if ctx.tool_name == "execute_command":
        return HookDecision(action=HookAction.SKIP,
                           reason="高危命令需用户确认",
                           fallback_result="操作已被拦截，等待用户确认")
    return HookDecision(action=HookAction.CONTINUE)

# Type 3: 审计日志
@hook("after_tool_call", priority=200, hook_type=HookType.OBSERVE,
      description="记录所有工具调用审计日志")
def audit_tool_call(ctx: HookContext) -> None:
    logger.info(f"[AUDIT] tool={ctx.tool_name} args={ctx.tool_args} "
                f"result_preview={str(ctx.tool_result)[:200]}")
```

#### 3.2.3 HookRegister

```python
class HookRegister:
    """Hook 注册中心。

    启动时扫描项目中所有 @hook 注解的方法，按检查点分组存储。
    """

    def __init__(self):
        self._hooks: dict[str, list[_HookEntry]] = {
            "before_tool_call": [],
            "after_tool_call": [],
            "before_llm_call": [],
            "after_llm_call": [],
        }

    def register(self, func: Callable, checkpoint: str, priority: int,
                 hook_type: HookType, description: str) -> None:
        """注册一个 Hook 方法。"""

    def auto_discover(self, package: str = "graph_agent") -> None:
        """自动扫描项目中所有 @hook 注解的方法并注册。"""

    def get_hooks(self, checkpoint: str) -> list:
        """获取指定检查点的 Hook 列表（已按优先级排序）。"""
```

#### 3.2.4 HookExecutor

```python
class HookExecutor:
    """Hook 执行器，按编排顺序执行同一检查点的所有 Hook。

    Type 1（MODIFY）Hook 链式传递修改后的 HookContext。
    Type 2（CONTROL）Hook 遇 SKIP/ABORT 立即终止后续 Hook 执行。
    Type 3（OBSERVE）Hook 静默执行，不影响数据流和控制流。
    """

    def __init__(self, register: HookRegister):
        self._register = register

    def execute(self, checkpoint: str, ctx: HookContext) -> tuple[HookContext, HookDecision | None]:
        """执行指定检查点的所有 Hook。

        Returns:
            (最终 HookContext, 流程决策)。决策为 None 表示 CONTINUE。
        """
```

### 3.3 Hook 执行流程

```
埋点触发 hook_before_tool_call(tool_name, tool_args, state)
    │
    ▼
构建 HookContext(checkpoint="before_tool_call", tool_name=..., tool_args=..., ...)
    │
    ▼
HookExecutor.execute("before_tool_call", ctx)
    │
    ├─→ Hook A (Type 1, priority=10): ctx = hook_a(ctx)    ← 数据修改
    ├─→ Hook B (Type 2, priority=50): decision = hook_b(ctx) ← 流程控制
    │       │
    │       ├─ CONTINUE → 继续下一个 Hook
    │       ├─ SKIP     → 终止，返回 SKIP 决策
    │       └─ ABORT    → 终止，抛出 HookAbortError
    ├─→ Hook C (Type 3, priority=200): hook_c(ctx)          ← 纯观测
    │
    ▼
返回 (修改后的 ctx, 决策)

如果决策 == SKIP:
    工具调用被跳过，返回 fallback_result
    或 LLM 调用被跳过，返回空响应
如果决策 == CONTINUE / None:
    正常执行工具/LLM 调用
```

### 4.3 埋点实现方式

在 `AgentTool.run()` 和 LLM 调用路径统一埋点，使所有调用路径（简单图、编排图）均自动覆盖：

**工具调用埋点（`src/graph_agent/tools/base.py` 的 `AgentTool.run`）：**

```python
def run(self, **kwargs: Any) -> str:
    from graph_agent.hook import get_hook_executor, HookContext, HookAction, HookAbortError

    executor = get_hook_executor()
    ctx = HookContext(
        checkpoint="before_tool_call",
        tool_name=self.name,
        tool_args=kwargs,
    )
    ctx, decision = executor.execute("before_tool_call", ctx)

    if decision and decision.action == HookAction.SKIP:
        return decision.fallback_result or f"工具 '{self.name}' 被 Hook 跳过"
    if decision and decision.action == HookAction.ABORT:
        raise HookAbortError(decision.reason)

    # 应用 Type 1 Hook 可能修改的参数
    kwargs = ctx.tool_args or kwargs

    result = self.func(**kwargs)

    ctx.tool_result = result
    ctx.checkpoint = "after_tool_call"
    executor.execute("after_tool_call", ctx)

    return result
```

**LLM 调用埋点（`src/graph_agent/llm/base.py` 的 `LLMProvider` 层）：**

在 `LLMProvider.get_chat_model()` 返回的模型上包装拦截层，利用 LangChain 的 Runnable 链插入 before/after 逻辑：

```python

# before_llm_call: 传入 messages、model、caller_name
# after_llm_call:  传入 messages、response、token_usage
```

```
    ▼

    │
    ▼

    │
    ▼
Agent 正常运行，检查点自动触发 Hook
```

### 5.2 自动发现机制

与 `ToolCenter.auto_discover()` 一致的设计模式：

1. 递归遍历 `graph_agent` 包下所有 `.py` 文件
2. 检查每个模块的全局属性，收集被 `@hook` 装饰的函数
3. `@hook` 装饰器在被调用时将函数元信息存入函数的 `__hook_meta__` 属性
4. `auto_discover()` 读取 `__hook_meta__` 并调用 `register()`
5. 每个检查点的 Hook 列表按 `priority` 升序排列

```python
def auto_discover(self, package: str = "graph_agent") -> None:
    import importlib, pkgutil
    pkg = importlib.import_module(package)
    for _, mod_name, _ in pkgutil.walk_packages(pkg.__path__, prefix=package + "."):
        module = importlib.import_module(mod_name)
        for attr_name in dir(module):
            attr = getattr(module, attr_name)
            if callable(attr) and hasattr(attr, "__hook_meta__"):
                meta = attr.__hook_meta__
                self.register(attr, meta["checkpoint"], meta["priority"],
                             meta["hook_type"], meta["description"])
```

## 6. Tracer 模块重构：将 LLMCallbackHandler 迁移为 Type 3 Hook

### 6.1 现状分析

当前 Tracer 模块（`src/graph_agent/tracer/`）包含两部分职责：

| 职责 | 实现 | 触发方式 | Hook 对应 |
|------|------|---------|-----------|
| LLM I/O 追踪 | `LLMCallbackHandler.on_llm_start/end` | LangChain Callback 自动拦截 | → `before_llm_call` / `after_llm_call` (Type 3) |
| 工具调用追踪 | `LLMCallbackHandler.on_tool_start/end` | LangChain Callback 自动拦截 | → `before_tool_call` / `after_tool_call` (Type 3) |
| 阶段切换追踪 | `OrchestrationTracer.trace_phase/decision/end` | Node 函数手动埋点 | 不迁移（属不同抽象层级） |

`LLMCallbackHandler` 的四个回调与 Hook 的四个检查点一一对应：

```
on_llm_start  → before_llm_call
on_llm_end    → after_llm_call
on_tool_start → before_tool_call
on_tool_end   → after_tool_call
```

### 6.2 迁移方案

#### 6.2.1 新建 `tracer_hooks.py`

在 `src/graph_agent/hook/builtin/tracer_hooks.py` 中实现四个 Type 3 Hook，复用现有的 `format.py` 终端格式化函数：

```python
"""从 LLMCallbackHandler 迁移的终端输出 Hook（Type 3 — 纯观测）。

替代 tracer/llm_callback.py 的功能，通过 Hook 框架在工具/LLM 调用前后
自动触发终端格式化输出。
"""

import os
from graph_agent.hook import hook, HookContext, HookType
from graph_agent.tracer.format import (
    print_llm_request, print_llm_response,
    print_tool_call, print_tool_result,
)

_LOG_LEVEL = os.getenv("GRAPHAGENT_LOG_LEVEL", "llm_io")


def _show_llm() -> bool:
    return _LOG_LEVEL in ("llm_io", "full")


def _show_tools() -> bool:
    return _LOG_LEVEL == "full"


@hook("before_llm_call", priority=500, hook_type=HookType.OBSERVE,
      description="终端输出 LLM 请求内容（System Prompt + User Message）")
def tracer_before_llm(ctx: HookContext) -> None:
    if not _show_llm():
        return
    caller = ctx.llm_caller or "LLM"
    for msg in (ctx.llm_messages or []):
        print_llm_request(caller, _extract_system(msg), _extract_user(msg))


@hook("after_llm_call", priority=500, hook_type=HookType.OBSERVE,
      description="终端输出 LLM 响应内容和 Token 用量")
def tracer_after_llm(ctx: HookContext) -> None:
    if not _show_llm():
        return
    caller = ctx.llm_caller or "LLM"
    token_info = _format_token_info(ctx.llm_token_usage)
    print_llm_response(caller, ctx.llm_response or "", token_info)


@hook("before_tool_call", priority=500, hook_type=HookType.OBSERVE,
      description="终端输出工具调用参数（仅 full 级别）")
def tracer_before_tool(ctx: HookContext) -> None:
    if not _show_tools():
        return
    print_tool_call(ctx.tool_name or "unknown", ctx.tool_args or {})


@hook("after_tool_call", priority=500, hook_type=HookType.OBSERVE,
      description="终端输出工具调用结果（仅 full 级别）")
def tracer_after_tool(ctx: HookContext) -> None:
    if not _show_tools():
        return
    print_tool_result(ctx.tool_name or "unknown", str(ctx.tool_result)[:500])
```

#### 6.2.2 精简 OrchestrationTracer

`tracer/tracer.py` 移除对 `LLMCallbackHandler` 的依赖，仅保留阶段级别的埋点方法：

```python
class OrchestrationTracer:
    """全局可观测性管理器（单例）—— 仅保留阶段/决策级埋点。

    LLM/工具调用的终端输出已迁移到 hook/builtin/tracer_hooks.py。
    """

    _instance: Optional["OrchestrationTracer"] = None

    def __new__(cls, level: Optional[str] = None):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._initialized = False
        return cls._instance

    def __init__(self, level: Optional[str] = None):
        if self._initialized:
            if level is not None:
                self.log_level = level
            return
        if level is None:
            level = os.getenv("GRAPHAGENT_LOG_LEVEL", "llm_io")
        self.log_level = level.lower()
        self._call_sequence = 0
        self._initialized = True

    # install() 不再注册 LLMCallbackHandler，仅作为初始化标记
    def install(self) -> None:
        pass

    # 以下方法保持不变
    def trace_phase(self, phase_name: str, agent_name: str, detail: str = "") -> None: ...
    def trace_decision(self, agent_name: str, decision: str, reason: str = "") -> None: ...
    def trace_error(self, agent_name: str, error: str) -> None: ...
    def trace_phase_end(self, status: str = "完成") -> None: ...
```

### 6.5 优先级设计说明

Tracer Hook 的优先级设为 `500`（远高于默认值 `100`），确保：
- Type 1 数据修改 Hook（如参数校验 priority=10）先执行，Tracer 输出的是修改后的数据
- Type 2 流程控制 Hook（如用户确认 priority=50）先执行，若被 SKIP 则 Tracer 不会收到对应事件
- Tracer Hook 在所有业务 Hook 之后执行，忠实记录最终状态

## 7. 与现有系统的关系

### 7.1 与 ACP 协议的关系

Type 2 Hook 的 `SKIP` 决策可触发 ACP 的推送事件（如 `TOOL_CALL` 事件中携带 `interrupted: true`），通知前端等待用户确认。用户确认后通过 `RESUME_SESSION` 恢复执行。

### 7.2 与 SubAgent 架构的关系

每个 SubAgent 在 ReAct 循环中的工具调用和 LLM 调用均会触发 Hook。Hook 接收的 `HookContext.agent_state` 可区分当前是哪个 SubAgent 在执行，支持按 SubAgent 粒度定制 Hook 行为。

### 7.3 与 LLMFactory callback 机制的关系

Hook 机制落地后，`LLMFactory` 的 `register_callback` / `get_callbacks` 可标记为 deprecated 并最终移除。LLM/Tool 级别的拦截完全由 Hook 框架接管，Provider 不再需要传入 `callbacks=` 参数。

## 9. 错误处理

### 9.1 Hook 执行异常

- **Type 1 / Type 3 Hook 抛出异常**：记录错误日志，跳过该 Hook，继续执行后续 Hook 和主流程。不阻断正常业务。
- **Type 2 Hook 抛出异常**：视为 `ABORT` 决策，抛出 `HookAbortError`，中断当前调用。

### 9.2 超时保护

每个 Hook 执行设置超时（默认 5 秒），超时后记录警告并跳过：

```python
TIMEOUT = 5  # 秒
```

### 9.3 异常隔离

Hook 内发生的异常不会导致 Agent 崩溃（Type 2 ABORT 除外）。异常信息记录到日志并可选上报。

## 11. 附录

### 11.1 完整类型定义

```python
# src/graph_agent/hook/base.py

from dataclasses import dataclass, field
from enum import Enum
from typing import Any, Callable


class HookType(Enum):
    MODIFY = "modify"    # 可修改 HookContext 数据
    CONTROL = "control"  # 可返回 HookDecision 控制流程
    OBSERVE = "observe"  # 只读，不可修改数据或流程


class HookAction(Enum):
    CONTINUE = "continue"
    SKIP = "skip"
    ABORT = "abort"


@dataclass
class HookDecision:
    action: HookAction
    reason: str = ""
    fallback_result: str | None = None


@dataclass
class HookContext:
    checkpoint: str
    tool_name: str | None = None
    tool_args: dict | None = None
    tool_result: str | None = None
    tool_error: str | None = None
    llm_messages: list | None = None
    llm_model: str | None = None
    llm_caller: str | None = None
    llm_response: str | None = None
    llm_token_usage: dict | None = None
    session_id: str | None = None
    agent_state: dict | None = None


# Hook 函数类型别名
ModifyHook = Callable[[HookContext], HookContext]
ControlHook = Callable[[HookContext], HookDecision]
ObserveHook = Callable[[HookContext], None]
HookFunc = ModifyHook | ControlHook | ObserveHook


class HookAbortError(Exception):
    """Type 2 Hook 返回 ABORT 时抛出的异常。"""
    def __init__(self, reason: str = ""):
        self.reason = reason
        super().__init__(f"执行被 Hook 中断: {reason}")
```

### 11.2 Hook 执行伪代码

```python
def execute(self, checkpoint: str, ctx: HookContext) -> tuple[HookContext, HookDecision | None]:
    hooks = self._register.get_hooks(checkpoint)
    final_decision = None

    for entry in hooks:
        try:
            if entry.hook_type == HookType.MODIFY:
                ctx = entry.func(ctx)
            elif entry.hook_type == HookType.CONTROL:
                decision = entry.func(ctx)
                if decision.action in (HookAction.SKIP, HookAction.ABORT):
                    final_decision = decision
                    break
            elif entry.hook_type == HookType.OBSERVE:
                entry.func(ctx)
        except Exception as e:
            if entry.hook_type == HookType.CONTROL:
                raise HookAbortError(f"Control hook '{entry.name}' 异常: {e}")
            # Type 1 / Type 3 异常不阻断
            log_warning(f"Hook '{entry.name}' 执行异常: {e}")

    return ctx, final_decision
```
